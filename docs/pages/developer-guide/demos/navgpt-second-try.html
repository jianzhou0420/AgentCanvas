NavGPT-CE (Second Try) — Graph Design Rationale

This page explains the design rationale behind the navgpt_second_try canvas graph — a refined AgentCanvas representation of the NavGPT zero-shot VLN agent (Zhou et al., 2023). It documents why each node exists, how the data flows, and what improvements this version makes over the first try.

1. Design Goals

The second-try graph was designed with three goals:

  1. Faithfully represent the original NavGPT paper — every LLM chain in the paper (planner, orchestrator, history summarizer, observation summarizer) has a corresponding prompt+LLM node pair on the canvas
  2. Use visible state — navigation history and visited viewpoints are tracked in explicit state containers rather than hidden internal variables
  3. Stay within the existing node type system — use only promptTemplate, llmCall, vlmCall, and the registered navgpt__* / env_habitat__* nodeset nodes

3. Node-by-Node Rationale

3.1 Seed Nodes (Run Once)

Node Type Why it exists
Habitat: Episode Info env_habitat__episode_info Provides the natural language instruction for the episode. In the original NavGPT, this comes from the R2R annotation file.
Habitat: Observe env_habitat__observe Gets the initial RGB observation at the starting viewpoint. This feeds the first iteration of the loop.
Habitat: Localize env_habitat__localize Gets the agent's initial position and heading. NavGPT uses position/heading to compute relative angles for navigable viewpoints.
Skill: Action Format navgpt__action_output_format Injects the action output format as a system prompt for the orchestrator LLM. Tells the LLM how to format its Action: / Action Input: output so the parser can extract viewpoint IDs.

3.2 Planner Chain (Run Once)

Node Type Why it exists
Planner Prompt promptTemplate Implements NavGPT's PLANNER_PROMPT — takes the full instruction and asks the LLM to decompose it into numbered action steps with visual verification landmarks.
Planner LLM llmCall Generates the action plan. Uses temperature=0.0 for deterministic output. Writes result to the plan state entry via write_to_state.

Why a separate planner? The original NavGPT paper found that decomposing the instruction into sub-goals before navigation significantly helps the orchestrator track progress. The planner runs once before the loop starts, and the plan is available to the orchestrator at every step via graph state.

3.3 Loop Nodes

Iteration Gate

Node Type Why it exists
Iteration Gate iterIn Entry point of the dataflow loop. Receives RGB and state from the previous step (or from seeds on the first iteration). Paired with iter_out — when iter_out fires, data flows back to iter_in for the next step.

Scene Processing (3 nodes)

Node Type Why it exists
VLM Scene Caption vlmCall Converts the RGB observation into a textual scene description. This is the AgentCanvas equivalent of NavGPT's pre-computed observations_list_summarized/ data — in the original paper, BLIP-2 captions were pre-generated offline; here we run a VLM in the loop for flexibility.
Obs Summary Prompt promptTemplate Takes the detailed VLM caption and asks for a one-sentence summary. Mirrors NavGPT's obs_summarizer.py script which used GPT-3.5 to condense scene descriptions.
Obs Summary LLM llmCall Generates the one-sentence summary. This compact representation is used in two places: (1) the orchestrator prompt's {summary} slot, and (2) the history update chain.

Why both a full caption AND a summary? The original NavGPT maintains two representations:

  • Detailed observation (obs['detail']): full 8-directional scene descriptions fed to the orchestrator for decision-making
  • Summary observation (obs['summary']): one-sentence summary used in navigation history to keep the history compact

This dual-representation prevents the history from growing too large while giving the orchestrator rich scene context for the current step.

History Formatting

Node Type Why it exists
Format History navgpt__format_history Reads the accumulated navigation history from state and formats it as a numbered list of previous steps. max_entries=20 prevents the history from exceeding the LLM context window.

Why a custom node? History formatting involves reading from state containers and applying truncation logic. This is registered as a custom navgpt__format_history handler rather than a generic prompt template because it needs programmatic access to the state accumulator.

Orchestrator Chain (2 nodes)

Node Type Why it exists
Orchestrator Prompt promptTemplate The core reasoning prompt — closely mirrors NavGPT's VLN_GPT4_PROMPT. It receives 7 inputs: {plan}, {history}, {scene}, {summary}, {pos}, {heading}, {instruction}, {step}. The prompt instructs the LLM to: (1) check if destination is reached, (2) if not, select the next viewpoint based on the plan and scene.
Orchestrator LLM llmCall Generates the reasoning trace + action. Uses temperature=0.0 for deterministic navigation. The system prompt slot receives the action format skill, which tells the LLM to output in Thought: ... Action: action_maker Action Input: "viewpointID" format.

Why this prompt structure? The prompt follows the ReAct (Reasoning + Acting) pattern from the original NavGPT paper:

  1. The LLM first reasons about its current situation in a Thought: section
  2. Then it outputs an Action: (tool name) and Action Input: (viewpoint ID)
  3. The output parser extracts the viewpoint ID for execution

This explicit reasoning is NavGPT's key contribution — it makes the decision process transparent and debuggable.

Action Execution (2 nodes)

Node Type Why it exists
Parse Action navgpt__parse_action Extracts the viewpoint ID from the LLM's output using the regex Action\s*:.*Action\s*Input\s*:.*([a-fA-F0-9]{32}). Handles parsing errors gracefully. Also detects "Final Answer: Finished!" to trigger termination.
Habitat: Step env_habitat__step Executes the action in the Habitat simulator — moves the agent to the selected viewpoint and returns the new RGB observation, state, done signal, and metrics.

History Update Chain (2 nodes)

Node Type Why it exists
History Update Prompt promptTemplate Implements NavGPT's HISTORY_PROMPT — takes the previous history, the action just taken, and the new observation summary, then asks the LLM to append a concise one-line summary of this step.
History Update LLM llmCall Generates the updated history string. Writes to navigation_history state via write_to_state, making it available to format_history on the next iteration.

Why LLM-based history instead of simple concatenation? The original NavGPT paper offers both approaches:

  • Simple mode (use_history_chain=False): concatenate angle + viewpoint + summary as a string
  • LLM mode (use_history_chain=True): use a dedicated LLM chain to summarize and compress the history

The LLM mode produces more compact, informative history entries. In the second-try graph we use the LLM mode because it better represents the paper's full architecture. The first-try graph omitted this chain entirely.

Loop End

Node Type Why it exists
Iteration End iterOut Collects outputs from the current step (RGB, state, action, response) and feeds them back to the paired iter_in for the next iteration.
Termination termination Receives the done signal from env_step. Stops the dataflow loop when the agent issues a STOP action or the environment signals episode completion.

3.4 Output Viewers

Node Type What it shows
Output outputPort Aggregates final outputs (RGB, state, action, done, metrics) for downstream consumers or composite node ports.
Observation imageViewer Displays the current RGB image from the agent's viewpoint — lets you see what the agent sees.
Thinking Log thinkingLog Streams the orchestrator LLM's reasoning trace — shows the Thought: section where the agent explains its decision.
Action Log actionLog Shows the sequence of parsed actions (viewpoint IDs) the agent has taken, with step numbers.
Metrics metrics Displays VLN evaluation metrics (SPL, SR, nDTW, SDTW) computed by the environment after each step.

4. Data Flow Walkthrough

A single navigation step flows through the graph as follows:

Step 0 (Before Loop)

  1. Episode Info fires → provides instruction text
  2. Planner Prompt receives instruction → fills template
  3. Planner LLM generates action plan → writes to graph_state.plan
  4. Observe fires → produces initial RGB
  5. Localize fires → produces initial position/heading state
  6. Both feed into IterIn as seed data

Step N (Each Loop Iteration)

  1. IterIn emits: RGB (from previous step or seed), state, response, action
  2. VLM Caption receives RGB → generates detailed scene description text
  3. Two parallel paths:
    • VLM output → Obs Summary Prompt → Obs Summary LLM → one-sentence summary
    • VLM output → directly to Orchestrator Prompt as {scene}
  4. Format History reads accumulated history from state → formatted text
  5. Orchestrator Prompt assembles all inputs: plan + history + scene + summary + position + instruction
  6. Orchestrator LLM reasons and outputs: Thought: ... Action: action_maker Action Input: "abc123..."
  7. Parse Action extracts viewpoint ID abc123...
  8. Habitat: Step moves agent to that viewpoint → returns new RGB, state, done, metrics
  9. History Update Prompt combines: old history + action + new summary
  10. History Update LLM generates updated history → writes to graph_state.navigation_history
  11. IterOut collects new RGB + state + action + response → feeds back to IterIn
  12. If done=true: Termination stops the loop

Termination

The loop ends when either:

  • The orchestrator outputs "Final Answer: Finished!" (parsed by navgpt__parse_action → triggers STOP action → done=true)
  • The environment signals episode completion
  • step_budget (500) is reached

5. State Design

5.1 Graph State (Auto-Accessible)

The graph_state container is accessible by all nodes without explicit state edges. It holds the core navigation state:

Entry Reducer Value Type Purpose
action_history accumulator ACTION Full history of all actions taken (viewpoint IDs)
step_count counter ANY Current step number (incremented at each IterOut)
instruction lastWrite TEXT The episode instruction (written once by episode_info)
plan lastWrite TEXT The decomposed action plan (written once by planner LLM)
navigation_history lastWrite TEXT LLM-summarized history text (updated each step by history update LLM)

Why lastWrite for navigation_history? Unlike action_history which accumulates raw action IDs, the navigation history is a rewritten text document — the history update LLM produces a complete updated history each step, replacing the previous version. This matches NavGPT's self.agent_executor.agent.history list which is appended to then serialized as a whole.

5.2 Navigation Memory (Visible Container)

A visible StateContainer rendered on the canvas with a dashed border and database icon:

Entry Reducer Value Type Purpose
visited_viewpoints accumulator TEXT Accumulates every viewpoint ID the agent has visited — used to avoid revisiting
scene_descriptions accumulator TEXT Accumulates scene descriptions at each viewpoint — available for back-trace reasoning

Connected via state edges to:

  • env_step — writes visited viewpoint IDs and scene data after each action
  • orchestrator_prompt — reads visited viewpoints to include in reasoning context
  • format_history — reads accumulated data for history formatting

Why a separate container? The original NavGPT tracks visited viewpoints to avoid revisiting (the VLN_GPT4_PROMPT says "avoiding revisiting viewpoints by comparing current navigable and previously visited IDs"). Making this a visible state container means:

  1. You can see the accumulated viewpoints on the canvas during execution
  2. It's separate from the ephemeral navigation history — viewpoint IDs are structured data, not prose
  3. It maps directly to NavGPT's self.traj[0]['path'] accumulation

6. Improvements Over First Try

Aspect First Try (navgpt_ce) Second Try (navgpt_second_try)
Nodes 20 24 (+4)
Edges 26 35 (+9)
Observation summarizer Missing — VLM caption used directly everywhere Present — separate obs_summary_prompt → obs_summary_llm chain mirrors obs_summarizer.py
History update Missing — history managed internally by format_history node Present — dedicated history_update_prompt → history_update_llm chain mirrors NavGPT's HISTORY_PROMPT
State entries 4 (action_history, step_count, instruction, plan) 5 (adds navigation_history for LLM-maintained text)
State containers None (only graph_state) 1 visible "Navigation Memory" container with 2 accumulators
State edges 0 3 (connecting env_step, orchestrator, format_history to Navigation Memory)
Orchestrator prompt 7 template variables 8 template variables (adds {summary} from obs summarizer)
Planner prompt Basic decomposition Enhanced with landmark verification pattern
VLM prompt Generic scene description Navigation-focused: emphasizes spatial layout, doors, corridors
LLM chains 2 (planner, orchestrator) 4 (planner, orchestrator, obs summarizer, history updater)

What stayed the same

  • Same node types — no new custom handlers needed
  • Same iteration pattern — IterIn/IterOut with paired IDs
  • Same termination logic — done signal from env_step
  • Same output viewers — observation, thinking, action, metrics
  • Same nodesets required — env_habitat + NavGPT custom nodes

What's different from the paper

  1. VLM instead of pre-computed text — NavGPT uses offline BLIP-2 captions stored as JSON; the canvas graph runs a VLM in-loop for flexibility with any environment
  2. No back_tracer tool — the original NavGPT has two tools (action_maker + back_tracer); the canvas graph uses single-action mode like the paper's GPT-4 configuration (use_single_action=True), which is what achieved the best results
  3. Continuous environment — NavGPT operates on discrete Matterport3D viewpoints; the canvas graph targets Habitat-CE which has continuous movement (hence step_budget=500 instead of 10)
  4. 8-direction text vs single image — NavGPT receives 8 pre-computed directional descriptions per viewpoint; the canvas graph gets a single RGB image and uses VLM captioning
