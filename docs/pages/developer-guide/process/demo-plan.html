Demo Plan — Showcase vs Tutorial

Proposal for the demo portfolio the project should build. Demos fall into two categories with very different design constraints — showcase (grab attention in 10–30 seconds) and tutorial (walk a researcher through a full loop in 5–30 minutes). This page is the living plan; each demo gets promoted into tutorials/ or a standalone landing asset once it is produced.

1. Design Tradeoff

Axis Showcase Tutorial
Audience Hasn't read the README — reviewer, potential collaborator, X scroller Has decided to try it — reads carefully
Length 10–30 s visible payoff 5–30 min end-to-end
Voice Visual impact, minimal text Step-by-step, every step has a "why"
Win condition Stars / shares / lead capture Completion rate + reproducibility
Kiss of death Front-loaded setup Hand-waving, skipped steps

The project's nine capabilities (see capabilities/) are the pool of ideas to hit: ① graph-expressible agents · ② customizable node system · ③ isolated runtime envs · ④ nested graphs · ⑤ graph executor · ⑥ real-time observability · ⑦ hook system · ⑧ batch evaluation & job queue · ⑨ visual canvas editor. Each demo should land on at least one of these explicitly.

2. Showcase Demos

Four demos, each targeted at one distribution surface (hero GIF, tweet video, paper teaser, architecture poster).

2.1 Drag three nodes, watch an agent navigate

Capability target: §1 Graph-Expressible Agents + §6 Real-Time Observability

Empty canvas. Drag env_habitat__observe + llmCall + env_habitat__step + iterIn / iterOut out of the sidebar. Connect. Press Play. The right panel renders live Habitat RGB while the agent drives. One unbroken screen-recording, ~20 seconds.

Why this one first: it's the cleanest expression of one JSON = one agent = one graph. No code, no config files, no batch harness — just a canvas.

Distribution surface: hero GIF on index.md + README top.

2.2 Live state + multi-panel observability

Capability target: §6 Observability + state container system (ADR-dataflow-002, ADR-dataflow-004)

Run a MapGPT-style graph where a StateContainer holds the accumulated topo-map. Four viewer sinks on screen at once: RGB, depth, topo-map rendered from the container, textScroll of the LLM's thought stream. All four update in lock-step on every iterOut boundary.

Why this one: a poster-worthy still frame and a hypnotic 60-second video. Most competing agent frameworks show a log tail or a single image; four synchronized panels drives home that observability is a first-class concern.

Distribution surface: landing-page tweet + arXiv companion page header.

2.3 Swap one node, different agent

Capability target: §2 Customizable Node System + decoupling

Same graph; swap llmCall from GPT-4o to Claude Sonnet (just change the model on the node's inline config) — re-run. Then swap the policy node from policy_cma__forward to policy_hamt__forward — re-run. No wires touched. Show a small bar chart of SPL on val_unseen across the three variants.

Why this one: reviewers love comparisons, and the framing "change one block, rerun, compare" is exactly the argument for the framework. Doubles as a mini ablation.

Distribution surface: paper teaser figure + blog post cover image.

2.4 Save graph as node, drop into another graph

Capability target: §4 Nested Graph System (ADR-dataflow-001, ADR-canvas-003)

Open a working NavGPT graph. Select all → "Save as Graph Node" → name it NavGPTSubagent. Open a larger "high-level planner" graph. The NavGPTSubagent appears in the Explorer sidebar. Drag it onto the canvas — it shows up as a single block with the PortIn/PortOut ports exposed. Wire it up. Run. Same behaviour.

Why this one: the nested graph system is the most architecturally interesting feature and the hardest to convey in a still image. A live drag-and-drop sells it in 15 seconds.

Distribution surface: "advanced architecture" callout on the landing page; reused inside F12 AAS tutorial.

3. Tutorial Demos

Four tutorials organized "from user to author", each aligned with one existing capability doc and ending in a working artifact the reader can commit back.

3.1 Your first agent in 5 minutes

Capability target: §1 Graph-Expressible Agents + §6 Observability

  • Install via scripts/install/ + run_dev.sh
  • Open workspace/graphs/straightforward.json
  • Edit the prompt in promptTemplate → change one phrase
  • Select val_unseen split + episode 0 in the EnvContextBar
  • Press Play; watch SPL in the metrics sink at the end

Takeaway: the reader has run a real VLN-CE agent without writing code.

3.2 Build a custom node in Python

Capability target: §2 Customizable Node System (ADR-components-004 Python-driven UI)

  • Create workspace/nodes/word_count.py with a WordCountNode(BaseCanvasNode) subclass
  • Declare input_ports = [PortDef("text", "TEXT")], output_ports = [PortDef("count", "METRICS")]
  • Add a ui_config = NodeUIConfig(color="amber", config_fields=[...])
  • Hot-reload via POST /api/components/reload
  • Drag the node onto a canvas, wire it, run

Takeaway: one Python class = one visible canvas node. No frontend code touched.

3.3 Wrap a simulator in server mode

Capability target: §3 Isolated Runtime Environments (ADR-server-001)

  • Build a minimal MiniSimNodeSet(BaseNodeSet) with two tools (reset, step) in workspace/nodesets/env/env_mini_sim.py
  • Set server_python = "/path/to/other/env/python"
  • Load via the NodeSet Manager page with mode=server
  • Observe: framework spawns a subprocess, fetches /manifest, generates proxy canvas nodes automatically
  • Confirm the proxy nodes are indistinguishable from local nodes

Takeaway: any tool that needs a different Python env can still appear on the canvas. Zero server-side code written by hand.

3.4 Run batch eval with the worker pool

Capability target: ADR-eval-002 Worker-Pool Batch Eval

  • Open the Eval page
  • Pick the graph from §3.1; pick val_unseen split; set worker_count=4
  • Launch; watch four per-worker lanes render in parallel
  • Export the resulting EvalRun JSON

Takeaway: the same canvas graph runs at 4× throughput under batched evaluation — no graph-side changes, tagged env subprocesses handled invisibly.

5. Roadmap Alignment

The tutorial demos directly serve three open TODO items:

  • TODO #22 Install instructions — 3.1 forces us to finish this.
  • TODO #41 Per-category contributor guide — 3.2 and 3.3 become the per-category pilots.
  • TODO #48 Validate /implement-graph pipeline end-to-end — 3.2 + 3.3 in sequence is exactly this exercise.

The showcase demos piggy-back on existing graphs (straightforward.json, NavGPT-CE from TODO #6) — they do not block on new features.

6. Production Order

Slot Demo Rationale
1 Showcase 2.1 + Tutorial 3.1 (paired) Share the same straightforward.json; one screen-recording session produces both assets
2 Showcase 2.2 Highest "wow per second" for social distribution; uses the existing MapGPT graph
3 Tutorial 3.2 Fastest tutorial to produce; also unblocks TODO #41
4 Showcase 2.3 Needs a second LLM profile + HAMT (M5) — gate by whichever lands first
5 Tutorial 3.3 Requires writing a toy sim; medium effort
6 Showcase 2.4 Needs a polished NavGPT graph (TODO #6) first
7 Tutorial 3.4 Hangs off the existing Eval page; smallest new content

The 1 + 3 pairing is the best ROI: one recording run, two published assets.
