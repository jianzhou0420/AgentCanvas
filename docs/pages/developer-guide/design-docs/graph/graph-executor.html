1. The drawn graph and its contract

A canvas graph is nodes joined by typed wires. Its execution semantics rest on exactly two things: a firing rule that decides when any node runs (§1.1), and one recognised structure — the loop pivot pair — that expresses iteration, halting, and the episode verdict (§1.2). Everything else on the canvas is ordinary dataflow. This section states the contract; §2–§5 describe the algorithm that honours it.

1.1 The firing rule

The firing rule. A node fires when every required input port holds a value. The fire consumes those inputs and produces an output dict; the engine writes each output onto its outgoing wires, filling downstream ports — which may make further nodes ready. Fire → propagate → repeat, until nothing is ready.

Three consequences follow, and they shape everything below:

  • Order is derived, not authored. There is no step list; whichever node is ready runs. A branch that never receives data never runs, so conditional paths need no special construct.
  • Inputs are consumed. A fire empties the ports it read; the node fires again only when fresh values arrive. One delivery is one firing opportunity, not a standing subscription.
  • No drawn cycles. The graph is acyclic as drawn — a wired cycle would give the data no valid order. Iteration is expressed instead by the pivot structure of §1.2 (ADR-executor-001).

1.2 Anatomy of an agent loop

Plain dataflow ends when data stops flowing — enough for a pipeline, not for an agent that must observe, act, and observe again. Iteration is expressed by the one structure the engine recognises: a paired IterIn / IterOut, the loop pivots. The smallest agent loop:

Reading the figure in run order:

  1. Run start. The entry node fires once — it needs no inputs — and its init_* edges hand the run-invariants (instruction, episode config) to the IterIn's init side.
  2. Each iteration. The IterIn emits the loop-carry bundle; the body — here predict and env_step, ordinary nodes with no special treatment — computes; the IterOut collects the next carry. The engine hands the carry back to the IterIn by pairing: the top edge is not a wire you draw, and the drawn graph stays acyclic.
  3. Halting. Any BOOL wired into the IterOut's stop input — here the env's done — ends the loop after the current iteration. Left unwired, the loop runs to its step budget.
  4. Termination. The IterOut's final_* handles mirror its carry ports (plus a constant final_stop). They are real wires, but deliver exactly once — the terminal iteration's values.
  5. Verdict. Everything downstream of final_* — here evaluate and graphOut — is the after-loop band: it runs once, after the loop, and may not take inputs from the loop body.

2. Where the engine sits

One class owns everything this page describes — GraphExecutor, entered through a single method, run(graph, session, …). The layers above it (HTTP endpoint, LoopRunner) own lifecycle — pause, stop, resume — and never look inside the graph; the layer below (the node handlers) computes and never schedules. The executor is the only place that decides what fires when.

A fresh executor is built per run, so no state leaks between episodes. For parallel batch eval it exposes two per-worker overrides — get_env_panel(name) and get_server_url(nodeset) — so each eval worker talks to its own env subprocess (ADR-eval-002); outside batch eval they pass through.

Inside run(), every run has the same three-phase shape — the outline of the rest of View 2, and the engine-side mirror of §1.2's three places: pre-loop builds live state and primes the queue (§3) · in-loop fires until a halt (§4) · after-loop delivers the verdict and finalises (§5).

3. Pre-loop — build & discover entries

Before a single node fires, run() does a deterministic, once-per-run setup in two movements. BUILD turns the static graph JSON into live, mutable runtime state; ENTRY DISCOVERY finds the entry nodes — the nodes that can fire with no inputs — and primes the queue with them. Nothing here loops — it is pure preparation, identical on every run.

3.1 BUILD — compile the graph into live state

Seven steps, strictly in order — plain top-to-bottom code, no concurrency. Each step turns one part of the authored GraphDefinition into something the loop can read or mutate. (Run-start args need no build step of their own: they arrive over ordinary canvas edges into the iterIn's init_* handles — ADR-dataflow-008.)

3.2 ENTRY DISCOVERY — find the entry nodes, prime the ready-queue

An entry node is discovered automatically — no author marking — as any node satisfying all three structural conditions; every entry node is queued at run-start:

  • not an iterIn — IterIn is the loop's entry; it fires only when its init_* edges deliver (iter 0) or via the IterOut transfer (iter 1+), never via entry discovery;
  • no incoming edges — nothing upstream will ever fill it;
  • no required input ports — it can fire on nothing.

validate_graph_connectivity (in graph_def.py) rejects required-but-unwired ports at load time, so the third condition only ever excludes well-formed leaf consumers — it never silently drops a real entry node. With the queue primed, three things arm the loop:

  • run_start signal — state containers with lifetime="run" reset to a clean baseline;
  • the GraphStart hook fires;
  • max_firings = step_budget × nodes × 3 is computed — a runaway-loop backstop set far above any real run.
Sharp edge. IterIn is never an entry node (excluded by type). A loop is entered only via the init-side edges (upstream producers wired into the IterIn's init_* handles) or via an entry node that fills IterIn's iterout_* slots on iteration 0. A loop whose IterIn has no incoming init edge will never start — the body simply starves.

4. In-loop — the firing loop

With the queue primed, the engine starts turning. This chapter walks one turn from the inside out — the loop skeleton (§4.1), the guards before each fire (§4.2), the IterOut boundary that closes every iteration (§4.3), the propagation path every other node takes (§4.4) — then reassembles the pieces into the whole-run flowchart (§4.5) and a concrete trace (§4.6).

4.1 One iteration & the loop skeleton

Keep §1.2's graph in mind — entry node, two pivots, a two-node body; this chapter is what the engine does with it. The main loop is deliberately tiny. Each turn:

  1. Gate. Run while the queue holds a node and total firings stay under the runaway cap (while ready_queue and total_firings < max_firings); honour pause; a user Stop breaks out.
  2. Pop. Take the front of the FIFO ready-queue.
  3. Guard, then fire. Two structural guards (§4.2), then _fire_node — the per-node fire path (graph_executor.py).
  4. Fork. An IterOut runs the once-per-iteration boundary (§4.3); every other node takes the ordinary propagation path (§4.4).

Nothing else is special: an agent's body is plain dataflow firing, and IterOut is the only node type the engine schedules differently.

4.2 The guards before a fire

A popped node is not fired blindly — two cheap structural guards run first, in order; each either skips the node or readies state for the fire:

  1. Scope barrier. A node belonging to an inner scope that has already terminated is frozen (pending_inputs cleared, fire skipped) until that scope re-enters — otherwise in-flight inner-body nodes fire one extra time after termination and leak per-iteration values through cross-scope wires. Exempt: iterIn / graphIn, the cascade that reopens the scope.
  2. IterIn step_start. An iterIn broadcasts step_start (per scope) so lifetime="step" state resets to a clean slate for the iteration about to run; a re-fired iterIn whose inner scope was terminated resets that scope's terminated flag + counter — the multi-scope re-entry path.

Then the node fires. A StopExecution raised by the fire halts the run immediately; any other exception is published to the ErrorBus and the node's result becomes {"error": …} so downstream nodes can still react (in AGENTCANVAS_STRICT_ERRORS mode it re-raises and fails fast instead).

(Until 2026-06-11 there were four guards. The termination-defer and post-loop-skip guards were deleted with the termination node and config.post_loop — the stop signal and the after-loop stage now live on the iterOut pivot itself, so there is nothing left to defer or skip.)

4.3 The IterOut boundary — four phases

IterOut fires exactly once per iteration, so the engine hangs every per-iteration chore on it. The order of the four phases is load-bearing — in particular Settle runs before Decide:

What each phase does, and the subtle rule inside it:

  • ① Record. Advance the per-scope step counter (the root scope's counter also mirrors into self.step_counter + session._current_step); broadcast step_end; checkpoint all state containers (root scope only); flush the execution log; broadcast the consolidated nav_step (root only). IterOut's own outgoing edges are not routed per iteration — they are final-side handles, emitted once at termination (③).
  • ② Settle. Drain this iteration's leftover sinks (bounded at 64, restricted to the iterOut's own scope, skipping only iterIn). This is where in-loop viewer / telemetry sinks scheduled in this iteration's wave still emit — including on the terminal step. One exception to the scope filter (2026-07-04): a root boundary also drains graph-scope nodes — dead-end sinks are never on a path to any iterOut, so scope analysis leaves them in the graph scope, and without the exception they would miss the terminal step entirely (the run exits with them still queued). Mid-loop this only fires them earlier than the main loop would have; inner boundaries keep the strict filter.
  • ③ Decide. Read the just-fired iterOut's own stop input: truthy means this scope is done. A root-scope stop halts the run; an inner one ends only that loop. Then the per-scope step_budget exhaust check. Either way, termination triggers the final-side emission: the terminal iteration's collected values go out once on the final_* handles (root → seeds the after-loop stage, §5; inner → feeds outer-body consumers and the graphOut latch flush). The stop signal sits on the scope's own iterOut, so it is structurally bound to the right loop and read exactly once per iteration.
  • ④ Hand off. Sleep step_delay_ms (root only), then write each IterOut output X into the paired IterIn's iterout_X slot and re-enqueue the IterIn — arming the next iteration. (stop is not loop-carried, and the final-side handles never fire here — only at termination.)
What the final side bought. Before 2026-06-11 the episode verdict came from an in-loop evaluate whose terminal-step fire depended on the settle drain catching it — the SIMPLER / smartway metric-drop bug class (roadmap #64: 5/20 successes recorded as 0). Now the verdict consumes final_* values handed over by the engine at termination, so it is correct by construction; in-loop sinks are telemetry only.

4.4 Ordinary nodes & propagation

Every non-IterOut fire ends the same way, in four moves:

  1. Consume. The node's pending_inputs are cleared — inputs are used up by the fire.
  2. Special cases. An iterIn additionally drops its persist=false slots, so one-shot values emit once and are gone. An inner-scope graphOut buffers its value into a latch (state["latched_value"]) instead of propagating outward — the latch is flushed to outer-scope consumers only when its scope terminates.
  3. Route. For every other node the engine walks its outgoing edges and writes each output onto the target's input port (_route_value_to_port). This is also where LIST[T] coercion happens (ADR-027): a scalar producer feeding a list consumer is auto-wrapped, and fan-in is concatenated in edge order.
  4. Enqueue. Any target whose required inputs are now all present joins the ready-queue.

(Run-start args need no special branch: the iterIn's init_* handles are fed by ordinary canvas edges — ADR-dataflow-008 two-sided iterIn.)

4.5 The whole run, end to end

Zoom out from one iteration to the whole of run(). The flowchart below stitches §3 and §4 into one picture, with the ready-queue at its centre — ENTRY DISCOVERY primes it, every turn pops it, and both return paths feed it again. Read it top-down:

  1. Set up once. BUILD compiles the graph JSON into live state (§3.1); ENTRY DISCOVERY finds the no-input entry nodes and primes the ready-queue (§3.2).
  2. Turn the loop. While the queue has a ready node — and neither the global firing cap nor a user Stop has hit — pop the front one and fire it: guards first (§4.2), then _fire_node.
  3. Fork on what fired. An ordinary node takes the fast path: route its outputs, enqueue whatever became ready (§4.4), turn again — this is almost every fire. An IterOut instead runs the once-per-iteration boundary (§4.3): Record, Settle, Decide, Hand off. Unless Decide halts, Hand off arms the paired IterIn and the loop turns again — that is one full iteration.
  4. Exit. Two doors out. Decide halts — the stop signal or the step budget — and the terminal iteration's values leave exactly once on the final_* handles. Or the main loop runs dry — queue drained, firing cap, user Stop — and the fallback reconstructs the final-side values best-effort. Every door lands in the same after-loop + finalise stage (§5); §7 tables every exit, including the StopExecution escape hatch.

That is the entire control flow — and the engine-view counterpart of §1.2: there the boxes were the nodes of one drawn graph; here they are the engine's actions on any graph. Part II §6 prints this same shape as code — the colored region bands there are exactly these boxes.

5. After-loop — the verdict stage & finalise

However the loop exits — the stop signal fired, the step budget hit, the queue drained, or a node crashed — control lands in one finalise stage. It has four movements: the after-loop pass (§5.1), the node-error conviction gate and clean finalisation (§5.2), and the error path (§5.3).

5.1 The after-loop pass

The after-loop band is every node wired downstream of the root iterOut's final_* handles — the verdict chain, typically evaluate → graphOut, possibly with adjudicator steps in between. Two facts keep it dormant while the loop runs: the final side never emits mid-loop, and the validator forbids edges from the loop body into the band. Band nodes therefore cannot become ready before termination — there is nothing to skip, defer, or flag.

When the final-side values arrive, _after_loop_pass drains whatever they made ready: fire a ready node, route its outputs, enqueue the consumers that became ready in turn. A node whose upstream band node hasn't fired yet is re-queued behind it, so a chain resolves in dependency order without a topological sort; the queue is bounded (max(nodes×4, 64)), so a never-satisfiable node cannot spin forever.

Two rules keep the pass honest:

  • Graph-scope only. The loop is over, so re-firing a loop-body node — or any iterIn / iterOut, or any node inside an author (loop) scope — is meaningless and barred. Only graph-scope nodes (the band and its sinks) fire.
  • graphOut is harvested, not fired. A graph-scope graphOut the chain reaches is routed-to and re-snapshotted into state["_last_inputs"] — the channel the metric harvest reads — but never itself fired; inner-scope graphOuts are skipped entirely (their latch flush is scope-bound).

Membership is topological, not declared: there is no flag to set (the former config.post_loop was removed 2026-06-11) — a node is after-loop because it hangs off the final side. The input contract is engine-guaranteed: the band receives exactly the terminal iteration's values ("verdict inputs ride the pivot" — anything the verdict needs must be an iterOut port, consumed via final_<name>). If the loop ends without a boundary emission (drained queue, max_firings, user stop, or a crash), _emit_final_fallback reconstructs the last completed iteration's values from the paired iterIn's iterout_* slots and emits best-effort before the drain.

5.2 Clean finalise

The conviction gate comes first (2026-07-04). At fire time the executor records every node failure — a raised exception, or an {"error": ...}-shaped result such as a server-mode proxy surfacing an HTTP 500 — into a per-run node_errors ledger (nodes that declare a legitimate error output port are exempt; returned errors also emit a NODE_RESULT_ERROR bus event). If the ledger is non-empty here, a NodeErrorAggregate naming the culprits is raised and control transfers to the error path (§5.3). The placement is deliberate: the verdict stage (§5.1) has already run, so final-side metrics are collected either way — but the run can no longer masquerade as completed while its downstream silently starved (the old status="completed" / step_count=0 eval pathology).

On a clean exit (empty ledger), in order:

  1. Harvest metrics — from any node's state["metrics"]; a value stored on session._metrics overrides node values.
  2. Broadcast run_end — lifetime="run" state containers clear.
  3. Broadcast nav_complete — final step and metrics, to the frontend.
  4. Fire the GraphComplete hook.
  5. Flush the execution log. Session status becomes done.

5.3 The error path

If the loop body raises — or the conviction gate fired (§5.2) — the finalise stage still runs, and runs the verdict first:

  1. Fallback emission + after-loop pass, best-effort. The ordering is deliberate: even after a crash the verdict evaluate can often still emit fresh metrics from the env's real final step, so the harvest reports the true result rather than a stale pre-crash value. The pass is tolerant — a band node that raises because the env is already torn down is logged and skipped (and both stages are idempotent, so re-entry from the conviction gate is a no-op).
  2. Publish the exception to the ErrorBus — GRAPH_CRASH for a real crash, NODE_ERRORS for a conviction; session status becomes error, broadcast to the frontend as a nav_status error.
  3. run_end still fires, carrying the error — run-lifetime state resets even on a crash.
  4. Fire the GraphError hook, then the final log flush.
