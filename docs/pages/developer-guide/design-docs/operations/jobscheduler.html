JobScheduler

submit → queue → measured resource admission (VRAM + RAM) → setsid spawn → reap · ResourceStatsTracker attribution + calibration · reservation decay · per-run quota cap

The JobScheduler is the backend's single gatekeeper for batch-eval runs: POST /api/eval/v2/start only queues a job; a 1-second tick loop decides when it may start, spawns it as an isolated process group, watches it, and accounts for GPU and system memory with real measurements rather than trusted declarations. Its hard promise is that every admission decision is explainable from one GET /api/eval/v2/queue snapshot — the per-resource charges (admit_charges), where they came from (admit_basis), the measured free per resource, and the standing reservations are all in the response. For the run that executes inside the spawned subprocess see batch-eval; for how auto_host server processes work see server-mode.

One scheduler, one closed loop. Submit persists intent to disk and queues; the tick loop observes the machine (GPU per-PID via the sampler, RAM per-PID via one /proc walk), reaps finished children (sedimenting what it measured into the calibration store), and admits the next job by comparing its calibrated per-resource estimates against measured free minus standing reservations — one inequality per measurable resource. The spawned subprocess's own usage flows back into the next observation — closing the loop.

1. What it does

One machine, one GPU, one pool of RAM, many uncoordinated eval jobs — the scheduler is the only place they meet. Every batch eval in AgentCanvas goes through it: submitting a run writes its full spec to disk and puts it in a FIFO queue; nothing executes until the scheduler's once-per-second tick decides the machine can take it. The decision is made against the measured machine: a ResourceSampler reads per-process GPU memory once a second, one /proc walk reads every process's RSS, and a ResourceStatsTracker attributes each PID to a shared model server, a running job's process tree, or "external" (anything the scheduler doesn't own — another session's backend, Xorg, the OS itself), and admission asks the same question once per resource: does this job's estimated need fit in what is physically free, after subtracting what already-admitted jobs are still expected to claim? A resource the machine cannot measure is simply not gated — a CPU-only host has no VRAM dimension and gates on RAM alone.

The estimates come from a calibration store that the scheduler feeds by watching real runs (resource_stats.py · CalibrationStore). There is no offline profiling step and no pre-run probe: the first run of a graph admits on its declared fallback (or alone, if nothing was declared), and everything measured during that run — each shared singleton's peak, the job tree's peak at that worker count, on every resource at once — is written to outputs/system/resource_calibration.json when the job is reaped. From the second run on, admission charges the measured numbers, keyed to the nodeset's source hash so code edits automatically invalidate stale statistics. Pre-rename VRAM-only stores (vram_calibration.json, v1) are adopted and migrated on first load; their RAM slice refills on the next run per graph.

Once admitted, a run executes in its own OS process (app/eval_subprocess_main.py), in its own setsid process group — cancel is one killpg, and a crash cannot take the backend down. The subprocess additionally caps its own torch allocator at the admission charge (AGENTCANVAS_VRAM_CAP_MB), so a run that blows past its estimate OOMs itself rather than its neighbors (VRAM only — RAM has no enforceable per-process cap; see §6).

1.1 The smallest run

A graph name and an episode count; everything else defaults. The response is immediate — the run is queued, not started:

curl -X POST http://127.0.0.1:8000/api/eval/v2/start \
  -H "Content-Type: application/json" \
  -d '{"graph_name": "straightforward", "episode_count": 1}'
# → {"run_id": "20260704_200243", "status": "queued", "via_subprocess": true}

Within a tick or two the job admits (this graph is calibrated, so the charges are the measured per-resource estimates — not the profile's declaration), spawns, runs one episode, writes summary.json + _DONE, and is reaped; the calibration store gains one more observation.

2. Lifecycle of one job

A job is disk-first: every state transition is visible in outputs/eval_runs/{run_id}/ before or as it happens in memory.

  • submit (job_scheduler.py · JobScheduler.submit) — rejects declarations that exceed the machine's physical ceiling with ValueError → HTTP 400 (eval.py · start_eval_v2), so an impossible job fails loudly instead of pending forever. Then writes spec.json + shared_urls.json + an initial pending summary, bumps the shared-singleton refcounts, and enqueues. Shared model singletons were already loaded by the endpoint before submit (registry.py · ensure_shared_nodesets_for_graph) — which is why a calibrated estimate at admit time counts them at ~0 marginal.
  • admitted (JobScheduler._admit, §3) — picked FIFO when the gates clear.
  • running (JobScheduler._spawn) — Popen(python -m app.eval_subprocess_main, start_new_session=True); the child group is captured for cancel, stdout/stderr tee to the run dir, and the admission charge rides in as AGENTCANVAS_VRAM_CAP_MB. The tracker opens an observation window (resource_stats.py · note_job_started) covering the job's shared nodesets — including local in-process ones, via a graph-derived union (JobScheduler._graph_shared_nodesets).
  • reaped (JobScheduler._reap) — on child exit: terminal status from the rc × _DONE matrix below, ephemeral servers released, refcounts dropped (zero-refcount singletons auto-unload next tick), and the observation window sediments into the calibration store (resource_stats.py · note_job_finished).
child exit_DONE on diskterminal status
any rcpresentwhatever summary.json says (completed / error / cancelled)
any rcabsentaborted (external kill or crash mid-run; rc=-15 = SIGTERM)

3. The measured admission gate

Admission compares an estimate against physics, not a declaration against a constant. Per queued job, in FIFO order, JobScheduler._admit applies four gates:

  1. canvas lock — canvas Play holds the GPU; nothing admits.
  2. exclusive — an exclusive_gpu job waits for an empty machine, and blocks everything while running.
  3. the measured gate — estimate ≤ measured_free − Σ reservations, checked once per resource (VRAM, RAM); a resource that is unmeasurable on this machine is skipped. A job that doesn't fit is skipped, not blocking (smaller jobs behind it may admit first).
  4. fallback — when no resource is measurable (no sampler at all: unit tests), the gate reverts to the pre-2026-07 declared ledger: marginal_vram_mb ≤ usable − Σ declared.

The estimate is resolved per job at admit time (JobScheduler._resolve_estimate), against the registry's current loaded state:

Reservation decay closes the lazy-loading race. A just-admitted job holds a reservation equal to its full charge; models take a minute to load, so without it the next tick would see the GPU still free and over-admit. As the sampler sees the job's real allocation, the reservation shrinks — per resource, max(0, admit charge − measured tree) per running job (JobScheduler._pending_reservations) — until the ledger has fully handed over to measurement. 4. Measurement, calibration, estimation

The tracker turns two raw per-PID streams — GPU memory from the sampler, RSS from /proc — into per-nodeset and per-graph statistics on each resource, and the estimator turns those back into admission charges. All of it lives in one module, services/resource_stats.py, owned by the scheduler.

4.1 Attribution — whose memory is this?

Each tick, two per-PID usage lists — GPU PIDs from the latest sample (nvidia-smi pmon, which lists graphics-type contexts too — habitat/MatterSim render via EGL and are invisible to the compute-only query; resource_sampler.py · _gpu_processes) and every live process's RSS from one /proc walk (resource_stats.py · _proc_table) — are attributed with the same ancestor walk (resource_stats.py · _attribute_pid):

  1. shared nodeset server — the PID, or one of its ancestors, matches the registry's _auto_servers table;
  2. running job — the PID's PPID-ancestor chain reaches a job's subprocess (PPID walk, not pgid match: the job's replicated auto_host children setsid into their own groups but keep the parent link);
  3. external — everything else (other sessions' backends, Xorg, and on the RAM side the OS itself). Visible in the /queue snapshot; measured free already excludes it.

4.2 Sedimentation — writing the calibration store

While a job runs, its window records peaks on each resource: the job tree's total, and each of its shared singletons' server usage plus the concurrent worker pressure. On reap the window is written to outputs/system/resource_calibration.json (EWMA + observed max, per resource, per nodeset and per (graph, worker_count)). Two honesty guards: a window with zero attribution samples sediments nothing (no blind zeros when the sampler was down), and a resident server observed at 0 MB is recorded as a real "none of this resource" measurement — distinct from "never observed". The calibration key includes the nodeset's source-tree hash (content_hash.py · hash_nodeset_tree): edit the code, and its old statistics silently retire. Legacy v1 stores (vram_calibration.json, VRAM-only) are adopted on first load (resource_stats.py · _migrate_v1): their statistics survive as the vram slice, and the ram slice starts uncalibrated.

4.3 Estimation — GET /api/eval/v2/estimate

The estimator (resource_stats.py · estimate_tree_mb / estimate_shared_mb) prices a run per resource as: one-time bases for needed-but-unloaded singletons, a worker-pressure slope for loaded ones, the job tree at the requested worker count, plus the safety margin. Basis degrades explicitly: measured (exact worker count seen) → fitted (linear fit over ≥2 worker counts) → scaled (single point, proportional). For singletons there is one more rung below measurement: the author preset — a nodeset may declare expected_vram_mb / expected_ram_mb as ClassVars (bases.py · BaseNodeSet, e.g. a SAM nodeset shipping "≈600 MB"), which prices it (basis hint) whenever no valid measurement exists on this machine — fresh installs, or right after a source edit retired the measured stats. Presets are deliberately not hash-gated: weights dominate an FM's footprint, so the preset is a long-term assertion that survives code churn; measurement always wins once available. The honesty rule, applied per resource: estimate_mb is null unless every component is calibrated or preset — a missing singleton could be 10 GB, so a partial sum never masquerades as a total. The endpoint (eval.py · estimate_eval_v2) is read-only and also reports per-resource max_workers plus a combined one — the minimum across the resources measurable on this machine.

Everything admission used is in GET /api/eval/v2/queue (JobScheduler.list_active). usable_vram_mb / reserved_vram_mb (elided above) are the legacy declared ledger, kept for the no-sampler fallback and for clients that still read them; the decision fields are measured_free_mb and pending_reservation_mb (both per resource) and the per-job admit_charges / admit_basis pair. Cancel is POST /runs/{run_id}/cancel — queued jobs cancel immediately, running jobs get a SIGTERM to their whole process group.

6. Where it deviates from the model

Queued jobs do not survive a backend restart. The model says "disk-first, so state survives"; in reality reconcile_aborted_runs (job_scheduler.py) marks both running and pending runs as aborted at startup — the spec is on disk but nothing re-enqueues it (M1; queue durability is an acknowledged gap in the code comment).
Local in-process nodesets are calibrated at 0 by fiat, not by measurement. A parallelism="shared" nodeset loaded in-process (no auto_host server, e.g. env_adapter) has no PID of its own; per-PID attribution structurally cannot see it. It is recorded as 0 MB on every resource so it never blocks estimate coverage (resource_stats.py · _is_loaded_local) — a local nodeset that secretly loads a model into the backend process would be under-estimated.
The quota cap is best-effort, not a hard sandbox. AGENTCANVAS_VRAM_CAP_MB caps the torch allocator of the eval subprocess itself (eval_subprocess_main.py · _apply_vram_cap). Its child env servers, non-torch CUDA allocations, and EGL contexts are uncapped. The measured gate is the primary defense; the cap is a backstop.
RAM is gate-only — there are no RAM quota teeth. There is no RAM sibling of AGENTCANVAS_VRAM_CAP_MB: RLIMIT_AS counts virtual address space and would kill mmap-heavy simulators, and cgroup memory limits need privileges the backend doesn't assume. The defenses are the gate itself and the permanent floor held out of the budget (resource_stats.py · RAM_FLOOR_MB, 4 GB below MemAvailable).
Job-tree RAM over-counts shared pages. Per-process RSS counts copy-on-write pages once per process, so a forked worker tree's summed RSS exceeds its true footprint. Errs conservative for admission; PSS would be exact but costs a smaps_rollup read per process per tick.
A declared job's RAM rides ungated. The declared rung predates the RAM dimension — profiles only ever declared marginal_vram_mb — so when calibration is incomplete and a declaration exists, admission gates VRAM on the declaration and RAM not at all, until the first run calibrates both.
Attribution lags reality by up to ~5 seconds. Per-PID data refreshes on the sampler's slow cadence (resource_sampler.py: gpu_procs cache ~5s; totals 1s). Reservation decay therefore lags, which errs conservative — and a sub-5-second job can be reaped with an empty window (nothing sediments; guarded by samples_seen).
FIFO admission can starve a large job. A job that doesn't fit is skipped, not blocking (_admit walks on), so a continuous stream of small jobs can overtake a large one indefinitely. The priority field is accepted and persisted but ignored (M1 FIFO).
The VRAM ledger is a scalar. Totals and free are summed across all visible GPUs (detect_total_vram_mb, sampler totals); there is no per-GPU accounting or placement. Correct on this single-GPU box, wrong in both directions on a multi-GPU one.
Unseen worker counts are extrapolated, not measured. With one calibration point the tree cost scales proportionally (estimate_tree_mb, basis scaled); a fixed per-run overhead makes this over-estimate at high worker counts and the safety margin absorbs the reverse error. Real multi-worker runs upgrade the basis to fitted automatically.
