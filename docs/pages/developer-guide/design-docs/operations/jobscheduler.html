JobScheduler

Decides when an eval run may start: measure what is free, estimate what the job needs (from its past runs), admit only if it fits — for GPU and system memory alike.

The JobScheduler answers one question: is it safe to start this eval run right now? When you submit a run, nothing executes yet — the job waits in a queue. Once a second, the scheduler looks at how much GPU and system memory is actually free, estimates how much this job will need (learned from watching its previous runs), and starts it only if it fits. Everything it used to make that call is visible in one API response (GET /api/eval/v2/queue), so when a job doesn't start, you can always see why. For what happens inside a run, see batch-eval; for how model server processes work, see server-mode.

The whole system is one loop. Submitting writes the job to disk and puts it in the queue. Every second the scheduler does three things: it measures who is using memory right now (observe), it cleans up finished jobs and saves what they used (reap), and it decides whether the next queued job fits (admit). The job it starts becomes a new process — whose memory usage feeds straight back into the next measurement.

1. What it does

One machine runs many eval jobs at once, and none of them know about each other. If two big jobs start together, the GPU runs out of memory and both crash. The scheduler is the single place where every job must ask permission first.

An earlier version asked each job to declare how much GPU memory it would use — and trusted the number. Declarations were routinely wrong (the default was 0), so the gate protected nothing. The current version measures instead. Once a second, a sampler reads every process's GPU memory and a walk over /proc reads every process's RAM; each megabyte is then attributed to an owner — a shared model server, a running job's process tree, or "external" (anything the scheduler doesn't manage, like another session's backend or Xorg). Admission then checks, for each resource: does this job's estimated need fit into what is actually free right now? A machine without a GPU simply has no VRAM dimension to check — it gates on RAM alone.

Where do the estimates come from? They are learned, not configured (resource_stats.py · CalibrationStore). There is no profiling step: the first time a graph runs, the scheduler just watches it, and when the job finishes it writes down what it saw — each model server's peak, the job tree's peak, for both VRAM and RAM — into a calibration file (outputs/system/resource_calibration.json). We call this sedimenting: every finished run leaves a layer of measurements behind. From the second run on, admission uses those measured numbers. Each entry is keyed to a hash of the nodeset's source code, so editing the code automatically retires stale statistics.

An admitted run executes in its own OS process (app/eval_subprocess_main.py) and its own process group: cancelling it is one signal to the group, and a crash cannot take the backend down. As a backstop, the subprocess also caps its own torch allocator at the admitted VRAM amount (AGENTCANVAS_VRAM_CAP_MB) — a job that badly exceeds its estimate crashes itself instead of its neighbours. This cap exists only for VRAM; RAM is checked at admission but not enforced afterwards (§6).

1.1 The smallest run

A graph name and an episode count; everything else defaults. The response is immediate — the run is queued, not started:

curl -X POST http://127.0.0.1:8000/api/eval/v2/start \
  -H "Content-Type: application/json" \
  -d '{"graph_name": "straightforward", "episode_count": 1}'
# → {"run_id": "20260704_200243", "status": "queued", "via_subprocess": true}

The response comes back immediately: the run is queued, not started. A second or two later the scheduler admits it — this graph has run before, so the charge is what was measured last time, not what any profile declared. The subprocess runs its episode, writes summary.json and a _DONE marker, and is reaped — leaving one more layer of measurements in the calibration file.

2. Lifecycle of one job

A job moves through four states. Every transition is written to outputs/eval_runs/{run_id}/ as it happens, so the on-disk record is always the truth — restart the backend and the completed history is still there.

  • submit (job_scheduler.py · JobScheduler.submit) — first checks the job is possible at all: declaring more memory than the machine physically has gets an HTTP 400 on the spot, instead of a job that waits forever. Then it writes spec.json and an initial pending summary, and puts the job in the queue. The model singletons the graph needs were already loaded by the API handler before submit (registry.py · ensure_shared_nodesets_for_graph) — which is why, at admit time, an already-loaded model costs the job almost nothing.
  • admitted (JobScheduler._admit, §3) — picked FIFO when the gates clear.
  • running (JobScheduler._spawn) — the scheduler starts python -m app.eval_subprocess_main in its own process group, sends its output to files in the run dir, and passes the admitted VRAM amount in as the allocator cap. It also opens an observation window for the job (resource_stats.py · note_job_started): from now until the job ends, every measurement sample updates the window's peaks — for the job's own process tree and for every model server it uses.
  • reaped (JobScheduler._reap) — when the child exits, its final status follows the small table below. The model servers it was using are released (one that nobody uses any more gets unloaded), and its observation window is sedimented into the calibration file (resource_stats.py · note_job_finished).
child exit_DONE on diskterminal status
any rcpresentwhatever summary.json says (completed / error / cancelled)
any rcabsentaborted (external kill or crash mid-run; rc=-15 = SIGTERM)

3. The measured admission gate

Admission guards against two failures at once: starting a job that doesn't fit (an OOM can kill several runs), and holding back a job that would fit (an idle GPU). Every second, JobScheduler._admit walks the queue in submission order and applies four checks:

  1. canvas lock — someone is using canvas Play; the GPU is theirs, nothing admits.
  2. exclusive — a job marked exclusive_gpu waits for an empty machine, and blocks everything else while it runs.
  3. the fit check — for each resource the machine can measure (VRAM, RAM): estimated need ≤ measured free − standing reservations. A job that doesn't fit is skipped, not blocking — smaller jobs behind it may start first.
  4. fallback — a machine with no sampler at all (unit tests) falls back to the old declared bookkeeping: marginal_vram_mb ≤ usable − Σ declared.

Where does the job's estimated need come from? The scheduler tries the most trustworthy source first and falls down a ladder (JobScheduler._resolve_estimate). The rung it landed on is recorded on the job as admit_basis, so you can always see what kind of number a decision was based on:

admit_basiswhencharge
calibratedthis graph and every model it needs have been measured before, on every resourcethe measured estimate (§4) plus a safety margin (1500 MB VRAM / 1024 MB RAM); already-loaded models count ~0
hintno measurement for some model, but its author shipped an expected size (expected_vram_mb / expected_ram_mb)same formula, with the preset standing in for the missing measurement
declaredno calibration; the profile declared marginal_vram_mbthe declared number — VRAM only; nothing ever declared RAM, so RAM is not checked on this rung

One race needs closing. A job that was just admitted hasn't allocated its memory yet — models take a minute to load. If admission only looked at measured free, the next tick would see the GPU still empty and admit too much. So every running job holds a reservation: the amount it was admitted for, minus what it has actually been measured using so far (JobScheduler._pending_reservations). 4. Measurement, calibration, estimation

Everything measurement-related lives in one module, services/resource_stats.py. It has three jobs: work out whose memory each megabyte is (attribution, §4.1), remember what each graph and each model actually used (calibration, §4.2), and turn those records into the number the gate needs (estimation, §4.3).

4.1 Attribution — whose memory is this?

Two lists arrive every second: GPU memory per process, and RAM per process. The GPU list comes from nvidia-smi pmon (resource_sampler.py · _gpu_processes) — this variant matters because it also shows graphics contexts: habitat and MatterSim render through EGL, which the ordinary compute-only query cannot see at all. The RAM list is one walk over /proc (resource_stats.py · _proc_table). Each process is then matched to an owner by walking up its parent chain (resource_stats.py · _attribute_pid):

  1. shared model server — the process (or one of its ancestors) is a server the registry started;
  2. running job — the chain reaches a job's subprocess. Parent links are used rather than process groups, because a job's workers move into groups of their own but keep their parent link — so the whole worker tree is still found;
  3. external — everything else: other sessions' backends, Xorg, the OS itself. Nobody is charged for it, but it occupies memory, and the gate sees that automatically through "measured free".

4.2 Sedimentation — writing the calibration store

While a job runs, its observation window keeps the peak of everything it owns — its own process tree, and each model server it uses — separately for VRAM and RAM. When the job is reaped, the window is written into outputs/system/resource_calibration.json, as a running average plus an observed maximum. Two guards keep the data honest. First, a window that never received a single sample writes nothing — otherwise a sampler outage would record fake zeros. Second, a server that was running but used 0 MB is recorded as a real zero: "measured zero" and "never measured" are different facts. Every entry is keyed to a hash of the nodeset's source code (content_hash.py · hash_nodeset_tree), so editing the code silently retires its old numbers. Calibration files from before the RAM dimension existed are migrated automatically (resource_stats.py · _migrate_v1); their RAM half fills in on the next run.

4.3 Estimation — GET /api/eval/v2/estimate

The estimator (resource_stats.py · estimate_tree_mb / estimate_shared_mb) prices a run in three parts: the one-time cost of models the graph needs but that aren't loaded yet, the cost of the job's own process tree at the requested worker count, and a fixed safety margin on top. Every number states where it came from: measured (this exact worker count was seen before), fitted (a line through two or more observed worker counts), or scaled (one observation, scaled proportionally). For models there is one more fallback: the author preset. A nodeset can ship expected_vram_mb / expected_ram_mb (bases.py · BaseNodeSet — say, a SAM nodeset declaring ≈600 MB), and that number is used whenever no valid measurement exists — on a fresh install, or right after a code edit retired the old statistics. Presets deliberately survive code edits: a model's footprint is dominated by its weights, so the number stays true across refactors — and a real measurement replaces it as soon as one exists.

One honesty rule governs the total: if any component of the run has no number at all, the total is null — a missing model could be 10 GB, and a partial sum pretending to be a total is worse than no answer. All of this can be asked without running anything: GET /api/eval/v2/estimate (eval.py · estimate_eval_v2) returns the per-resource breakdown and max_workers, the largest worker count that currently fits.

Two older fields, usable_vram_mb / reserved_vram_mb (not shown above), still exist for the no-sampler fallback. The fields a decision actually uses are measured_free_mb, pending_reservation_mb, and each job's admit_charges + admit_basis. To cancel a run: POST /runs/{run_id}/cancel — a queued job is cancelled on the spot; a running one gets a SIGTERM to its whole process group.

6. Where it deviates from the model

Queued jobs do not survive a backend restart. §2 says "the on-disk record is the truth", but on startup reconcile_aborted_runs (job_scheduler.py) marks both running and pending runs as aborted — the spec is still on disk, but nothing re-enqueues it. Queue durability is an acknowledged gap.
Local in-process nodesets are calibrated at 0 by fiat, not by measurement. A parallelism="shared" nodeset loaded in-process (no auto_host server, e.g. env_adapter) has no PID of its own; per-PID attribution structurally cannot see it. It is recorded as 0 MB on every resource so it never blocks estimate coverage (resource_stats.py · _is_loaded_local) — a local nodeset that secretly loads a model into the backend process would be under-estimated.
The quota cap is best-effort, not a hard sandbox. AGENTCANVAS_VRAM_CAP_MB caps the torch allocator of the eval subprocess itself (eval_subprocess_main.py · _apply_vram_cap). Its child env servers, non-torch CUDA allocations, and EGL contexts are uncapped. The measured gate is the primary defense; the cap is a backstop.
RAM is gate-only — there are no RAM quota teeth. There is no RAM sibling of AGENTCANVAS_VRAM_CAP_MB: RLIMIT_AS counts virtual address space and would kill mmap-heavy simulators, and cgroup memory limits need privileges the backend doesn't assume. The defenses are the gate itself and the permanent floor held out of the budget (resource_stats.py · RAM_FLOOR_MB, 4 GB below MemAvailable).
Job-tree RAM over-counts shared pages. Per-process RSS counts copy-on-write pages once per process, so a forked worker tree's summed RSS exceeds its true footprint. Errs conservative for admission; PSS would be exact but costs a smaps_rollup read per process per tick.
A declared job's RAM rides ungated. The declared rung predates the RAM dimension — profiles only ever declared marginal_vram_mb — so when calibration is incomplete and a declaration exists, admission gates VRAM on the declaration and RAM not at all, until the first run calibrates both.
Attribution lags reality by up to ~5 seconds. Per-PID data refreshes on the sampler's slow cadence (resource_sampler.py: gpu_procs cache ~5s; totals 1s). Reservation decay therefore lags, which errs conservative — and a sub-5-second job can be reaped with an empty window (nothing sediments; guarded by samples_seen).
FIFO admission can starve a large job. A job that doesn't fit is skipped, not blocking (_admit walks on), so a continuous stream of small jobs can overtake a large one indefinitely. The priority field is accepted and persisted but ignored (M1 FIFO).
The VRAM ledger is a scalar. Totals and free are summed across all visible GPUs (detect_total_vram_mb, sampler totals); there is no per-GPU accounting or placement. Correct on this single-GPU box, wrong in both directions on a multi-GPU one.
Unseen worker counts are extrapolated, not measured. With one calibration point the tree cost scales proportionally (estimate_tree_mb, basis scaled); a fixed per-run overhead makes this over-estimate at high worker counts and the safety margin absorbs the reverse error. Real multi-worker runs upgrade the basis to fitted automatically.
