10Graph SDK

Build and run the same graphs as a Python library — no canvas, no JSON authoring.

Every graph on the platform is a GraphDefinition — a pure dataclass the canvas emits, the JSON stores, and the GraphExecutor runs (Graph-Expressible Agents). The Graph SDK adds a third way to produce and drive that same object: write it in Python, imported like LangGraph / LangChain rather than authored as JSON or wired on the canvas. Same nodes, same executor, same eval — just a builder in front and a runner behind. Because it is the same GraphDefinition, code ⇄ JSON ⇄ canvas stay fully reversible.

Design docs: Graph Executor · builds on Graph-Expressible Agents, Nested Graph System, Hook System, Batch Evaluation

1. Three ways to the same graph

Author with Surface Produces

JSON Hand-written workspace/graphs/**.json GraphDefinition JSON

The Graph SDK surface is a thin, ergonomic wrapper over the existing dataclass model (app/graph_def.py) and the in-process engine (app/agent_loop/graph_executor.py). It adds no new execution machinery — every method assembles the same NodeDef / EdgeDef / ContainerDef the canvas emits.

2. Build a graph in code

Three moves — add a node, connect two ports, mark the graph's boundary — then run. Node ids auto-generate; extra kwargs become the node's config. A graphOut's portName becomes the run's output key.

from agentcanvas import Graph

g = Graph(name="demo")
seven = g.add("const", value=7)
five  = g.add("const", value=5)
add   = g.add("add")
out   = g.graph_out("total")

g.connect(seven.out("value"), add.in_("a"))
g.connect(five.out("value"),  add.in_("b"))
g.connect(add.out("sum"),     out.in_("value"))

result = g.run()            # in-process — no FastAPI, no GPU
print(result["total"])     # -> 36

Shared state is first-class too: g.container(id, states=…) declares a blackboard and g.grant(node, container_id) gives a node read/write access — the same containers + access grants the canvas draws.

3. Run it in-process

g.run() drives GraphExecutor.run() directly — the same engine as the canvas Play button — with no FastAPI, no scheduler, and no GPU-admission gate (that lives in JobScheduler, not the executor). Pure-Python / builtin graphs run untouched. Nodeset graphs (mapgpt__*, env_mp3d__*, …) run too: load_nodesets defaults to "auto", so run() scans the workspace registry and auto-loads exactly the nodesets the graph needs — spawning server-mode subprocesses for env nodesets, precisely as the backend does — then tears down what it started.

from app.mapgpt_mp3d_sdk import build

g = build()                                    # the real MapGPT-MP3D graph, in code

# One episode, in-process. load_nodesets="auto" (the default) scans the
# workspace registry and auto-loads every nodeset the graph needs — spawning
# the env_mp3d simulator subprocess — then tears it down afterwards.
r = g.run()
print(r.outputs["metrics"])                    # {'success': 0.0, 'spl': 0.0, 'nav_error': 5.16, ...}

# Batch eval — drives the SAME BatchEvalRunner the backend eval API uses:
# one executor run per episode, metrics averaged over completed episodes.
ev = g.eval(episodes=10, dataset="R2R", split="val_unseen")
print(ev.metrics)                              # aggregate: {'success': ..., 'spl': ...}
print(ev.episodes[0]["metrics"])               # per-episode breakdown

The env node never runs inside the calling process. ensure_nodesets_for_graph() spawns an auto_host subprocess in the nodeset's own conda env (e.g. ac-mp3d), registers HTTP-proxy node classes into the global NODE_HANDLERS, and the executor reaches the simulator through those proxies:

  g.run(load_nodesets="auto")
      │  scan_all()  +  ensure_nodesets_for_graph()   → register node types into NODE_HANDLERS
      ▼
  GraphExecutor.run(graph, DefaultSession)             ← same engine as the canvas Play button
      │  mapgpt__*  nodes  → run in-process (local mode)
      │  env_mp3d__* nodes ──HTTP──▶  auto_host subprocess  (ac-mp3d env, MatterSim + GPU)
      ▼
  RunResult.outputs["metrics"]                          → teardown: env server unloaded

A real env run is an experiment. Spawning the simulator uses the GPU and the LLM, so a multi-episode sweep belongs behind /experiment:run (GPU admission, port regulation), not a bare call. The code path is identical either way — only admission control differs.

4. Batch evaluation

g.eval(episodes=N, dataset=…, split=…) is the Graph SDK entry to Batch Evaluation. It constructs the very same BatchEvalRunner the backend eval API uses (app/agent_loop/eval_batch.py) — one GraphExecutor run per episode, episode placement pushed through the env panel, per-episode metrics harvested from the graph's metrics graphOut, and the aggregate computed as the mean over completed episodes. The returned EvalResult carries .metrics (aggregate), .episodes (per-episode), and .by_task. "Correct" here is structural: g.eval does not re-implement evaluation, it reuses the audited runner.

5. Authoring sugar: loop / hook / composite

The fiddly parts of a graph get intent-named helpers. g.loop(init=…, carry=…) declares a two-sided iterIn/iterOut episode loop (Graph-Expressible Agents §3) with the correct persist flags set automatically — init-only ports persist across steps, carried ports do not persist on the init side (the iterOut carry supersedes) — so the loop can't accidentally starve on iteration 1. Wire it through the returned handle:

loop = g.loop(
    init=[("instruction", "TEXT")],            # seed-only; persists across steps
    carry=[("viewpoint_id", "TEXT")],          # written back each step by iterOut
)

loop.seed("instruction", reset.out("instruction"))     # run-start value
loop.feed("viewpoint_id", observe.in_("viewpoint_id")) # wires BOTH init + carry sides
loop.carry("viewpoint_id", step.out("viewpoint_id"))   # per-step write-back
loop.stop(budget.out("done"))                          # termination signal

Two more: g.hook(event, command, …) attaches a lifecycle shell hook (Hook System) — GraphStart, PreNodeExecute, PostNodeExecute, GraphComplete, GraphError. g.composite(id, subgraph) wraps a nested graph as a composite node (Nested Graph System); flatten_graph() expands it before execution, so composites run in-process like everything else.

6. Reverse — graph → code

The inverse direction (roadmap F4): compile any GraphDefinition — canvas-authored, loaded from JSON, or code-built — back into a self-contained builder script via graph_to_code(gd) (or Graph.to_code()). The emitted script rebuilds the same topology through this very API; the round-trip is semantically exact (node types + configs, wire multiset, containers, grants, hooks), verified against the reference MapGPT-MP3D graph.

import json
from agentcanvas import Graph, graph_to_code
from app.graph_def import GraphDefinition

gd = GraphDefinition.from_dict(json.load(open("mapgpt_mp3d.json")))
print(graph_to_code(gd))            # a standalone, runnable builder script
# ...or on a live Graph:  Graph.from_definition(gd).to_code()

7. Packaging & install

The SDK ships as an installable package: pyproject.toml at agentcanvas/backend/ plus a top-level agentcanvas namespace that re-exports the surface, so pip install -e . makes from agentcanvas import Graph work with no PYTHONPATH. The SDK core is stdlib-only; the heavy runtime dependencies a real nodeset run needs (FastAPI, torch, simulators) stay in requirements.txt / the conda env, kept out of pyproject.toml so the import stays light. Runnable examples live under agentcanvas/backend/examples/.
