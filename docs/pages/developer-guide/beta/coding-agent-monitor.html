Coding-Agent Monitor

BETA

Run a stock coding agent on a VLN episode through MCP — and watch every thought, action, and frame live from the AgentCanvas UI.

This experimental subsystem measures what a vanilla coding agent — a Claude Agent SDK session with zero embodied specialization — scores on R2R-CE when the only world it can touch is the env_habitat nodeset behind an MCP bridge. Scores come from habitat's own measures via env_habitat__evaluate: the same ruler as every verified graph on this site. The Coding Agent tab in the frontend starts a run with one click and streams the trajectory as a unified live log, with the agent's egocentric frames embedded inline at the exact observe() calls that produced them.

1. What it is

Five pieces, two of them outside the backend proper:

PieceWhereRole
MCP bridgebeta-coding-agent/mcp_bridge.pyStdio MCP server exposing the agent-facing toolset (observe, step); forwards to an env_habitat auto_host over POST /call/{fn}. One bridge process per agent session per episode. Optionally dumps every observed frame to a live directory.
Episode driverbeta-coding-agent/run_episodes.pyPlaces episodes via the env panel, runs one clean Claude Agent SDK session per episode, evaluates driver-side, writes per-episode trajectory JSONL (flushed per event) and an incrementally-updated summary.json.
Runner servicebackend/app/services/coding_agent_runner.pySingleton owned by the FastAPI lifespan. One-click host: spawns the auto_host (via BaseServer, on a free port outside the reserved pools) plus the driver as a process-group child, and derives run status purely from the driver's artifacts.
REST routerbackend/app/api/execution/coding_agent.py/api/coding-agent — POST /start, POST /stop, GET /status, plus per-episode textlog (offset-based JSONL slices) and frames / frame/{name}.
Monitor pagefrontend/src/pages/coding/CodingAgentPage.tsxTop-level Coding Agent tab: run control panel, per-episode selector with success badges, and the unified live log.

The driver and bridge are deliberately independent of the backend — the CLI recipe in beta-coding-agent/README.md runs the same experiment with no AgentCanvas backend at all. The monitor tab is a convenience shell around them, not a dependency.

2. The agent's contract

The vanilla claim rests on what the agent is not given. Its entire world:

ToolReturns
observe()One egocentric RGB frame (PNG, YAML-default resolution 224×224). Nothing else — no pose, no depth, no map, no panorama.
step(actions)Executes a sequence of discrete actions — 0=STOP, 1=forward 0.25 m, 2=left 15°, 3=right 15° (max 50 per call) — and returns only bookkeeping: steps executed, budget remaining, whether the episode ended.
  • The navigation instruction arrives in the system prompt; episode placement and evaluate stay driver-side, so the agent can never read its own metrics.
  • The agent must issue STOP itself (R2R-CE success requires stopping within 3 m of the goal). The driver never force-stops — a non-stopping agent honestly scores its distance-to-goal at budget exhaustion.
  • No built-in tools: the SDK session runs with an empty tool list plus the two MCP tools above. Filesystem-as-memory and other coding-agent affordances are a later ablation.

3. Run & watch

Open the Coding Agent tab, pick a split (rand100 — the paper-canonical 100-episode subset of R2R-CE val_unseen — or full val_unseen), an episode range, a max-turns cap, optionally a model, and press Run. The backend brings up the env, drives the episodes, and the page follows the active episode automatically. v1 runs a single worker: one episode at a time, one run at a time.

Everything lands under outputs/beta-coding-agent/{run}/: episode_{i}.jsonl (the trajectory, one JSON event per line, flushed as it happens), live_{i}/obs_NNNN_stepNNN.png (every frame the agent observed), and summary.json (per-episode metrics + token usage + aggregate). The trajectory is the ground truth the UI renders:

{"t": 0.0,  "kind": "episode_meta", "index": 0, "instruction": "Walk between the bar and chairs. Stop at the corner of the bar."}
{"t": 12.4, "kind": "thinking",     "chars": 141, "text": "The bar counter is ahead-left; I should pass between it and the chairs."}
{"t": 13.0, "kind": "tool_use",     "name": "mcp__env__step", "input": {"actions": [2, 2, 1, 1, 1]}}
{"t": 14.9, "kind": "tool_result",  "texts": ["{\"executed\": 5, \"steps_taken_total\": 23, \"episode_over\": false}"]}

4. How the live log works

  • 1 Hz polling, offset-based. The page asks textlog?offset=N for lines it hasn't seen; the router returns whole-line JSONL slices plus the next offset. No websocket — the trajectory file, flushed per event, is the transport.
  • Frames embed inline. The k-th observe call in the trajectory pairs with the k-th dumped frame, rendered at native 224 px inside the log right where the call happened, with a click-to-zoom lightbox. At the live edge a same-size “frame pending” placeholder holds the slot until the file lands (usually the next poll).
  • Thinking is summarized, not raw. The driver requests summarized thinking display, so the log shows readable reasoning summaries; the raw chain of thought is never returned by the API.
  • Badges. Each finished episode gets ✅/❌ from its own evaluate call; the header aggregates SR / SPL / stop-rate as episodes land.

5. Early numbers

First calibration (10 episodes of rand100, max-turns 80, claude-fable-5):

SRSPLOracle SRnDTWStop rate
0.300.300.500.530.50

Failure taxonomy: 3 clean successes (each near-optimal, SPL ≈ 1.0), 2 near-miss stops (3.2–3.5 m from goal), 5 max-turns truncations — the cap binds half the failures, so the headline is a floor, not a ceiling. Reference points on the same ruler: AO-Planner port SR 0.35 on rand100; CMA 0.38 on val_unseen.

Cost structure (why episodes are expensive): every LLM call re-reads the whole conversation prefix (~16k tokens of harness + instruction before anything happens), and the vanilla agent makes an LLM call per micro-decision — cost is almost exactly linear in turn count (≈ $0.036/turn API-equivalent on fable-5; cache re-reads are 57% of the bill). Images are a rounding error at 224 px (~70 tokens each). The planned macro-action arm (step_hightolow) cuts calls ~3× and is also the scientifically interesting ablation.

6. Beta status & known limits

  • Single worker. The CLI driver runs N-way parallel across multiple auto_hosts; the monitor tab intentionally hosts one worker in v1.
  • Driver outlives a backend restart. The driver is spawned without a parent-death signal; restarting the backend orphans (does not kill) a live run. Stop the run first.
  • Frame pairing is order-based. If an observe call fails before the bridge dumps its frame, subsequent inline images shift by one. Not yet observed in practice.
  • Numbers are calibration-grade. The 10-episode results above predate a session-hygiene fix (project instructions were leaking into the agent's context); the paper-grade run will be a clean rerun at rand100 scale.
  • Roadmap: macro-action ablation arm, run-history browser with A/B comparison, trajectory-on-map view, cost ticker.

Everything on this page describes the implementation on branch dev/coding-agent; interfaces may change without an ADR while the feature is in beta.
