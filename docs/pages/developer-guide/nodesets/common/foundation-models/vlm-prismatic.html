VLM Prismatic NodeSet

The VLM Prismatic NodeSet (VLMPrismaticNodeSet) wraps the Prismatic VLM (TRI-ML, 2024) as a generic foundation-model nodeset. It exposes domain-agnostic primitives that any method nodeset can wire in via canvas wires:

Primitive Purpose
vlm_prismatic__score_tokens Token-likelihood scoring: (image, prompt, candidate_tokens) → softmax probs. Building block for multi-choice answering, calibrated frontier scoring, confidence estimation.
vlm_prismatic__generate Free-form generation: (image, prompt) → text.

Method nodesets stay decoupled from Prismatic — they only know about generic primitives, so swapping in another open-source VLM that exposes token-likelihoods (LLaVA, Qwen-VL, InternVL, ...) is a one-class change inside this nodeset rather than a method-side rewrite.

1. Why a foundation-model nodeset

Method nodesets (e.g. explore_eqa) used to bundle Prismatic loading, prompt formatting, and VLM-specific math inline. Three problems with that:

  • Coupling: replacing the VLM forced a method-side rewrite.
  • Duplication: two methods that need token-likelihood scoring would each load Prismatic separately.
  • Boundary leakage: model.get_loss(...) is a Prismatic-API detail; it leaked into method code that shouldn't care.

The fix is the same pattern as policy_cma / policy_vla / vlm_llava (future): give the foundation model its own nodeset with a generic interface (score_tokens, generate, embed, ...). Method nodesets reach it via canvas wires, not Python imports. The boundary is enforced by the auto-host single-class-per-subprocess constraint — two nodesets can't share Python state, only wire-level data.

Codified in roadmap TODO #56 ("Clarify method vs foundation-model boundary in nodeset design").

2. Canvas Nodes

vlm_prismatic__score_tokens

Field Detail
Inputs image (IMAGE), prompt (TEXT), tokens (ANY — list of candidate strings)
Outputs probs (ANY — softmax distribution aligned with tokens)
Config temperature (float, default 1.0)
Backend call model.get_loss(image, prompt, return_string_probabilities=tokens) then softmax with temperature

Empty tokens list → empty probs. Missing image / VLM unavailable → uniform distribution (degraded mode — see §5).

vlm_prismatic__generate

Field Detail
Inputs image (IMAGE), prompt (TEXT)
Outputs text (TEXT)
Config max_new_tokens (int, default 128), temperature (float, default 0.2)
Backend call model.generate(image, prompt, do_sample, temperature, max_new_tokens)

3. Server Mode

Auto-routed to server mode by the registry because server_python points at a non-default Python:

class VLMPrismaticNodeSet(BaseNodeSet):
    server_python = os.environ.get(
        "HMEQA_PYTHON", "python"
    )

The hmeqa env (Python 3.9 + torch 2.2.1 + Prismatic + CUDA bfloat16) is reused — no second conda env required for the VLM.

Singleton bundle

Prismatic weights load once per subprocess via _ensure_loaded(), mirroring policy_cma / explore_eqa's old loader. The initialize() hook warms the weights eagerly (a few minutes for prism-dinosiglip+7b on first call); subsequent score_tokens / generate calls hit the cached bundle.

_VLM_BUNDLE: dict | None = None
_VLM_LOAD_LOCK = threading.Lock()

def _ensure_loaded(model_id=None):
    global _VLM_BUNDLE
    if _VLM_BUNDLE is not None: return _VLM_BUNDLE
    with _VLM_LOAD_LOCK:
        if _VLM_BUNDLE is not None: return _VLM_BUNDLE
        from prismatic import load
        model = load(mid, hf_token=hf_token)
        model.to(device, dtype=torch.bfloat16)
        _VLM_BUNDLE = {"model": model, "device": device, "model_id": mid}
    return _VLM_BUNDLE

Parallelism

Default parallelism="shared" (or singleton — one subprocess per Prismatic instance). All worker_count > 1 env subprocesses fan into the same VLM via HTTP. Future: bump to "replicated" if the platform grows multi-GPU semantics so each worker can pin its own Prismatic instance to a different GPU.

HF_TOKEN (required for gated weights) HuggingFace read token — Prismatic weights are gated

5. Degraded Mode

When Prismatic isn't importable (e.g. component-registry scan from the agentcanvas env, GPU unavailable, weights download failed), every node falls back to a deterministic stub:

  • score_tokens → uniform distribution [1/n, 1/n, ...] over the candidate tokens
  • generate → empty string

The graph keeps running; downstream methods see "I have no information" semantics and behave as if VLM signal is uninformative. Useful for structural smoke tests without a GPU.

6. Why Not a Commercial VLM API

The score_tokens primitive needs token-level logits for candidate strings. Commercial multimodal APIs as of 2026-05 don't expose this:

  • GPT-4o, Claude Sonnet/Opus, Gemini multimodal — text-out only, no logprobs for image inputs
  • OpenAI text-completion APIs expose top_logprobs but the multimodal path doesn't

Consequences:

  • Methods needing calibrated confidence (explore-eqa style) require an open-source VLM — Prismatic, LLaVA, etc.
  • A "commercial-VLM" path could only support generate (free-form text), not score_tokens. Those methods would have to redefine "confidence" as parsed-text heuristics ("rate 0-100..."), which is unreliable.

This is why vlm_prismatic is a server-mode local nodeset rather than an HTTP-API client.
