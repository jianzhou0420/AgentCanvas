VLA Policy NodeSet

The VLA Policy NodeSet (policy_vla) is a generic VLA inference framework for running any vision-language-action policy on any robot environment. The framework is implemented via the adapter system: each forward pass is decomposed into four atomic translation stages (env → canonical → model → action canonical → env action), and the four canvas nodes are 1:1 with those stages.

Single nodeset, single pipeline, every VLA. Every VLA shipped or planned for AgentCanvas — Pi0, SmolVLA, Diffusion Policy, DROID-DP, RT-1-X, and the next ones (Octo, OpenVLA, RT-2, π₀-FAST, …) — lives inside this one nodeset and runs through this one 5-node graph. There are no special-case nodes, no bypass paths, no per-policy nodesets. A new VLA is always an adapter + policy + adapter triple dropped into the existing folders; selecting it is always picking three filenames in three dropdowns. That uniformity is the load-bearing rule of policy_vla — see §3 for why, §9.1 for the recipe, §9.1.1 for RT-1-X as the worked example.

Robots follow the same rule: LiberoRobot and SimplerRobot are the two shipped instances; new robots (Franka, Droid, …) are first-class extensions, not exceptions.

The adapter system itself is vendored verbatim from the upstream vlaworkspace training repo (vlaworkspace/src/vlaworkspace/adaptors/) — per AgentCanvas's no-upstream-import policy, source is copied with rewritten imports. This nodeset is therefore both runtime-coupled with vlaworkspace (same CanonicalDict, same Adaptor classes) and deployment-decoupled (no live import — bug-for-bug parity with the snapshot taken at vendoring time).

⚠️ This page is a code-derived snapshot, not the design spec. It captures what workspace/nodesets/policy/policy_vla/ currently does. Design principles for "how a new VLA gets onboarded" are pending the maintainer's annotations — sections marked TBD by maintainer are placeholders.

1. What it does

policy_vla is the canvas-side runtime for any VLA policy. Inference is decomposed into four atomic translation stages, each exposed as one canvas node, hosted in a single subprocess on the ac-vla-policy conda env (server mode per ADR-server-001):

env_obs ──[1]── canonical ──[2]── model_batch ──── predict ──── model_output ──[3]── canonical_action ──[4]── env_action
        env→canonical      canonical→model     (BasePolicy.        model→canonical                     canonical→env
        (RobotAdaptor)     (ModelAdaptor)       predict_action)    (ModelAdaptor)                      (RobotAdaptor)
            ↓                  ↓                      ↓                  ↓                                  ↓
   adapt_env_to_canonical  adapt_canonical_        predict        adapt_model_to_canonical          adapt_canonical_to_env
                           to_model

5 canvas nodes, 1:1 with the 4 atomic adapter stages + 1 model forward. Each adapter stage is one canvas node — perfect symmetry between obs side (stages 1↔2) and action side (stages 3↔4). The canonical_action CanonicalDict is a real wire on the canvas, not an internal value, so it can be inspected, intercepted, or fed into future "action filter" nodes.

Why decompose at all:

  • Each config-owning node owns one slice of config (robot kwargs / model kwargs / policy kwargs); the two action-side nodes carry no config. Per feedback_env_panel_scope: env panels manage env knobs (suite/episode), not graph artefacts.
  • Each stage corresponds to one abstract method on RobotAdaptor or ModelAdaptor, so swapping any one stage in the canvas is symmetric to swapping its underlying class. New robot ⇒ new RobotAdaptor (stages 1+4); new model ⇒ new (ModelAdaptor + BasePolicy) pair (stages 2+3 + GPU).
  • Canvas-level abstraction mirrors the framework abstraction (vlaworkspace's 4 atomic methods). No fused boundaries — the canvas does not hide an intermediate that exists at the Python level.
  • Heavy GPU load happens in predict; the four adapter nodes are pure-CPU dict shape transforms, fast to re-run when only config changes.

1.1 Where it fits

Upstream (obs) RobotAdaptor Downstream (action)
env_libero__reset / env_libero__step LiberoRobot (delta or absolute variant) env_libero__step
env_simpler__reset / env_simpler__step SimplerRobot (WidowX + Google Robot) env_simpler__step

The adapter nodes are pure functions of their inputs + config; they don't talk to the env directly. Coupling with the env is via wire shape (obs ports), not API.

2. Architecture: the Adaptor system

The framework is a 4-stage translation pipeline mediated by the CanonicalDict lingua franca. Each stage is one abstract method, packed into one of two adapter classes:

Stage Direction Owner class Method
1 env_obs → canonical RobotAdaptor env_to_canonical(env_obs)
2 canonical → model_batch ModelAdaptor canonical_to_model(canonical)
3 model_output → action_canonical ModelAdaptor model_to_canonical(model_output, info)
4 action_canonical → env_action RobotAdaptor canonical_to_env(canonical_action, state) 
RobotAdaptor   ×   ModelAdaptor   →   Adaptor (composed)   +   BasePolicy
   (stages 1+4)     (stages 2+3)      (bridge per pair)         (per model — stage 3's nn.Module)

Any robot × any model is a valid pair. RobotAdaptor knows nothing about which model it'll be fed into; ModelAdaptor knows nothing about which robot produced its inputs. They communicate only through CanonicalDict + CanonicalInfo. The composed Adaptor exposes both single-stage methods (output_transforms = stage 3 then stage 4) and pre-fused convenience methods (env_input_transforms = stages 1+2) — but the canvas calls each atomic stage individually via dedicated manager methods (model_to_canonical_action, canonical_action_to_env), keeping each canvas node 1:1 with one abstract method.

Source: all four method signatures, CanonicalDict, CanonicalInfo, and the composed Adaptor are vendored verbatim from vlaworkspace/src/vlaworkspace/adaptors/. AgentCanvas does not import vlaworkspace at runtime — it owns a snapshot.

2.1 CanonicalDict — the lingua franca

Every robot maps to this format; every model maps from it. Defined in adapters/canonical.py:

CanonicalDict = {
    "data": {
        "images": {"front": ndarray | None, "wrist": ndarray | None},
        "state": {"pos": …, "rot": …, "gripper": …, "joint_position": …},  # all optional
        "actions": {"pos": …, "rot": …, "gripper": …, "joint_position": …},  # output side
        "prompt": str,
    },
    "info": CanonicalInfo(state_type, state_rot_repr, action_type, action_rot_repr,
                         state_dims, action_dims),
}

The info block is metadata that lets a ModelAdaptor know what kind of rotation (axis-angle vs quaternion vs euler), what keys are populated, etc., so it can normalise correctly without having to know which robot it's paired with.

2.2 RobotAdaptor (abstract)

adapters/base_robot.py. Translates between an env's native obs/action format and CanonicalDict. Abstract methods:

get_canonical_info()           → CanonicalInfo
dataset_to_canonical(data)     → CanonicalDict   # for training
env_to_canonical(env_obs)      → CanonicalDict   # for inference (no actions)
canonical_to_env(canonical_action, state=None) → dict   # action chunk for env
get_state_dim() / get_action_dim() / get_norm_stats_keys() / env_obs() / env_action() / datasets()

Currently shipped subclass: LiberoRobot only.

2.3 ModelAdaptor (abstract)

adapters/base_model.py. Translates between CanonicalDict and model-specific input/output dicts, and owns normalisation (norm_stats, loaded from a JSON file at construction). Abstract methods:

canonical_to_model(canonical) → dict     # produces the batch fed into BasePolicy
model_to_canonical(model_output, info) → CanonicalDict
canonical_to_norm_stats_format(canonical) → dict   # for fitting/loading norm stats
get_norm_stats_mode() → str
get_norm_stats_keys() → tuple[str, ...]
model_input() / model_output() → dict   # shape descriptors

The base class handles norm_stats_path JSON loading uniformly (mean/std/q01/q99/min/max per key), so subclasses just declare which keys they normalise and how.

Currently shipped subclasses: Pi0Model, SmolVLAModel, DPModel (used by both dp and droid_dp).

2.4 Composed Adaptor

adapters/adaptor.py. Constructed inside VlaPolicyManager once a robot + model_adaptor pair is built. Provides:

Adaptor(robot=…, model=…)
    .env_input_transforms(env_obs)   # robot.env_to_canonical → model.canonical_to_model
    .output_transforms(model_output) # model.model_to_canonical → robot.canonical_to_env
    .get_state_dim() / .get_action_dim() / .get_norm_stats() / …

The 5 canvas nodes do not go through the composed Adaptor at all — each calls one atomic method on the manager (env_to_canonical, canonical_to_model, predict, model_to_canonical_action, canonical_action_to_env), keeping every canvas node 1:1 with one abstract method on RobotAdaptor / ModelAdaptor / BasePolicy. The composed Adaptor lives on the manager but is no longer the canvas's runtime path; it stays around for the model_to_env backwards-compat alias used by direct manager callers.

2.5 BasePolicy (abstract, separate from ModelAdaptor)

policies/base_policy.py. The actual GPU model. Distinct from ModelAdaptor: a ModelAdaptor is pure-CPU shape-shifting + normalisation, a BasePolicy is the trained nn.Module. They're paired but not merged because:

  • Two policies can share one ModelAdaptor (DiffusionUnetHybridImagePolicy and DroidDiffusionPolicy both use DPModel — confirmed via hydra configs diffusion_policy_libero.yaml + droid_dp_mlp_libero.yaml).
  • A policy class owns weights + predict_action() + load_checkpoint() + dtype/device — no canonical-format knowledge.

Currently shipped subclasses: Pi0Policy, SmolVLAPolicy, DiffusionUnetHybridImagePolicy, DroidDiffusionPolicy.

3. Design principle: env-side vs policy-side adapter coupling

The four adapter stages split into two halves with very different ownership rules. This split is the load-bearing principle of policy_vla — it dictates what the user can change in the canvas and what is locked.

Half Stages Owner class Bound to User-changeable?
Env-side 1 (env→canonical) + 4 (canonical→env) RobotAdaptor The graph's environment (simulator / benchmark) No — hardwired to the graph's env nodes
Policy-side 2 (canonical→model) + 3 (model→canonical) ModelAdaptor The chosen policy (Pi0, SmolVLA, OpenVLA, …) Yes — selecting a policy picks both adapters as a triple

3.1 Env-side: locked to the graph's environment

A graph's env nodes (env_libero__*, or env_simpler__*, or env_franka__*, …) determine which RobotAdaptor is in play. The two env-side adapter nodes (adapt_env_to_canonical + adapt_canonical_to_env) consume that robot — they do not choose it. It is illegal to mix env-side adapters from different robots in the same graph: a LIBERO graph cannot use a SIMPLER robot adapter, and vice-versa, because the CanonicalDict[obs] produced by the env-side stage 1 must match what the env-side stage 4 expects to invert.

UI contract. Today the env-side adapter nodes expose a robot select dropdown populated by file discovery in adapters/robots/ — see §4.1. Per this principle the robot identity itself should be locked to whatever the graph's env nodes imply, and the dropdown is a temporary state on the way there: any robot file that's incompatible with the current env (e.g. picking a franka_robot on a LIBERO graph) is structurally invalid even though the dropdown will let you do it. The end-state UI is to either (a) hide robot files that don't match the graph's env, or (b) prompt a confirmation modal when the user swaps the env nodeset, explaining that env nodes and env-side adapters are joined at the hip — different CanonicalInfo, different state schema, different action contract. Per-robot kwargs (e.g. use_delta_actions, delta_action_mask) are no longer free-form fields on the canvas — they live as DEFAULT_KWARGS module constants in each robot file (e.g. adapters/robots/libero_robot.py); selecting a different file = selecting a different defaults preset.

3.2 Policy-side: policy + 2 adapters as a linked triple

A single VLA is fully specified by three artefacts that move together as one unit:

  1. The BasePolicy subclass (e.g. Pi0Policy) — owns weights, predict_action, checkpoint loading.
  2. The ModelAdaptor.canonical_to_model direction (e.g. Pi0Model.canonical_to_model) — packs canonical obs into the model's input batch (stage 2).
  3. The ModelAdaptor.model_to_canonical direction (same Pi0Model) — unpacks raw model output into CanonicalDict[action] (stage 3).

One linked dropdown drives all three. On the canvas, the model select on adapt_canonical_to_model is the single source of truth. Changing it from pi0 → openvla simultaneously:

  • swaps the policy class loaded inside predict,
  • swaps the canonical_to_model packing on this node,
  • swaps the model_to_canonical unpacking on adapt_model_to_canonical.

The user does not pick the two policy-side adapters separately — they are a triple, not three independent choices. The downstream predict and adapt_model_to_canonical nodes don't carry their own model selector — adapt_model_to_canonical is a stateless reader that calls into the same _model_adaptor slot the manager singleton owns, so it auto-follows whatever was last selected on adapt_canonical_to_model. Today the canvas has two coupled selects: model on adapt_canonical_to_model (picks the ModelAdaptor file from adapters/models/) and policy on predict (picks the BasePolicy file from policies/); per the principle these will eventually collapse to one selector that drives both, but pairing is still the user's responsibility (the _resolve_classes() legacy table documents the canonical pairings for now). Per-policy kwargs are no longer free-form JSON on the canvas — each policy file owns a DEFAULT_KWARGS module constant (transcribed from the upstream Hydra YAMLs); only checkpoint_path and num_inference_steps remain as free-form fields on the predict node.

3.3 Adding a new policy: the minimal contract

This split makes onboarding a new VLA (OpenVLA, Octo, RT-2, π₀-FAST, …) a fixed three-step recipe:

  1. Implement the policy — a BasePolicy subclass under policies/<name>_policy.py (weights + predict_action + load_checkpoint).
  2. Implement the two policy-side adapters as a single ModelAdaptor subclass under adapters/<name>_model.py:
    • canonical_to_model(canonical) → model_batch (stage 2)
    • model_to_canonical(model_output, info) → canonical_action (stage 3)
  3. Register the triple in _resolve_classes() so the policy dropdown wires the policy class to its two adapters as one atomic unit.

You do not write env-side adapters when adding a new policy — those are owned by whichever robot is in the graph. The new policy must consume the existing CanonicalDict[obs] shape produced by the env's RobotAdaptor and emit a CanonicalDict[action] that the env's RobotAdaptor.canonical_to_env can invert. That is the entire interface contract: no robot-specific code in the policy, no policy-specific code in the robot. (Cross-cutting concerns — e.g. a model's image preprocessing being incompatible with an env's image format — are handled by the canonical layer's info block, not by smuggling env knowledge into the model.)

3.4 UI rules summary

Selector Linked targets User-changeable on canvas?
Robot identity (robot select on adapt_env_to_canonical) adapt_canonical_to_env (auto-follows via singleton) Today: yes — file-discovery dropdown. End-state per §3.1: locked to graph's env identity, dropdown filtered or prompt-modal on swap.
Model identity (model select on adapt_canonical_to_model) adapt_model_to_canonical unpacking (auto-follows via singleton) Yes — change one, the mirror reader follows on next tick
Policy identity (policy select on predict) The BasePolicy class loaded onto GPU Yes — paired with model per the canonical pairings; future end-state collapses both into one select
Per-robot / per-model / per-policy kwargs The selected file's DEFAULT_KWARGS module constant No on canvas — edit the .py file (sourced from upstream Hydra YAMLs); reload to pick up
checkpoint_path, num_inference_steps predict node Yes — free-form fields, ckpt path is per-graph

Mental model. A policy_vla graph is a sandwich: the bread (env-side adapters) is fixed by which kitchen you're in (the env nodeset), and the filling (policy + 2 policy-side adapters) is the one thing the user picks — and they pick it as one item from the menu, not three independent ingredients.

4. The five canvas nodes

# Node type Backing method Owned config
1 policy_vla__adapt_env_to_canonical RobotAdaptor.env_to_canonical (via ensure_robot) robot (select — files in adapters/robots/); kwargs come from selected file's DEFAULT_KWARGS
2 policy_vla__adapt_canonical_to_model ModelAdaptor.canonical_to_model (via ensure_model_adaptor) model (select — files in adapters/models/); kwargs come from selected file's DEFAULT_KWARGS
– policy_vla__predict BasePolicy.predict_action (via ensure_policy) policy (select — files in policies/), checkpoint_path, num_inference_steps; rest of policy kwargs come from selected file's DEFAULT_KWARGS
3 policy_vla__adapt_model_to_canonical ModelAdaptor.model_to_canonical (via model_to_canonical_action) (none — reads _model_adaptor from manager singleton)
4 policy_vla__adapt_canonical_to_env RobotAdaptor.canonical_to_env (via canonical_action_to_env) (none — reads _robot from manager singleton)

4.1 policy_vla__adapt_env_to_canonical
Inputs image (IMAGE), wrist_image (IMAGE), state (ANY, 8-D float32), prompt (TEXT, optional)
Outputs canonical (ANY = CanonicalDict)
Behaviour Drops None values from env_obs before calling RobotAdaptor.env_to_canonical (line 718) — so envs without wrist (e.g. SIMPLER) can pass through None for that slot

Calls ensure_robot(robot_module) first — idempotent rebuild of the selected RobotAdaptor subclass if (module_name, kwargs) cache key changed; cached otherwise. The robot dropdown options are computed at import time by _discover_modules("adapters/robots") — every .py file in that folder (excluding __init__, files starting with _ or base_) becomes a dropdown entry. Kwargs are pulled from the selected file's DEFAULT_KWARGS module constant; Hydra-style _target_: dicts inside DEFAULT_KWARGS are recursively materialized by _instantiate().

4.2 policy_vla__adapt_canonical_to_model
Inputs canonical (ANY)
Outputs model_batch (ANY — model-specific numpy dict)
Behaviour If model selection changes, drops the loaded policy (Pi0Policy ≠ SmolVLAPolicy) so predict reloads weights

Owns the model select dropdown (today: dp_model / pi0_model / smolvla_model — every file in adapters/models/ except base_model.py and dp_defaults.py). Calls ensure_model_adaptor(model_module). Switching model invalidates the loaded _policy (since checkpoint shape is bound to the model class) so predict will reload weights on next tick.

4.3 policy_vla__predict
Inputs model_batch (ANY)
Outputs model_output (ANY — typically {"action": ndarray[K, action_dim]})
Behaviour Heavy node. Builds BasePolicy instance, loads checkpoint, moves to GPU, recomposes Adaptor. First call is the expensive one (Pi0 also does JAX→PT conversion + torch.compile — default_per_step_budget_sec = 60s)

Owns the policy select dropdown (today: dp_policy / droid_dp_policy / pi0_policy / smolvla_policy — every file in policies/ except base_policy.py), the checkpoint path, and num_inference_steps. Calls ensure_policy(policy_module, checkpoint_path=…, num_inference_steps=…, device=…). Per-policy kwargs come from the selected file's DEFAULT_KWARGS module constant. Requires adapt_canonical_to_model to have executed first so the _model_adaptor is built on the manager.

4.4 policy_vla__adapt_model_to_canonical
Inputs model_output (ANY)
Outputs canonical_action (ANY = CanonicalDict[action])
Behaviour No config. Calls ModelAdaptor.model_to_canonical(model_output, info) via the manager's model_to_canonical_action; info comes from the loaded RobotAdaptor's get_canonical_info(). Pure-CPU shape transform, no GPU

Stage 3 of the adapter pipeline. Symmetric to stage 2 (canonical→model), just on the action side. Producing canonical_action as a real wire (instead of fusing with stage 4) means action-side processing nodes — filters, value-trackers, debugging viewers — can sit between this node and adapt_canonical_to_env.

4.5 policy_vla__adapt_canonical_to_env
Inputs canonical_action (ANY), current_state (ANY, optional — needed for delta-action robots to anchor against)
Outputs action_chunk (TEXT — JSON [[ax,ay,az,arx,ary,arz,grip], …])
Behaviour No config. Calls RobotAdaptor.canonical_to_env(canonical_action, state) via the manager's canonical_action_to_env. JSON-encodes the resulting (K, 7) ndarray as a list-of-lists for the env wire

Stage 4 of the adapter pipeline. The output goes into env_libero__step.action (or env_simpler__step.action), which decodes the JSON list-of-lists into a (K, 7) action chunk and steps the env once per row. current_state comes from iter_in.iterout_state so each iteration anchors deltas on the freshest eef pose.

4.6 Wire formats (opaque between nodes)

The four intermediate wires (canonical, model_batch, model_output, canonical_action) are typed ANY in the canvas — the contract is enforced by Python dict shape, not by wire_type:

  • canonical: CanonicalDict[obs] per §2.1 (make_canonical_obs factory).
  • model_batch: model-specific numpy dict (see each ModelAdaptor.model_input()).
  • model_output: model-specific dict, normalised by the manager to at least contain "action" of shape [K, action_dim] (B=1 batch dim is squeezed).
  • canonical_action: CanonicalDict[action] — {"data": {"actions": {"pos": …, "rot": …, "gripper": …, "joint_position": …}}, "info": CanonicalInfo} (make_canonical_action factory in adapters/canonical.py:75). Same CanonicalInfo schema as obs side, but data only contains actions.

Only the final action_chunk is a typed (TEXT) wire because it crosses the env boundary.

5. VlaPolicyManager — runtime singleton

VlaPolicyManager (__init__.py:113-624) owns the entire pipeline state in this subprocess:

  • Threading: a single-thread ThreadPoolExecutor (max_workers=1) pinned to one CUDA-affine thread; all _run_sync calls funnel onto it.
  • Lock: threading.Lock guarding every public method.
  • State (instance attrs):
  • _robot: LiberoRobot | None — the RobotAdaptor instance.
  • _model_type: str — "pi0" | "smolvla" | "dp" | "droid_dp" | "".
  • _model_adaptor: ModelAdaptor | None.
  • _policy: BasePolicy | None — on GPU.
  • _adaptor: Adaptor | None — composed pair, recomposed when robot or model_adaptor changes.
  • _loaded_meta: dict — surfaced to logs.
  • Three config caches: _robot_cfg_cache, _model_adaptor_cfg_cache, _policy_cfg_cache — drive idempotency.

5.1 The three ensure_* methods

Method Caller node What it rebuilds What it invalidates
ensure_robot(robot_module) adapt_env_to_canonical The RobotAdaptor subclass picked from adapters/robots/<robot_module>.py, kwargs from its DEFAULT_KWARGS composed _adaptor (forces recompose on next ensure_policy)
ensure_model_adaptor(model_module) adapt_canonical_to_model The ModelAdaptor subclass picked from adapters/models/<model_module>.py, kwargs from its DEFAULT_KWARGS; on module change, also drops _policy composed _adaptor
ensure_policy(policy_module, checkpoint_path, num_inference_steps, device) predict The BasePolicy subclass picked from policies/<policy_module>.py, kwargs from its DEFAULT_KWARGS; calls load_checkpoint(...).to(device).eval(); recomposes _adaptor —

Cache keys are (module_name, json.dumps(kwargs, sort_keys=True)); identical key → no rebuild. The mirror nodes on the action side (adapt_model_to_canonical, adapt_canonical_to_env) carry no own dropdowns — they read _robot / _model_adaptor straight off the singleton, so flipping a dropdown on stage 1 / stage 2 automatically changes stage 4 / stage 3 behavior on the next tick.

5.2 Inference path (per step)

Every iteration of the canvas loop:

  1. adapt_env_to_canonical → _robot.env_to_canonical(env_obs) → CanonicalDict[obs]
  2. adapt_canonical_to_model → _model_adaptor.canonical_to_model(canonical) → numpy batch
  3. predict → _batch_to_torch (adds B=1) → _policy.predict_action() → numpy out (B=1 squeezed)
  4. adapt_model_to_canonical → manager.model_to_canonical_action(model_output) → _model_adaptor.model_to_canonical(model_output, info) → CanonicalDict[action]
  5. adapt_canonical_to_env → manager.canonical_action_to_env(canonical_action, current_state) → _robot.canonical_to_env(canonical_action, state={...}) → {"actions": (K, 7)} → JSON

current_state is wired into stage 4 only (not stage 3) so RobotAdaptor.canonical_to_env can convert delta actions → absolute by anchoring on the current eef pose. Stage 3 is stateless.

│   ├── action_expert/             # Pi0 action-expert internals
│   ├── openpi/                    # Pi0 backbone (vendored from openpi)
│   ├── smolvla/                   # SmolVLA backbone
│   ├── dp/                        # Diffusion Policy backbone
│   ├── droid/                     # DROID DP backbone
│   ├── ema_model.py / flow_matching.py / z_pytorch_util.py / z_utils/
│   └── convert_jax_model_to_pytorch.py   # Pi0 JAX-checkpoint → PyTorch
└── policies/                      # ⬅ BasePolicy wrappers (auto-discovered by canvas)
    ├── __init__.py                # re-exports BasePolicy + 5 concrete subclasses
    ├── base_policy.py             # BasePolicy (abstract); excluded from discovery
    ├── pi0_policy.py              # Pi0Policy + DEFAULT_KWARGS
    ├── smolvla_policy.py          # SmolVLAPolicy + DEFAULT_KWARGS
    ├── dp_policy.py               # DiffusionUnetHybridImagePolicy + DEFAULT_KWARGS
    ├── droid_dp_policy.py         # DroidDiffusionPolicy + DEFAULT_KWARGS
    ├── rt1_policy.py              # Rt1Policy — TF SavedModel wrapped as a degenerate nn.Module;
    │                              #   DEFAULT_KWARGS = {policy_setup: widowx_bridge}
    └── rt1_policy_google_robot.py # variant — re-exports Rt1Policy with
                                   #   DEFAULT_KWARGS = {policy_setup: google_robot}

Note that rt1_inference.py sits at the package root (alongside __init__.py), not under policies/. That's intentional: it's vendored upstream code (the TF SavedModel inference wrapper from SimplerEnv), not a BasePolicy subclass. Rt1Policy (the thin nn.Module wrapper that the framework actually instantiates) lives under policies/ like every other policy. By keeping vendored code separate from the framework-conformant wrapper, re-vendoring upstream changes is a single-file overwrite.

Horizontal split mirrors vlaworkspace upstream: adapters/robots/ is the env-coupled side (stages 1+4), adapters/models/ is the model-coupled side (stages 2+3). Three layers of vendored code total: adapters/ is the user-facing abstraction, top-level models/ is the raw nn.Module weights, policies/ is the inference wrapper that holds weights + checkpoint loading + predict_action.

Discovery rules — _discover_modules(subpkg_relpath) (in __init__.py) lists .py stems in the target folder, excluding __init__, files starting with _ or base_, and the literal name dp_defaults. The result is sorted and exposed as the dropdown options on the matching canvas node (adapters/robots/ → robot select on stage 1; adapters/models/ → model select on stage 2; policies/ → policy select on predict). The scan is filename-only — no module imports happen at scan time, so heavy deps (torch, diffusers, transformers) stay out of the framework process until the user actually picks an option.

8. Conda env and server-mode hosting

class PolicyVlaNodeSet(BaseNodeSet):
    name = "policy_vla"
    server_python = os.environ.get(
        "VLA_POLICY_PYTHON",
        "python",
    )
    parallelism = "replicated"   # per-worker torch + CUDA state
    default_per_step_budget_sec = 60.0   # Pi0 first-call JAX→PT + torch.compile
  • Hosted in its own subprocess via auto_host (ADR-server-001), bound to the ac-vla-policy conda env.
  • Install script: scripts/install/install_ac_vla_policy.sh.
  • parallelism="replicated" means each eval worker spins its own subprocess with its own GPU policy — no model sharing. batched=False everywhere (no shared rendezvous tier).

9. Extension cookbook (current understanding)

§3 captures the design principle for extension (env-side locked, policy-side as a triple). This section captures the mechanical steps observable in the current code. Where the two diverge — most notably the lack of a robot-pluggability mechanism in §9.2 — the principle wins; the code needs to catch up.

9.1 Adding a new model (e.g. OpenVLA)

Code-level steps in the current source — drop-in pattern, no central registry edits:

  1. Add a ModelAdaptor subclass at adapters/models/<name>_model.py. Implement canonical_to_model, model_to_canonical, get_norm_stats_keys, get_norm_stats_mode, canonical_to_norm_stats_format, model_input, model_output. Use the base class's _load_norm_stats_from_path for normalisation if your stats fit the standard schema. Append a DEFAULT_KWARGS: dict = {...} module constant transcribed from the upstream Hydra YAML (or any other source of truth for that model's defaults). Hydra-style _target_: dicts are supported — _instantiate() recursively materializes them.
  2. Add a BasePolicy subclass at policies/<name>_policy.py. Implement at least predict_action(batch) → dict, load_checkpoint(path), to(device), eval(). (Or reuse an existing one if your model's inference matches — DP and DROID-DP share DPModel but not the policy class.) Append its own DEFAULT_KWARGS.
  3. Vendored backbone: drop the actual nn.Module code under top-level models/<name>/. AgentCanvas vendors verbatim, doesn't import from upstream repos (per __init__.py docstrings).
  4. Hot-reload (POST /api/components/reload) — both new files appear in the model and policy dropdowns immediately. No _resolve_classes() edit, no _MODEL_TYPES edit, no ConfigField.options edit.
  5. Optional — re-export from adapters/models/__init__.py and policies/__init__.py if you want from policy_vla.adapters import OpenVLAModel to work for direct Python callers (the canvas path uses _load_module + _find_subclass and doesn't need the re-export).
  6. Class resolution — model/policy/robot classes are resolved by folder-scan discovery (ROBOT_OPTIONS / MODEL_OPTIONS / POLICY_OPTIONS) and instantiated via ensure_model_adaptor / ensure_policy; drop a file in, POST /api/components/reload. (The legacy hardcoded _resolve_classes() table + load_model() path — used only by the since-removed smoke_vla_libero.py runner — was deleted 2026-06-30.)

Open questions for this path:

  • What about norm_stats files that don't fit the mean/std/q01/q99 schema (e.g. OpenVLA's 256-bin discretisation)? The base class's loader assumes the schema; non-conforming stats need either a subclass override or a different loading path.
  • See §10 for the unresolved-questions list.

9.1.1 Worked example — onboarding RT-1-X (2026-05-04)

RT-1-X is the first VLA whose lineage is not vlaworkspace's Pi0/SmolVLA/DP family — it's a Google TF SavedModel served via tf-agents, with its own image preprocessing, USE language embedding, and per-embodiment action rescaling. Folding it into policy_vla without bending the rules in §3 is the canonical proof that "every VLA = adapter+policy+adapter" generalises beyond the original family.

What got added (no new node, no new nodeset):

File Role
policy_vla/rt1_inference.py Vendored verbatim from third_party/SimplerEnv/simpler_env/policies/rt1/rt1_model.py. Imported through a relative path so simpler_env/__init__.py's import mani_skill2_real2sim.envs doesn't fire — keeps SAPIEN out of ac-vla-policy.
policy_vla/adapters/models/rt1_model.py Rt1Model(ModelAdaptor). Pass-through: stage 2 packs {image: HWC uint8, instruction: str} from CanonicalDict[obs]; stage 3 unpacks (K, 7) action into CanonicalDict[action] with pos / rot / gripper slots. No norm-stats — RT-1's per-embodiment rescaling is baked into the SavedModel + the _unnormalize_action_widowx_bridge / _small_action_filter_google_robot paths in rt1_inference.py.
policy_vla/policies/rt1_policy.py Rt1Policy(BasePolicy). Inference-only nn.Module wrapper: registers a single frozen nn.Parameter(zeros(1)) so policy.to(device), policy.eval(), policy.parameters(), policy.device all work without touching the TF graph. predict_action(batch) strips _batch_to_torch's wrapping (image becomes torch uint8 (1,H,W,3) → numpy uint8 (H,W,3)), calls RT1Inference.step(image, instruction), packs the 7-D action into (B=1, K=1, 7) torch — the manager's predict() then squeezes the leading 1. compute_loss raises NotImplementedError. DEFAULT_KWARGS = {"policy_setup": "widowx_bridge"}.
policy_vla/policies/rt1_policy_google_robot.py Variant — re-exports Rt1Policy with DEFAULT_KWARGS = {"policy_setup": "google_robot"}. Same pattern as libero_robot_absolute.py: drop a file = add a dropdown entry that shares the class.

That's it. Zero edits to canvas node code, zero edits to VlaPolicyManager ensure_* logic, zero edits to graph topology. The 5-stage graph at workspace/graphs/vla_policy_simpler.json selects RT-1-X by setting model=rt1_model, policy=rt1_policy (or rt1_policy_google_robot), robot=simpler_robot — three filenames, three dropdowns.

What this proves:

  • The framework's "any robot × any model" Cartesian product is real — Rt1Model doesn't know which robot it's paired with; SimplerRobot doesn't know which model is downstream; the contract between them is CanonicalDict + CanonicalInfo.
  • BasePolicy = nn.Module doesn't force every VLA to be torch. RT-1's actual model is TF; the dummy parameter trick lets a TF/JAX/MLX backend live behind the BasePolicy interface as long as it overrides predict_action and load_checkpoint.
  • Dependency stacks coexist when they're additive. RT-1 adds tensorflow==2.15, tf-agents==0.19, tensorflow-hub==0.16, transforms3d to the existing ac-vla-policy env. Torch 2.6, JAX 0.5.3, Flax 0.10.2 all stay put. (Octo's jax==0.4.20 + flax==0.8.1 pins are not additive — that's why Octo is currently a separate nodeset; see §10.9.)

9.2 Adding a new robot (e.g. SIMPLER bridge)

  1. Add a RobotAdaptor subclass at adapters/robots/<name>_robot.py. Implement all 9 abstract methods. Append a DEFAULT_KWARGS: dict = {...} module constant.
  2. Hot-reload — the new file appears in the robot dropdown on adapt_env_to_canonical immediately. The manager no longer hardcodes LiberoRobot; ensure_robot(robot_module) resolves <name>_robot.py → RobotAdaptor subclass via _load_module + _find_subclass.
  3. Optional — re-export from adapters/robots/__init__.py for direct Python callers.

Mechanical pluggability is now in place. What's still TBD is the §3.1 principle constraint — that the robot dropdown should be filtered (or interlocked) by the graph's env nodes, so that picking a franka_robot on a LIBERO graph isn't possible. Today the dropdown lists every file in adapters/robots/, and the user is responsible for picking a compatible one. The libero_robot_absolute.py demo variant exists precisely to prove this: it shows up in the dropdown next to libero_robot, both are LIBERO-compatible, and the user picks between them by file name. Cross-env contamination is an open guard.

10. Open questions for the maintainer

The following are points where the code's current shape leaves the design intent ambiguous. The maintainer will resolve these by adding principles inline.

  1. Robot pluggability — env-locking interlock. §3.1's principle ("env-side adapters are locked to the graph's env nodes") is half-implemented: the mechanical pluggability is in place — ensure_robot(robot_module) resolves any <name>_robot.py via _load_module + _find_subclass, no LiberoRobot hardcode. What's still missing is the interlock — today the robot dropdown lists every file under adapters/robots/, so a user can pick franka_robot on a LIBERO graph and the pipeline will fail with an obs-shape mismatch deep in the model. End-state: the env nodeset advertises a compatible-robot list (e.g. env_libero → [libero_robot, libero_robot_absolute], env_simpler → [simpler_robot]) and the dropdown filters to that list, OR the canvas pops a confirmation modal on env swap.
  2. What counts as "a new VLA"? ~~Resolved by RT-1-X integration (2026-05-04):~~ Every VLA, regardless of lineage, joins as a (ModelAdaptor, BasePolicy) pair inside policy_vla. RT-1-X's TF SavedModel + USE-embedding pipeline fit cleanly behind the BasePolicy + ModelAdaptor interfaces — see §9.1.1. The only legitimate reason to spawn a separate nodeset is a hard Python environment conflict (Octo, currently — see §10.9), not a model-architecture difference. When/if Octo's jax/flax pin gets unstuck, it folds in too.
  3. ~~What goes in presets/?~~ Resolved (2026-05-04) — presets/ is gone. A centralized presets directory was explicitly rejected. Per-adapter / per-policy defaults now live as DEFAULT_KWARGS module constants inside each .py file, transcribed from the upstream Hydra YAMLs. Adding a defaults variant = drop a new .py file in the right folder; removing = delete the file. Per-checkpoint preset blobs (if ever needed) would attach to specific checkpoints, not to a global registry.
  4. Norm-stats schema. _load_norm_stats_from_path expects mean/std/q01/q99 per key. What's the design for stats that don't fit (e.g. OpenVLA's 256-bin token boundaries, RT-2's no-norm path)?
  5. canonical_to_env contract on action chunks. Today emits {"actions": ndarray (K, 7)} and adapt_canonical_to_env JSON-encodes. For non-7-DoF embodiments (mobile manipulators, wheeled bases, dual-arm) is (K, 7) still the contract or does the wire type evolve?
  6. Cross-env state vector. current_state is documented as 8-D LIBERO state at adapt_canonical_to_env. Is there a canonical state schema across envs, or does each RobotAdaptor own its own state shape and the wire just stays ANY?
  7. models/ vs policies/ split. Why two layers (raw nn.Module vs Policy wrapper) instead of one? Is a new VLA expected to follow the split or can it ship as a single fused class?

  8. Composite vs separate nodeset. When does a new VLA join policy_vla vs spawn its own nodeset? Resolved alongside Q2 (2026-05-04): every VLA joins policy_vla unless its Python deps hard-conflict with the shared ac-vla-policy env. Architecture, lineage, and training framework are all irrelevant to nodeset identity — only the env conflict matters. See §10.9.

  9. Octo and the env-conflict escape hatch. Octo (RAIL Berkeley, JAX/Flax) is the one VLA that did not fold into policy_vla on its first integration pass. Its upstream octo-1.0 pin is jax==0.4.20 + flax==0.8.1 — two major versions behind ac-vla-policy's jax==0.5.3 + flax==0.10.2, with breaking API changes in between (flax.training.checkpoints removed, nn.Module init/apply tightened, jax.random.KeyArray deleted, etc.). Downgrading would break Pi0/openpi; upgrading Octo source needs ~1–2 days of port + correctness validation against the published baseline. Octo therefore lives in its own policy_octo nodeset on a separate ac-octo conda env as the explicit escape hatch, not as a new pattern. The pattern remains "everything in policy_vla"; a separate nodeset is the temporary refuge until the env conflict resolves. Once Octo is ported forward (or once we adopt a JAX-version-agnostic loader), it folds back as octo_model.py + octo_policy.py like every other VLA.
