Policy Adapter Pipeline (VLA & VLN)

policy_vla · policy_vlnce · CanonicalDict · four adapter stages + one model-forward · singleton manager · ensure_* slices · filename discovery

AgentCanvas runs a neural policy not as one black-box node but as a short pipeline of adapter stages mediated by a CanonicalDict lingua franca. Two nodesets are built on this one shape — policy_vla (manipulation: LIBERO / SIMPLER) and policy_vlnce (VLN-CE navigation: Habitat R2R-CE / RxR-CE). The hard promise: the pipeline shape never changes. An env-side adapter and a model-side adapter sit on either side of the model, each pipeline stage is exactly one canvas node, and adding a new policy or a new env is always "drop a .py file in a folder, pick it from a dropdown" — never a new node type, never a bypass path. This page documents the shared paradigm and the precise places the two instances diverge. For the VLA-side deep dive (robot/policy triples, the onboarding recipe, RT-1-X) see VLA Policy; for the env contract see env_habitat.

1. What it does

A trained policy needs the env's raw observation reshaped into its input tensor, and its raw output reshaped back into the env's action. Rather than bury those two translations inside one opaque "run the policy" node, AgentCanvas splits each forward pass into four adapter stages bracketing one model-forward node, and exposes every stage as its own canvas node:

  1. env → canonical — the env-side adapter wraps the raw observation into a CanonicalDict[obs].
  2. canonical → model — the model-side adapter packs the canonical obs into the model's input batch.
  3. predict — the GPU policy's forward pass (the only heavy node).
  4. model → canonical — the model-side adapter unpacks the raw output into a CanonicalDict[action].
  5. canonical → env — the env-side adapter turns the canonical action into an env action.

The two adapters never know about each other. The env-side adapter knows nothing about which model consumes its canonical; the model-side adapter knows nothing about which env produced it. They communicate only through the CanonicalDict + its info carrier. That decoupling is the whole point: any env × any model is a valid pair, and onboarding either side is a local edit that doesn't touch the other.

Why each stage is its own node rather than a fused "predict" box:

  • Each intermediate (canonical, model_batch, model_output, canonical_action) is a real wire — inspectable, interceptable, and a future seam for action-filter / debug-viewer nodes.
  • Each stage maps to one abstract method on the env-side or model-side adapter base class, so swapping a stage on the canvas is symmetric to swapping a class in code.
  • Only predict touches the GPU; the four adapter nodes are pure-CPU dict transforms, cheap to re-run when only config changes.

A note on stage numbering. policy_vla's code calls these "4 adapter stages + 1 model forward" (predict is unnumbered); policy_vlnce's docstring calls it a "5-stage pipeline" (predict is stage 3). Same five nodes either way — this page uses "four adapter stages bracketing predict" to stay neutral. (policy_vlnce earlier shipped a sixth tokenize_instruction node; retired 2026-06-29 when tokenization moved into canonical_to_model — see §7.)

1.1 Mental model — standardize, then policy-process

The pipeline has one load-bearing division of labour: the env-side adapter only standardizes; every kind of processing happens after the waist, on the model side. Standardizing means reformatting the env's raw observation into one uniform shape without changing its content; processing means everything a specific policy needs — resize, normalize, augment, tokenize.

StageJobMay doMust NOT do
env→canonical (env-side)standardize onlyformat / convention normalization: RGBA→RGB, add depth channel, dtype, units→metres, HWC layout, key naming — goal: any env yields the same canonical shape, policy-agnostic, content unchangedresize, crop, mean/std normalize, noise, augmentation, tokenize
CANONICALthe standardized raw dataenv-native content in one uniform, physical, model-neutral format; nothing added or removed—
canonical→model (model-side)all policy-specific processingresize, center/random-crop, normalize, noise, history stacking, action chunking, instruction tokenize — each policy's own pipeline—

Six rules that fall out of this:

  1. Decoupling. M policies + N sims → write only M+N adapters; any pair composes (no M×N glue).
  2. Standardize ≠ process. env→canonical reformats only (content unchanged); canonical→model does resize / normalize / tokenize / augment.
  3. Physical, de-tokenized, neutral. The canonical carries physical quantities; each model's tokenization / normalization / chunking lives in its own model-side adapter, never in the waist.
  4. The env stays dumb. The simulator emits its raw output and knows nothing of the canonical; structure is imposed by the adapter.
  5. info is a plain dict on the wire (msgpack-safe), never a frozen dataclass.
  6. Incompatibility is handled elsewhere. A modality a policy needs but a sim can't produce is not the canonical's concern.

Scope. The shipped code targets manipulation VLAs (OpenVLA / Pi / Diffusion Policy) on a single arm + gripper in joint-position or eef-pose space; the per-part embodiment registry for other morphologies (dual-arm, mobile base, WAM torque, VLN navigation) is parked until a real second morphology arrives. policy_vlnce follows this split as of 2026-06-29: the env adapter standardizes (raw rgb / depth / raw instruction text), and tokenization + obs_transforms run in canonical_to_model (§7).

1.2 Smallest instance

The minimal pipeline is the five-node chain wired in series between an env's observe/step pair:

observe ─▶ adapt_env_to_canonical ─▶ adapt_canonical_to_model ─▶ predict
                                                                      │
        step ◀─ adapt_canonical_to_env ◀─ adapt_model_to_canonical ◀─┘

For policy_vlnce the recurrent CMA/Seq2Seq policy adds two wires (predict.hidden_out → iter_out.hidden, iter_in.iterout_hidden → predict.hidden_in) ; the per-episode instruction text rides in as a raw string through an iter_in init slot (CMA-vocab tokenization happens later, model-side). The verified R2R-CE graph that runs this end-to-end is workspace/graphs/vln/verified/straightforward.json (15 nodes, 27 edges).

2. The CanonicalDict lingua franca

Every env maps to this format; every model maps from it. It is a plain {"data": …, "info": …} dict travelling on an ANY-typed wire — the contract is enforced by dict shape, not by wire_type. The two nodesets carry domain-specific payloads but the same envelope.

2.1 VLA — manipulation payload

policy_vla/adapters/canonical.py · make_canonical_obs. Multi-camera images + a structured robot state + a language prompt; the info carrier (CanonicalInfo) records rotation representation and key dims so a model adapter can normalise without knowing the robot:

CanonicalDict[obs] = {
    "data": {
        "images":  {"front": ndarray | None, "wrist": ndarray | None},
        "state":   {"pos": ..., "rot": ..., "gripper": ..., "joint_position": ...},
        "actions": {"pos": ..., "rot": ..., "gripper": ..., "joint_position": ...},  # output side
        "prompt":  str,
    },
    "info": CanonicalInfo(state_type, state_rot_repr, action_type,
                          action_rot_repr, state_dims, action_dims),
}

2.2 VLN — navigation payload

policy_vlnce/adapters/canonical.py · make_canonical_obs. RGB-D frame + the raw instruction (raw text for R2R-CE, precomputed BERT feature for RxR-CE) + step metadata; the info carrier (CanonicalNavInfo) records env family, action cardinality, and instruction kind. The canonical carries standardized raw data only — tokenization is model-side (§1.1):

CanonicalDict[obs] = {
    "data": {
        "rgb":         ndarray uint8  (H, W, 3),
        "depth":       ndarray float  (H, W, 1),       # metres
        "instruction": {"kind": "raw_text", "text": str}                      # R2R-CE
                     | {"kind": "feat",     "embedding": ndarray (T, 768)},   # RxR-CE
        "step_meta":   {"step_index": int, "episode_id": str},
    },
    # asdict() so a frozen dataclass survives the msgpack ANY-wire
    "info": asdict(CanonicalNavInfo(env_family, action_dim, image_size,
                                    depth_max, instruction_kind)),
}

The canonical action mirrors the asymmetry: VLA's is a dict of continuous action components ({"data": {"actions": {pos, rot, gripper, …}}}) that decode to a (K, 7) chunk; VLN's is a single discrete index ({"data": {"action_index": int, "action_dim": int}}).

3. The stages as canvas nodes

Each stage is one node backed by one abstract method on the env-side or model-side adapter. The node names differ only by the policy_vla__ / policy_vlnce__ prefix:

— · forwardpredictBasePolicy.predict_actionVlnPolicy.forward

The two action-side nodes (stages 3 + 4) carry no config in either nodeset — they read the already-built adapter slot off the manager singleton, so they auto-follow whatever model/env the obs side selected. The obs-side nodes own the config; see §5 for where exactly it lives in each nodeset.

4. The singleton manager and its ensure_* slices

Both nodesets run in server mode (a dedicated conda env: ac-vla-policy / vlnce) with parallelism="shared": one subprocess hosts one manager singleton (VlaPolicyManager / VlnceManager), and K eval workers fan in through it. The manager owns the GPU policy and the adapters, and pins all torch/CUDA work to a single-thread executor for CUDA thread-affinity.

State is built lazily through idempotent ensure_* slices, one per config-bearing concern. Each caches a (module, kwargs) key; a cache hit short-circuits, a config change rebuilds only that slice. The slices form a dependency chain — changing the model adapter drops the loaded policy (a Pi0 checkpoint can't load into a SmolVLA net), changing the env adapter or canonical shape invalidates downstream:

ConcernVLA sliceVLN sliceOwned by node
env-side adapterensure_robotensure_env_adapterstage 1
model-side adapterensure_model_adaptorensure_model_adaptorstage 2 (VLA) / stage 1 (VLN, see §5)
GPU policy + checkpointensure_policyensure_policypredict (VLA) / stage 1 (VLN, see §5)

Because the manager is a singleton and the adapter nodes are pure readers of its slots, the heavy work (checkpoint load, GPU placement) happens once and is amortised across every step and every worker.

5. Filename discovery — drop-in extension

Neither nodeset hard-codes its variant list. At import time each scans three folders and turns every .py stem (excluding __init__, base_*, _*) into a dropdown option:

FolderBase class & module contractVLA filesVLN files
adapters/robots/ (VLA) · adapters/envs/ (VLN)env-side adapter subclass + DEFAULT_KWARGSlibero_robot, libero_robot_absolute, simpler_robotr2rce, rxrce
adapters/models/model-side adapter subclass + DEFAULT_KWARGSpi0_model, smolvla_model, dp_model, rt1_modelcma, seq2seq
policies/BasePolicy / VlnPolicy subclass + DEFAULT_KWARGSpi0_policy, smolvla_policy, dp_policy, droid_dp_policy, rt1_policycma_policy, seq2seq_policy

Each file carries its own DEFAULT_KWARGS module constant (transcribed from the upstream Hydra / YAML config), so a variant is fully described by its file — no canvas-side JSON config. Drop a file in or out, then POST /api/components/reload. Adding a policy is the same recipe in both domains: one model-side adapter file (both directions) + one policy file, consuming the existing CanonicalDict and emitting a canonical action the env-side adapter can already invert. (The VLA onboarding recipe with the robot×model pairing table is spelled out in VLA Policy §3.3.)

The single biggest structural difference is config placement. VLA treats robot, model, and policy as three independent choices on three nodes (because any robot × any model is meant to be mixable). VLN bundles all four strings (model adapter, exp-config YAML, policy module, checkpoint) into one variant selection on the chain-entry node, because a VLN-CE baseline is a fixed (architecture, training-config, checkpoint) tuple — picking "CMA +PM+DA+Aug" must pick all four together or the checkpoint won't load. The variant dropdown is populated from variants.REGISTRY (12 R2R-CE entries; only CMA_PM_DA_Aug and Seq2Seq_DA have released checkpoints — the other 10 are listed for topological completeness with a "(not released by paper authors)" label).

7. Where the instances deviate from the clean model

The shared paradigm is clean; the shipped code carries warts worth naming.

VLN config is not distributed — it's bundled on the chain-entry node. The clean model (and policy_vla) says "each config-bearing node owns one slice." policy_vlnce instead puts a single variant dropdown on stage 1 (adapt_env_to_canonical), and that node eagerly calls ensure_env_adapter + ensure_model_adaptor + ensure_policy up front. Stages 2 and 3 are config-less and just read the primed singleton. (The policy_vlnce/__init__.py module docstring and the stage-2 node description still describe the older distributed-config / "env panel variant" design — that prose is stale; the chain-entry variant is what ships.)

VLN's info crosses the wire as a plain dict, not its dataclass. CanonicalNavInfo is a frozen dataclass; the msgpack ANY-wire (app/server/serialization.py) can't pack one and returns a 500. make_canonical_obs therefore stores dataclasses.asdict(info), and validate_canonical_obs accepts either form. No wire consumer reads it back — downstream stages re-derive info via env_adapter.get_canonical_info(), so the wire copy is metadata only. policy_vla's make_canonical_obs still stores the raw CanonicalInfo dataclass; the two ports are asymmetric here.

VLN instruction: raw text in, tokenized model-side, required port. Per §1.1 the env adapter standardizes only — the per-episode instruction rides in as a raw string and the canonical carries it as {"kind":"raw_text","text"}. CMA's vocab tokenization (a faithful replica of the offline VocabDict preprocessing — verified 200/200 vs the dataset's precomputed tokens, self-contained adapters/r2r_instruction_vocab.json) runs in cma.canonical_to_model, memoized per episode. The raw text reaches stage 1 through an iter_in init slot (persist=true); the instruction input is required, not optional so the executor waits for it before firing (optional → stage 1 races ahead with instruction=None → 0-step episodes).

VLN predict runs batched=False despite parallelism="shared". This is a workaround, not a design choice: batched=True hits a framework asyncio loop-affinity bug under worker_count>1 (BatchedInferenceServer._delayed_flush future attached to the wrong event loop). Per-sample sequential is correct meanwhile — the shared GPU policy serialises forward passes anyway, so the throughput cost is negligible while env steps still parallelise across workers. policy_vla keeps batched=True (its TF SavedModel pins batch=1, so the K-flush is unrolled into a loop; the win there is one model on GPU, not throughput).

The VLA robot dropdown is a temporary state. Per VLA Policy §3, the env-side adapter's identity should be locked to whatever the graph's env nodes imply (a LIBERO graph can't use a SIMPLER robot adapter) — the free robot select will eventually be filtered or gated by the env nodeset. VLN sidesteps this by deriving its env module from the variant (r2rce is currently pinned for all 12 R2R-CE variants).

policy_vlnce now honours the standardize/process split (resolved 2026-06-29). The env adapter (r2rce) standardizes only — raw rgb / depth / raw instruction text, no _full_raw_obs passthrough; the CMA ModelAdaptor rebuilds its obs dict from the canonical and tokenizes there. The standalone tokenize_instruction node was retired (graph 16→15 nodes). Refactor verified behaviour-preserving: 3-ep val_unseen bit-identical to the pre-refactor baseline (success 0.333 / SPL 0.333 / nDTW 0.378 / path 10.33 / steps 56.7).

8. Provenance

The VLA adapter system (CanonicalDict, CanonicalInfo, the RobotAdaptor / ModelAdaptor base classes, the composed Adaptor) is vendored verbatim from the upstream vlaworkspace training repo per AgentCanvas's no-upstream-import policy — source copied with rewritten imports, bug-for-bug parity with the vendoring snapshot, no live import at runtime. policy_vlnce is the VLN-flavored sibling: the same envelope and stage shape, re-authored for navigation (the VlnEnvAdaptor / VlnModelAdaptor / VlnPolicy hierarchy is local, the CMA / Seq2Seq nets are vendored from vlnce_baselines). Both nodesets keep workspace/ standalone — no runtime dependency on third_party/.
