CI / CD

The repo's CI/CD in one line: a GitHub Actions workflow rebuilds the doc-site's nav/search data (one stdlib script, zero dependencies) and deploys it to GitHub Pages. There is no test CI yet — those checks run locally via pre-commit.

AgentCanvas's CI/CD is a single GitHub Actions workflow, .github/workflows/docs.yml — continuous deployment of the doc-site to GitHub Pages, with one dependency-free build step: the stdlib wrap pass that regenerates chrome + nav/search data from the checked-out tree. There is no test CI yet; code-quality gates (lint, format, secret scan, commit format) run locally via pre-commit, so keep running them before you push.

§1The "Deploy docs" workflow

It runs on a push to main that touches docs/**, .gitignore (which drives nav/search visibility), or the workflow file itself, and can also be triggered manually (workflow_dispatch). It runs the one build step — python3 docs/_lib/_wrap_handwritten.py, pure stdlib, no installs — to regenerate the baked chrome and the nav/search data from the checked-out tree, drops the dev-only *.local.json twins, then uploads docs/ as the Pages artifact and deploys it. A concurrency group ensures only one deploy runs at a time.

name: Deploy docs

on:
  push:
    branches: [main]
    paths:
      - 'docs/**'
      # nav/search visibility is gitignore-driven (docs/_lib/_nav.py), so a
      # .gitignore-only change can alter the public nav — redeploy on it too.
      - '.gitignore'
      - '.github/workflows/docs.yml'
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

# Avoid in-flight overlap; only one deploy at a time.
concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/checkout@v4

      # The one build step: regenerate baked chrome + nav.json/search-index.json
      # from the checked-out tree (pure stdlib, no pip installs). The generated
      # nav/search data is untracked — this is its single source of truth for
      # the published site. The *.local.json twins only matter for the local
      # dev server; on a public checkout they'd duplicate the public files, so
      # drop them from the artifact.
      - name: Build nav + chrome (stdlib-only wrap pass)
        run: |
          python3 docs/_lib/_wrap_handwritten.py
          rm -f docs/assets/*.local.json

      - uses: actions/configure-pages@v5

      - name: Upload Pages artifact
        uses: actions/upload-pages-artifact@v3
        with:
          # Hand-authored docs/ tree + the freshly built nav/search data above.
          path: ./docs

      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

§2What it implies

  • The whole docs/ tree is published. The artifact is path: ./docs — every file under docs/ ships to Pages as-is. Large binaries that don't need publishing should therefore live outside docs/ (e.g. the README hero GIFs are kept at the repo-root assets/readme/, not under docs/).
  • One dependency-free build step, no markdown step. The page HTML is hand-authored; the deploy job runs the same stdlib _wrap_handwritten.py pass the local dev server runs, so nav.json / search-index.json are generated at deploy time from the committed tree and are not tracked in git — the published nav can never go stale against the published pages (see html-authoring).
  • Triggered by docs / visibility changes only. A push that doesn't touch docs/**, .gitignore, or the workflow does not redeploy. Use workflow_dispatch to force a deploy.
  • No test/lint CI yet. Quality gates are local-only via pre-commit. Adding backend/frontend test CI is a future item.
