Git Conventions

How commits, branches, and pull requests are named in AgentCanvas — the commit format is machine-enforced by pre-commit.

AgentCanvas follows Conventional Commits. The commit-message format is enforced at the commit-msg stage by the conventional-pre-commit hook; branch and PR conventions are by agreement. The canonical source is .claude/standard/git-conventions.md.
on this page Commit messages· Branch naming· Pull requests

§1Commit messages

Format: type(scope): description.

  • Type (required) — one of: feat, fix, refactor, docs, chore, test, ci, perf.
  • Scope (optional) — one of: backend, frontend, eval, orchestrator, data, infra.
  • Description — lowercase, imperative mood, max 72 characters.
feat(backend): add state container system
fix(frontend): correct edge rendering on zoom
docs(infra): update architecture diagram
refactor(backend): simplify graph executor loop
Enforcement The conventional-pre-commit hook validates the subject line at commit time, but only after you enable the commit-msg stage: pre-commit install --hook-type commit-msg (see Pre-commit Hooks § Install). The accepted types are exactly the list above.

§2Branch naming

Branch from main; name the branch after the change type:

feat/short-description
fix/short-description
refactor/short-description
docs/short-description

§3Pull requests

PRs cover all contributions — a nodeset or graph under workspace/ gets a light review; a core change is welcome too (open a Discussion first if it's big). Keep one concern per PR, ensure the local checks pass, and follow the PR description template. The full process — what gets reviewed, required checks, and the description format — is in Contributing § PR format. PR titles follow the same type(scope): description convention as commits.
