Auto-Memory Store

How Claude Code's auto-memory is stored and managed in this repo — a symlinked, in-repo store, routed by topic, and shared selectively via git.

Claude Code keeps a persistent, file-based auto-memory — one fact per markdown file, indexed by a MEMORY.md that is loaded into context at the start of every session. By default this store lives per-project under ~/.claude/ and never enters the repository. AgentCanvas overrides that in three ways: the store is relocated into the repo (so git can version it), routed into topic subdirectories instead of a flat folder, and shared selectively — project knowledge travels between machines, personal preferences stay local. The routing rules are the source of truth in CLAUDE.md § Memory Layout; this page explains the mechanism behind them.

§1The default: per-project, flat, frontmatter

Out of the box, Claude Code's auto-memory is strictly per-project — there is no global, cross-project memory. Each project gets its own folder under ~/.claude/projects/<encoded-path>/memory/, where the encoded path is the repo's absolute path with every non-alphanumeric character replaced by -. Inside that folder:

  • One fact = one markdown file, written flat into the folder (no subdirectories).
  • A MEMORY.md index lives alongside the files and is loaded into context each session — it is how the agent knows what memories exist.
  • Four frontmatter types classify each file: user (who the user is), feedback (how to work), project (ongoing work / constraints), and reference (pointers to external resources). The filename prefix mirrors the type (feedback_*, project_*, user_*).

Because it sits under ~/.claude/, the default store is local to one machine and is never committed — switch machines and the memory is gone.

§2Our override: an in-repo store, reached by symlink

AgentCanvas keeps its memory inside the repository at .claude/memory/ so it can be versioned and synced like any other file. Claude Code still reads from the per-project path it expects — the two are bridged by a symlink, set up once per machine (§7):

~/.claude/projects/-data-ws-vln-vlnworkspace/memory   # path Claude Code reads at session start
        |  symlink
        v
<repo>/.claude/memory/                                # the real store — versioned inside the repo

The symlink is transparent: Claude Code follows it when loading MEMORY.md and when surfacing a relevant memory mid-session, so every memory behaviour works unchanged — the bytes just happen to live in the repo. On top of relocation, this repo adds topic routing: instead of a flat folder, a new memory is dropped into the subdirectory that matches its topic, never the .claude/memory/ root.

§4Anatomy of a memory file

Each file is one fact: YAML frontmatter (name, description, type, plus an originSessionId stamp) followed by the fact itself. For feedback and project memories the body carries a **Why:** line (what prompted the rule) and a **How to apply:** line (the concrete action), so a future session can act on it without re-deriving the context.

---
name: Graph JSON is configuration of hidden Python code
description: Don't describe graph JSON as "not configuration of hidden code" — it IS exactly that.
type: project
originSessionId: 98098be4-f739-4da1-a5c3-168a8f468ac3
---
A graph JSON is a configuration of hidden Python code. Each node `type` string
resolves to a registered BaseCanvasNode subclass; the JSON supplies the wiring.

**Why:** I previously framed graphs as "not a configuration of hidden code" — the
user corrected this; the framing was wrong.

**How to apply:** Frame a graph as "the complete declarative spec of an agent's
topology, wiring, and config — node behaviour lives in registered Python classes."
One fact per file. Memories are deliberately small and single-purpose. Before saving a new one, check for an existing file that already covers the topic and update it rather than spawning a near-duplicate; delete memories that turn out to be wrong. Don't store what the repo already records (code structure, past fixes, git history, CLAUDE.md).

§5The index: MEMORY.md

MEMORY.md is the only file loaded into context at session start — it is the table of contents the agent reads to know what it remembers; the individual files are read on demand (or surfaced automatically when relevant). Every memory gets one pointer line under a per-directory header. Because the routing is topic-based, the index uses one ## subdir/ — description header per directory, with subdir-prefixed links:

## platform/ — agentcanvas framework semantics (node / wire / state / runtime)

- [project_graph_is_config.md](platform/project_graph_is_config.md) — Graph JSON IS configuration of hidden Python code; frame as "complete topology spec".

The index is itself git-tracked (see below) and edited append-only. That fixes an earlier failure mode where a memory file synced across machines via git but its index line didn't — leaving the receiving machine unable to find a memory it physically had. Append-only edits also keep merge conflicts manageable.

§6Shareability: a per-directory .gitignore split

Not all memory should travel. Project knowledge — framework semantics, nodeset facts, research framing — is worth carrying between machines; personal context — collaboration habits, the user's environment — is local business. The split is enforced in .gitignore: ignore the whole store, then whitelist the directories that should sync.

.claude/memory/*
!.claude/memory/MEMORY.md
!.claude/memory/architect/
!.claude/memory/platform/
!.claude/memory/nodeset/
!.claude/memory/eval/
!.claude/memory/research/

So architect/, platform/, nodeset/, eval/, research/, and MEMORY.md are committed and shared; working-style/ and user/ stay on the machine that wrote them.

§7Machine setup

On a fresh clone, the in-repo store exists but the symlink Claude Code reads from does not yet. One idempotent script wires it:

# once per machine, after cloning the repo
bash scripts/dev/setup_memory_symlink.sh

The script computes the encoded project path (every non-alphanumeric character → -, the rule Claude Code uses), creates ~/.claude/projects/<encoded>/ if missing, and links its memory to <repo>/.claude/memory. It is safe to re-run: if the link is already correct it exits cleanly, and if a real directory already sits at the target path (an older, machine-local memory store) it refuses to overwrite and asks you to merge or move it aside — memory is never silently clobbered. The script only wires the link; syncing the files between machines is git's job.
