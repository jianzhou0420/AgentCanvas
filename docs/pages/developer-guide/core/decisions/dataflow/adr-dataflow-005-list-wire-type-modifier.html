ADR-dataflow-005

`LIST[T]` wire-type modifier with consumer-side coercion

Date
2026-04-15 18:00
Status
accepted
Field
dataflow
Old ID
ADR-027

Context

VLN methods like MapGPT, MC-GPT, DiscussNav, and VLN-Zero need to attach multiple items to a single LLM call — MapGPT in particular references up to 20 distinct viewpoint images positionally ("Image 3") in one prompt. The existing LLMCallNode.rgb port (agentcanvas/backend/app/agent_loop/builtin_nodes.py:78) carried exactly one IMAGE (np.ndarray (H,W,3)), and no list-valued image primitive existed anywhere in the backend. Prior workarounds (variadic fan-in into a single scalar port, state-container abuse, composite-grid tiling, rgb duck-typing to also accept lists) were all rejected — variadic fan-in violates the static-topology invariant, state-container abuse muddles the wire/memory boundary (ADR-dataflow-004), composite grids resize-destroy the multi-image semantics, and rgb duck-typing makes the port type a lie. A one-off IMAGE_LIST wire type was also rejected as the first step in a *_LIST proliferation chain (DiscussNav needs TEXT_LIST, MC-GPT topo memory needs POSE_LIST, search operators fanning candidate graphs need LIST[GraphDefinition]). Two generalisation shapes were considered and rejected during the design discussion: (a) auto-batching every array wire — promote every IMAGE/DEPTH to a leading-N batch dimension with single values stored as N=1, rejected because only 2 of the 10 wire types are ndarray-shaped (TEXT, POSE, BOOL, METRICS are not tensors), so the "uniform" model is leaky, and because MapGPT's 20 viewpoints are heterogeneous in H/W so a 4D tensor would force a lossy resize; (b) force every wire to be a list by default — make single values length-1 lists everywhere, rejected because this optimises the 1% multi-item case at the cost of the 99% single-item case, adds [0]/[x] noise to every existing producer/consumer/viewer/reducer, and pushes the system toward a Spark/Beam/dataflow-batch paradigm that contradicts the project's workflow mental model (feedback_workflow_vs_agentloop.md — wires are semantic items moving one-at-a-time through nodes). The need for LIST[T] is real, but the uniform/mandatory shapes were wrong.

Decision

LIST[T] is introduced as a wire-type modifier, not a new wire type, with consumer-side coercion. Five concrete rules, all landing in one change: (1) Modifier form — the wire type string LIST[T] for any existing inner type T ∈ WIRE_TYPES (IMAGE, DEPTH, TEXT, POSE, ACTION, BOOL, METRICS, OBSERVATION, STEP_RESULT, ANY). Helpers is_list_type, unwrap_list, wrap_list, is_valid_wire_type, is_valid_list_of in standard/wire_types.py. Nested LIST[LIST[T]] is rejected by is_valid_wire_type. (2) Producers stay scalar — existing nodes that fire once and produce one T keep their port type T. Zero churn across the 99% of nodes. A producer only declares LIST[T] when it semantically produces multiple items per fire (a future MapViewCollectorNode that renders 20 viewpoints in one forward() call). (3) Consumer-side coercion at one seam — the graph executor's port-binding loop (graph_executor.py:311–324) routes edge values through a new _route_value_to_port(target, tgt_handle, value) helper. When the consumer port's declared type is LIST[T]: a single T is wrapped as [value]; an incoming list is passthrough; multiple producers fanning into the same LIST[T] port concatenate in edge declaration order; list producers fanning in are flattened then concatenated. When the consumer port is scalar, write-through as before. A new helper _get_input_port_wire_type honours _resolve_ports (ADR-dataflow-003) so configurable-port nodes (LLMCallNode, IterIn/IterOut) pick up their instance-level overrides. (4) Edge validation — graph_def.validate_edge_wire_type(source_type, target_type) implements the compatibility table: equal → allow; ANY on either side → allow; T → LIST[T] → allow; LIST[T] → T → reject (lossy); different inner types → reject. Frontend onConnect in useFlowStore.ts:491 looks up both ends via node.data._schema and calls the same rule via isCompatibleWireConnection in layoutUtils.ts — invalid drags snap back without calling addEdge. Server-side validation is the helper; call-sites can wire it in as needed. (5) LLMCallNode.rgb migrates to LIST[IMAGE] — input_ports declares PortDef("rgb", "LIST[IMAGE]", ...), the port_list ConfigField default uses "LIST[IMAGE]", and the forward() image-collection loop iterates inputs["rgb"] unconditionally. Existing single-image graphs (Straightforward, NavGPT-CE presets) keep working unchanged — the env's scalar IMAGE producer is auto-wrapped to [ndarray] by the new seam, and vlm_complete() (llm/call.py:194) already takes list[str] and emits one image_url content block per item, so the LiteLLM dispatch is unchanged. The log panel shows LIST[T] payloads as "[N INNER(s)]" (e.g. "[20 IMAGE(s)]") via a new __type: "list" marker in log_serialize_with_assets — no per-tile thumbnails in v1. PortListEditor.tsx renders the inner type in the existing dropdown plus a new list checkbox that toggles LIST[...] wrap/unwrap on the canonical wire_type string. Handles on list-typed ports render with a doubled outer ring to distinguish them visually.

Alternatives

(a) Add IMAGE_LIST only — rejected for *_LIST proliferation: DiscussNav (TEXT_LIST), MC-GPT (POSE_LIST), search operators (LIST[GraphDefinition]), F6 (TEXT_LIST) would each add their own one-off type. One generic modifier is cheaper than five. (b) Auto-batch every ndarray wire with a leading N dimension — rejected: only IMAGE/DEPTH are ndarray-shaped (TEXT, POSE, BOOL, METRICS, OBSERVATION are not tensors), so the rule is leaky; MapGPT's 20 viewpoints are heterogeneous in H/W and a 4D tensor forces a lossy resize; and the viewer/reducer blast radius is enormous. (c) Force every wire to be a list by default — rejected: optimises the 1% multi-item case at the cost of the 99% single-item case; every existing producer wraps, every consumer unwraps, every viewer indexes [0], every reducer chooses between list and unwrapped — death by a thousand [0]s. Contradicts the project's workflow-paradigm stance (feedback_workflow_vs_agentloop.md). (d) Producer-side wrapping as the default coercion point — rejected: would force every existing scalar producer to know about downstream list ports, reintroducing the single-item tax that (c) has. Keeping coercion strictly at the consumer-binding seam means producers never need to change. (e) Support nested LIST[LIST[T]] in v1 — deferred: no current use case needs it; if grouped-views VLN methods surface later, revisit. The parser in is_valid_wire_type explicitly rejects nesting so the feature doesn't sneak in accidentally.

Rationale

The LIST[T] modifier keeps two good properties of the existing wire system — static topology, strongly typed, one-value-per-fire mental model — and only relaxes the arity at ports that explicitly opt in. The coercion rule collapses the six cases in the dispatch table (scalar↔scalar, scalar↔list, list↔list, fan-in in three flavours) into ~15 LOC at one file location, which is the right shape for a feature that should be invisible to most nodes. Producers stay scalar because that's how the system already works everywhere else, and because forcing producers into [x] would shift the cost from the 1% consumers to the 99% producers for no gain (the coercion is a single wrap regardless of which end does it). Consumer-side coercion also means that a LIST[T] port can later accept multiple data paths — the current scalar→list wrap and a future plugin-provided batching producer — without the consumer caring which. Fan-in determinism is fixed by edge declaration order in the graph JSON (already stable in the executor's adjacency iteration), which gives MapGPT-style positional prompt references a stable contract across runs. The MapGPT unblock is almost trivial at the call site — LLMCallNode.forward() iterates inputs["rgb"] unconditionally, vlm_complete() is already list-shaped, LiteLLM sends one image_url per item — so the total blast radius of the change is the wire type helpers (~30 LOC in wire_types.py), the executor seam + port-schema lookup (~70 LOC in graph_executor.py), the graph validation helper (~40 LOC in graph_def.py), the logger's list-payload marker (~10 LOC in logger.py), the LLMCallNode port switch (~15 LOC in builtin_nodes.py), and the frontend checkbox + compat check + color helper + handle decoration + log formatter (~70 LOC across five tsx files) — all landed together as one conceptually self-contained feature. Graphs never migrate: the on-disk shape stays a single wire_type string, and old graphs with single-image producers auto-coerce on load. The feature generalises cleanly — LIST[TEXT] for DiscussNav debate transcripts, LIST[POSE] for MC-GPT topo memory, LIST[ACTION] for trajectory-replay nodes, and population fan-in into one LIST[T] join for F3/F9/F10 — so the one-time cost amortises across the entire VLN method roadmap and the agent-architecture-search path.
