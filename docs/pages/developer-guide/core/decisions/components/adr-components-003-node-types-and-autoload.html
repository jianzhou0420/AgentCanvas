ADR-components-003

Standardized node types, auto-load nodesets, presets as data

Date
2026-03-29
Status
accepted
Field
components
Old ID
ADR-012

Context

After ADR-components-002 moved all environment interaction to nodeset tools, the framework had no hardcoded env or policy node types. But three problems remained: (1) Saved graphs referenced nodeset node types that might not be loaded, causing blank nodes with no ports/edges. (2) Node type naming was inconsistent (habitat_observe, tool_MeasureDistance, policy_forward_cma) — the frontend couldn't determine which nodeset a node type belonged to. (3) Graph presets were hardcoded in TypeScript (presets.ts), tightly coupling the framework to specific agent designs.

Decision

(1) Node type convention ${nodeset_name}__${node_name} with __ (double underscore) as separator. Examples: env_habitat__observe, env_habitat__step, policy_cma__forward. Built-in framework nodes (iterIn, llmCall, etc.) have no __ and are always available. (2) Auto-load nodesets on graph open: When the frontend loads a saved graph, it parses nodeset names from node types (split on __), fetches the nodeset list, and auto-loads any missing nodesets before rendering. After loading, it fetches updated node schemas and injects _schema into node data so GenericBlockRenderer can render ports and edges. (3) Presets as JSON data: Graph templates moved from presets.ts (TypeScript) to workspace/graphs/*.json (JSON files served via GET /api/graphs). No hardcoded graph templates in the framework. (4) Node IDs use ${type}_${hash8} format for readability. (5) NodeSet discovery probes tools: _scan_and_register_nodesets() calls get_tools() at scan time to populate _discovered_tool_names — the frontend can map node types to nodesets even before they're loaded. (6) Consolidated per-iteration WS broadcast: Output viewer sinks store their inputs, and a single nav_step WebSocket event is broadcast at the iterOut boundary (once per iteration) with merged data from all viewers.

Alternatives

(a) Keep presets in TypeScript — rejected, couples framework to specific domains. (b) Store port schemas in graph JSON — rejected, makes JSON larger and duplicates backend truth. (c) Require manual nodeset loading before opening graphs — rejected, poor UX. (d) Per-viewer WS broadcasts — rejected, caused conflicting partial data (e.g. observationViewer's broadcast had action: 0 default, overwriting actionLog's correct value).

Rationale

The __ convention makes nodeset detection parseable from the node type string alone — no backend round-trip needed to determine which nodeset provides which tool. Auto-loading bridges the gap between "nodesets are discover-only" and "graphs reference nodeset nodes." Presets as JSON follows the "data, not code" principle — researchers modify JSON files, not TypeScript. The consolidated broadcast ensures output viewers receive consistent, complete data per iteration.
