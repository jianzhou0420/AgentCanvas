Tutorial: Using the Execution Log System

The execution log system (ADR-observability-003) records per-node I/O for every firing during graph execution. It captures what went in, what came out, and how long it took — automatically, for every node. Nodes can also emit voluntary logs with domain-specific details (LLM prompts, token counts, parsed actions) that the executor cannot see from the outside.

This tutorial walks through adding logging to a custom nodeset, running a graph, and inspecting the results via REST API, JSONL files, and the frontend Log Viewer.

Reference: For the full system specification, see Execution Logs.

1. Prerequisites

  • The agentcanvas conda environment activated
  • The AgentCanvas backend running (cd agentcanvas && bash run_dev.sh)
  • Familiarity with BaseCanvasNode and NodeSets

2. How It Works

The log system has two layers:

┌──────────────────────────────────────────────────────┐
│  Node forward()                                      │
│                                                      │
│    self._self_log("prompt", prompt)     ◄── Interior │
│    self._self_log("tokens", 142)            (opt-in) │
│    return {"action": 1}                              │
│                                                      │
├──────────────────────────────────────────────────────┤
│  GraphExecutor (wraps every firing)               │
│                                                      │
│    inputs  = {port: value}              ◄── Exterior │
│    outputs = {port: value}                  (auto)   │
│    duration_ms = 25.4                                │
│    port_wire_types = {port: "TEXT"}                   │
│                                                      │
│    → log_node() → ring buffer + JSONL                │
│    → exec_log WS event → frontend                    │
└──────────────────────────────────────────────────────┘

Exterior layer — automatic, zero effort. The executor records port data, timing, and wire types for every firing.

Interior layer — opt-in. Nodes call self._self_log(key, value) inside forward() to record domain-specific data. These entries appear in the inner_log array of the log entry.

3. Building a Logged NodeSet

This tutorial ships with a ready-to-run example:

  • NodeSet: workspace/nodesets/example.py — five nodes with different logging patterns
  • Graph: workspace/graphs/text_analysis_demo.json — a DAG wiring five custom nodes + two in-graph viewers

No external dependencies beyond numpy. The nodeset is auto-discovered by WorkspaceComponentRegistry at startup.

3.1 The NodeSet File

Open workspace/nodesets/example.py to see the full implementation. Here are the key highlights:

TextSource — Entry node (no inputs). Outputs configurable text from a textarea config field:

class TextSource(BaseCanvasNode):
    node_type = "example__source"
    input_ports: ClassVar[list] = []  # no inputs → fires first as seed
    output_ports: ClassVar[list] = [PortDef("text", "TEXT", "The configured text")]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        text = self.config.get("text", "default text...")
        self._self_log("text_length", len(text))
        self._self_log("preview", text[:80])
        return {"text": text}

WordCounter — Logs simple scalars:

async def forward(self, inputs: dict, ctx: Any) -> dict:
    text = inputs.get("text", "")
    words = text.split()
    unique = set(w.lower() for w in words)

    self._self_log("word_count", len(words))          # scalar int
    self._self_log("unique_words", len(unique))        # scalar int
    self._self_log("avg_word_length", round(           # scalar float
        sum(len(w) for w in words) / max(len(words), 1), 1,
    ))
    return {"count": str(len(words)), "text": text}

SentimentTag — Logs a structured dict:

async def forward(self, inputs: dict, ctx: Any) -> dict:
    # ... keyword matching ...
    self._self_log("analysis", {                       # structured dict
        "positive_matches": sorted(pos_hits),
        "negative_matches": sorted(neg_hits),
        "positive_score": len(pos_hits),
        "negative_score": len(neg_hits),
        "result": sentiment,
    })
    return {"sentiment": sentiment, "text": text}

TextSummary — Logs multiple entries to trace the pipeline:

async def forward(self, inputs: dict, ctx: Any) -> dict:
    # ...
    self._self_log("input_lengths", {"text": len(text), ...})  # entry 1
    self._self_log("format_mode", fmt)                          # entry 2
    # ... build summary ...
    self._self_log("output_length", len(summary))               # entry 3
    return {"summary": summary}

RandomTriangleImage — Generates an IMAGE and logs shape/color metadata:

class RandomTriangleImage(BaseCanvasNode):
    node_type = "example__random_triangle"
    output_ports: ClassVar[list] = [PortDef("rgb", "IMAGE", "Generated RGB image")]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        w = int(self.config.get("width", 320))
        h = int(self.config.get("height", 240))
        img = np.full((h, w, 3), 255, dtype=np.uint8)

        color = [random.randint(30, 230) for _ in range(3)]
        pts = [(random.randint(0, w-1), random.randint(0, h-1)) for _ in range(3)]
        _fill_triangle(img, pts, color)

        self._self_log("image_shape", list(img.shape))    # asset logging trigger
        self._self_log("triangle_color", color)
        self._self_log("vertices", pts)
        return {"rgb": img}  # IMAGE wire type → saved as sidecar JPEG

Because the output port has wire type IMAGE, the executor's log_serialize_with_assets() automatically saves the numpy array as a JPEG file in outputs/runs/{id}/assets/ — no extra code needed.

3.2 In-Graph Viewers

The graph also includes two built-in viewer nodes — sink nodes that display data directly on the canvas during execution:

Viewer Node Type Input Port What It Shows
Observation Viewer imageViewer rgb (IMAGE) Renders the generated triangle image inline on the canvas
Scroll Text textScroll text (TEXT) Accumulates summaries across iterations as a scrollable stack on the canvas

Viewer nodes are built into the framework (graph_executor.py). They have no output ports — they consume data and display it. Wire any IMAGE or TEXT port to them to get live visual feedback without leaving the canvas.

3.3 Key Patterns Demonstrated

Node Logging Pattern What It Shows
TextSource _self_log("key", scalar) Entry node with config-driven output
WordCounter _self_log("key", scalar) Simple key-value pairs (numbers, strings)
SentimentTag _self_log("key", dict) Structured analysis data
TextSummary Multiple _self_log() calls Step-by-step pipeline tracing
RandomTriangleImage _self_log("key", list) IMAGE wire type with sidecar asset logging

All five nodes also get automatic exterior logging for free — the executor captures their inputs, outputs, duration_ms, and port_wire_types without any code in the nodes. The two viewer nodes also appear in the log — they log received_ports.

5. Querying Logs via REST API

Two routers cover all programmatic log access:

  • /api/logs/... — generic, execution-id-keyed. 6 endpoints: list, entries, node-history, summary, graph snapshot, asset (§ 5.1-5.6). Works for canvas runs and eval episodes alike (the latter use composite execution_id = "{run_id}_ep{NNNN}").
  • /api/replay/... — eval-only, walks the ADR-eval-004 per-episode directory layout directly. 3 endpoints (§ 5.7).

6. Reading JSONL Files Directly

Logs persist as JSONL files on disk. Each line is one NodeLogEntry:

# Find the log file
ls outputs/runs/

# Read it (one JSON object per line)
cat outputs/runs/${EXEC_ID}/log.jsonl | python -m json.tool --json-lines

# Count firings
wc -l outputs/runs/${EXEC_ID}/log.jsonl

# Extract just the inner_log entries using jq
cat outputs/runs/${EXEC_ID}/log.jsonl | jq '.inner_log[]'

6.1 Storage Layout

outputs/
├── runs/                                    # Canvas executions
│   └── exec_1775571161675_abc123/
│       ├── graph.json                       # Graph snapshot (for canvas replay)
│       ├── log.jsonl                        # One NodeLogEntry per line
│       └── assets/                          # Sidecar images (if any)
│           ├── s0_f0_Word__Counter__rgb.jpg
│           └── ...
└── eval_runs/                               # Eval executions (ADR-eval-004)
    └── {run_id}/                             # run_id = "20260515_143052" (timestamp)
        ├── spec.json  shared_urls.json
        ├── summary.json  graph.json
        ├── stdout.log  stderr.log  _DONE
        └── episodes/
            └── ep{idx:04d}/                  # one self-contained dir per episode
                ├── log.jsonl                 # this episode's firings; execution_id = "{run_id}_ep{idx:04d}"
                ├── assets/                   # this episode's images
                └── episode.json              # self-describing record

6.2 Programmatic Access

"""Read and analyze log files programmatically."""

import json
from pathlib import Path


def read_log(execution_id: str, source: str = "runs") -> list[dict]:
    """Read all log entries for a given execution.

    Canvas: ``execution_id`` is the standalone id, file at
    ``outputs/runs/{execution_id}/log.jsonl``.

    Eval: ``execution_id`` is the composite ``{run_id}_ep{idx:04d}``;
    pass ``source="eval_runs"`` and the function resolves it to
    ``outputs/eval_runs/{run_id}/episodes/ep{idx:04d}/log.jsonl``.
    """
    if source == "eval_runs" and "_ep" in execution_id:
        run_id, _, ep = execution_id.rpartition("_ep")
        path = Path("outputs") / "eval_runs" / run_id / "episodes" / f"ep{ep}" / "log.jsonl"
    else:
        path = Path("outputs") / source / execution_id / "log.jsonl"
    entries = []
    with open(path) as f:
        for line in f:
            if line.strip():
                entries.append(json.loads(line))
    return entries


def get_inner_logs(entries: list[dict], node_type: str) -> list[dict]:
    """Extract voluntary inner_log entries for a specific node type."""
    result = []
    for entry in entries:
        if entry["node_type"] == node_type:
            result.extend(entry.get("inner_log", []))
    return result


# Usage
entries = read_log("exec_1775571161675_abc123")

print(f"Total firings: {len(entries)}")
print(f"Node types: {set(e['node_type'] for e in entries)}")

# Get sentiment analysis details
sentiment_logs = get_inner_logs(entries, "example__sentiment")
for log in sentiment_logs:
    print(f"  {log['key']}: {log['value']}")

8. Standalone Script: Logger Without the Server

You can use ExecutionLogger directly in a standalone Python script — no backend server needed. This is useful for testing, scripting, or building custom analysis tools.

Save this as scripts/log_demo.py and run it:

#!/usr/bin/env python
"""Standalone demo of the ExecutionLogger — no server needed.

Run from the repo root:
    python scripts/log_demo.py
"""

from __future__ import annotations

import json
import os
import sys
import time

# Add backend to path so we can import the logging module
backend_dir = os.path.join(os.path.dirname(__file__), "..", "agentcanvas", "backend")
sys.path.insert(0, os.path.abspath(backend_dir))

from app.logging.logger import ExecutionLogger, log_serialize


def main():
    # ── 1. Create a logger with a persist directory ──
    execution_id = f"demo_{int(time.time())}"
    persist_dir = os.path.join("outputs", "runs", execution_id)

    logger = ExecutionLogger(
        execution_id=execution_id,
        source="canvas",
        persist_dir=persist_dir,
    )
    print(f"Logger created: {execution_id}")
    print(f"Persist dir:    {persist_dir}")

    # ── 2. Simulate node firings ──

    # Firing 1: Word Counter
    t0 = time.monotonic()
    time.sleep(0.001)  # simulate work
    elapsed = (time.monotonic() - t0) * 1000

    logger.log_node(
        step=0,
        node_id="counter",
        node_type="example__word_count",
        node_label="Word Counter",
        duration_ms=elapsed,
        inputs={"text": "The weather is great and wonderful"},
        outputs={"count": "6", "text": "The weather is great and wonderful"},
        inner_log=[
            {"key": "word_count", "value": 6},
            {"key": "unique_words", "value": 6},
        ],
        port_wire_types={"text": "TEXT", "count": "TEXT"},
    )
    print("  Logged: Word Counter (step 0)")

    # Firing 2: Sentiment Tagger
    logger.log_node(
        step=0,
        node_id="sentiment",
        node_type="example__sentiment",
        node_label="Sentiment Tagger",
        duration_ms=0.3,
        inputs={"text": "The weather is great and wonderful"},
        outputs={"sentiment": "positive", "text": "The weather is great and wonderful"},
        inner_log=[
            {"key": "analysis", "value": {
                "positive_matches": ["great", "wonderful"],
                "negative_matches": [],
                "result": "positive",
            }},
        ],
        port_wire_types={"text": "TEXT", "sentiment": "TEXT"},
    )
    print("  Logged: Sentiment Tagger (step 0)")

    # Firing 3: Text Summary
    logger.log_node(
        step=0,
        node_id="summary",
        node_type="example__summary",
        node_label="Text Summary",
        duration_ms=0.1,
        inputs={"text": "The weather is great and wonderful", "word_count": "6", "sentiment": "positive"},
        outputs={"summary": "6 words, positive sentiment"},
        inner_log=[
            {"key": "format_mode", "value": "brief"},
            {"key": "output_length", "value": 27},
        ],
        port_wire_types={"summary": "TEXT"},
    )
    print("  Logged: Text Summary (step 0)")

    # ── 3. Flush to disk ──
    logger.flush()
    print(f"\nFlushed to: {persist_dir}/log.jsonl")

    # ── 4. Read back and display ──
    print("\n── Log Entries ──\n")
    jsonl_path = os.path.join(persist_dir, "log.jsonl")
    with open(jsonl_path) as f:
        for i, line in enumerate(f):
            entry = json.loads(line)
            print(f"[{i}] {entry['node_label']} (step {entry['step']}, {entry['duration_ms']:.1f}ms)")
            print(f"    inputs:  {list(entry['inputs'].keys())}")
            print(f"    outputs: {list(entry['outputs'].keys())}")
            if entry["inner_log"]:
                print(f"    inner_log:")
                for item in entry["inner_log"]:
                    print(f"      {item['key']}: {item['value']}")
            print()

    # ── 5. Query the ring buffer ──
    print("── Ring Buffer Query ──\n")
    summary = logger.get_summary()
    print(f"Total firings: {summary.total_firings}")
    print(f"Node types:    {summary.node_types_fired}")
    print(f"Errors:        {summary.error_count}")

    # Filter by node type
    sentiment_entries = logger.get_entries(node_type="example__sentiment")
    print(f"\nSentiment node entries: {len(sentiment_entries)}")
    for e in sentiment_entries:
        print(f"  inner_log: {e.inner_log}")

    # ── 6. Demonstrate log_serialize ──
    print("\n── log_serialize() Examples ──\n")

    import numpy as np

    samples = {
        "short_string": "hello world",
        "long_string": "x" * 2000,
        "numpy_array": np.zeros((480, 640, 3), dtype=np.uint8),
        "nested_dict": {"a": {"b": {"c": [1, 2, 3]}}},
        "bytes_data": b"\x00" * 1024,
    }

    for name, value in samples.items():
        serialized = log_serialize(value)
        print(f"  {name:15s} → {json.dumps(serialized, default=str)[:80]}")


if __name__ == "__main__":
    main()

9. Advanced Patterns

9.1 Logging Images as Sidecar Assets

When a node outputs IMAGE or DEPTH wire types, the logger saves them as sidecar files instead of embedding them in JSONL. This happens automatically via log_serialize_with_assets():

class MyVisionNode(BaseCanvasNode):
    node_type = "my_nodeset__vision"
    output_ports = [PortDef("rgb", "IMAGE", "RGB observation")]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        import numpy as np
        rgb = np.zeros((480, 640, 3), dtype=np.uint8)  # your image here

        # No special logging needed — the executor handles IMAGE wire type
        # The image is saved as: assets/s{step}_f{idx}_{label}__rgb.jpg
        return {"rgb": rgb}

The JSONL entry references the asset file:

{
  "outputs": {
    "rgb": {
      "__type": "asset",
      "path": "assets/s0_f0_My__Vision__rgb.jpg",
      "wire_type": "IMAGE",
      "shape": [480, 640, 3],
      "dtype": "uint8"
    }
  }
}

Serve the image via the REST API:

curl -o image.jpg "localhost:8000/api/logs/${EXEC_ID}/assets/s0_f0_My__Vision__rgb.jpg"

9.2 Overriding the log() Method

The default log() returns the raw _self_log() buffer. Override it to add computed summaries or filter sensitive data:

class SecureNode(BaseCanvasNode):
    node_type = "my_nodeset__secure"

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        api_key = self.config.get("api_key", "")
        self._self_log("api_key_present", bool(api_key))
        self._self_log("api_key_value", api_key)  # sensitive!
        # ... do work ...
        return {"result": "ok"}

    def log(self) -> list[dict]:
        """Filter out sensitive values before they hit the log file."""
        return [
            entry for entry in self._log_buffer
            if entry["key"] != "api_key_value"
        ]

9.3 Logging in Dataflow (Cyclic) Graphs

In dataflow graphs with IterIn/IterOut, the same node fires multiple times across iterations. The step field tracks which iteration each firing belongs to:

class ProgressTracker(BaseCanvasNode):
    node_type = "my_nodeset__progress"

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        step = getattr(ctx, "step", 0)

        # Log per-iteration progress
        self._self_log("iteration", step)
        self._self_log("progress", f"Step {step} of estimated 10")

        return {"status": f"iteration_{step}"}

Query a specific iteration:

# Get logs for step 3 only
curl -s "localhost:8000/api/logs/${EXEC_ID}?step=3" | python -m json.tool

9.4 Flush Timing

The write queue is flushed at three points — not after every single firing:

When Why
IterOut boundary Batch-write at end of each iteration
GraphComplete Final flush on success
GraphError Flush even on failure (preserves debug data)

This batching reduces disk I/O while ensuring no data loss. In-memory queries via the ring buffer are always up-to-date.
