Tutorial: Creating the Habitat NodeSet

This tutorial walks through wrapping the Habitat-Sim environment as a NodeSet — a loadable group of canvas-node tools. We show two approaches:

  • Way 1: Local — everything runs in the same process and conda env
  • Way 2: Auto-Hosted Server — same code, deployed in a separate process automatically (ADR-server-001)

Both produce identical canvas nodes. The nodeset code is the same — the only difference is how you load it.

Note: For non-Python servers (C++, Rust) or custom protocols, the manual ServerNodeSet + ServerApp pattern still exists. This tutorial covers the common case: Python nodesets.

1. Prerequisites

  • Habitat-Sim installed in the ac-vlnce conda env (Python 3.8; the habitat-sim wheel is binary-locked to that version).
  • The production nodeset already exists at workspace/nodesets/env/env_habitat.py — it bundles HabitatEnvManager (the engine), the canvas tool classes, and EnvHabitatNodeSet in one file. This tutorial walks a simplified pedagogical version; the real file is ~1700 lines with extras covered in § 5.3.
  • Understanding of Wire Types and NodeSets (especially § 5 Env Nodeset Contract).

2. What We're Building

Five canvas nodes for the Habitat environment:

Node Ports In Ports Out Purpose
observe (none) rgb (IMAGE), depth (DEPTH) Get current view
step action (ACTION) rgb, depth, state, done, metrics Execute action
get_state (none) state (STATE) Query position/heading
episode_info (none) instruction (TEXT), episode_id (TEXT) Get episode metadata
panorama (none) composite (IMAGE), scene (TEXT) 360-degree view

After either approach, these nodes appear in the canvas sidebar and can be wired into agent graphs.

3. The NodeSet (one file, both ways)

Write one BaseNodeSet with 5 tool classes. This file is used by both Way 1 and Way 2 — no changes needed between them.

This listing is the simplified pedagogical version. The production workspace/nodesets/env/env_habitat.py is ~1700 lines and follows a 3-layer split: (1) HabitatEnvManager singleton (raw sim + GL-thread executor) · (2) thin BaseCanvasNode tool adapters (observe, step, localize, episode_info, navigate, panorama, frontiers, query_map) · (3) EnvHabitatNodeSet (lifecycle + ClassVars + env panel registration). Read the file alongside this tutorial; the extras the tutorial doesn't show are in § 5.3. 
workspace/nodesets/env/env_habitat.py

"""Habitat environment as a NodeSet (``EnvHabitatNodeSet``, name="env_habitat").

Works both as a local in-process nodeset and as an auto-hosted server.
Load locally:  POST /api/components/nodesets/env_habitat/load
Load as server: POST /api/components/nodesets/env_habitat/load?mode=server
"""

from __future__ import annotations

import asyncio
import logging
import os
from typing import Any

import numpy as np

from app.components import BaseCanvasNode, BaseNodeSet, PortDef

log = logging.getLogger(__name__)


# ── Helper ──

def _get_env_mgr():
    """Singleton accessor. The real ``HabitatEnvManager`` is defined in this
    same file (above) — wrapping the raw Habitat-Sim API with a
    threading.Lock and a single-thread ThreadPoolExecutor that pins all
    simulator work to one OS thread (GL / physics affinity).
    """
    return HabitatEnvManager.get()


async def _run_blocking(fn, *args):
    """Run a blocking function in the env manager's executor."""
    mgr = _get_env_mgr()
    loop = asyncio.get_running_loop()
    return await loop.run_in_executor(mgr._executor, fn, *args)


def _extract_rgb_depth(raw):
    """Extract rgb and depth ndarrays from raw Habitat observations."""
    rgb = depth = None
    if raw:
        for k in ("rgb", "RGB"):
            if k in raw:
                rgb = np.asarray(raw[k], dtype=np.uint8)
                break
        for k in ("depth", "DEPTH"):
            if k in raw:
                depth = np.asarray(raw[k], dtype=np.float32).squeeze()
                break
    return rgb, depth


# ── Tool Nodes ──


class HabitatObserve(BaseCanvasNode):
    node_type = "habitat_observe"
    display_name = "Habitat: Observe"
    description = "Get current RGB + depth without stepping"
    category = "environment"
    icon = "Eye"
    input_ports = []
    output_ports = [
        PortDef("rgb", "IMAGE", "Current RGB observation"),
        PortDef("depth", "DEPTH", "Current depth map"),
    ]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        mgr = _get_env_mgr()
        raw = await _run_blocking(mgr.get_raw_obs)
        rgb, depth = _extract_rgb_depth(raw)
        return {"rgb": rgb, "depth": depth}


class HabitatStep(BaseCanvasNode):
    node_type = "habitat_step"
    display_name = "Habitat: Step"
    description = "Execute a navigation action"
    category = "environment"
    icon = "Play"
    input_ports = [
        PortDef("action", "ACTION", "Action to execute (0-3)"),
    ]
    output_ports = [
        PortDef("rgb", "IMAGE", "Post-step RGB"),
        PortDef("depth", "DEPTH", "Post-step depth"),
        PortDef("state", "STATE", "Post-step agent state"),
        PortDef("action", "ACTION", "Echo of executed action"),
        PortDef("done", "BOOL", "Whether episode ended"),
        PortDef("metrics", "METRICS", "Final metrics (when done)"),
    ]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        mgr = _get_env_mgr()
        action = int(inputs.get("action", 1))
        result = await _run_blocking(mgr.step, action)
        raw = await _run_blocking(mgr.get_raw_obs)
        rgb, depth = _extract_rgb_depth(raw)
        return {
            "rgb": rgb,
            "depth": depth,
            "state": {
                "position": result.get("position", [0, 0, 0]),
                "orientation": result.get("orientation", [0, 0, 0, 1]),
            },
            "action": action,
            "done": result.get("done", False),
            "metrics": result.get("metrics"),
        }


class HabitatState(BaseCanvasNode):
    node_type = "habitat_state"
    display_name = "Habitat: State"
    description = "Get agent position and orientation"
    category = "environment"
    icon = "MapPin"
    input_ports = []
    output_ports = [
        PortDef("state", "STATE", "Agent position + heading"),
    ]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        mgr = _get_env_mgr()
        state = await _run_blocking(mgr.get_state)
        return {"state": state}


class HabitatEpisodeInfo(BaseCanvasNode):
    node_type = "habitat_episode_info"
    display_name = "Habitat: Episode Info"
    description = "Get current episode instruction and ID"
    category = "environment"
    icon = "Info"
    input_ports = []
    output_ports = [
        PortDef("instruction", "TEXT", "Navigation instruction"),
        PortDef("episode_id", "TEXT", "Episode ID"),
    ]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        mgr = _get_env_mgr()
        info = await _run_blocking(mgr.get_episode_info)
        return {
            "instruction": info.get("instruction", ""),
            "episode_id": str(info.get("episode_id", "")),
        }


class HabitatPanorama(BaseCanvasNode):
    node_type = "habitat_panorama"
    display_name = "Habitat: Panorama"
    description = "Render 360-degree panoramic observation"
    category = "environment"
    icon = "Scan"
    input_ports = []
    output_ports = [
        PortDef("composite", "IMAGE", "Composite panorama image"),
        PortDef("scene", "TEXT", "View direction descriptions"),
    ]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        mgr = _get_env_mgr()
        n_views = self.config.get("n_views", 4)
        pano = await _run_blocking(mgr.render_panorama, n_views)
        composite = None
        b64 = pano.get("composite_base64", "")
        if b64:
            import base64, io
            from PIL import Image
            buf = io.BytesIO(base64.b64decode(b64))
            composite = np.array(Image.open(buf).convert("RGB"), dtype=np.uint8)
        scene = "\n".join(
            "%s: heading %s deg" % (v["direction"], v["heading_deg"])
            for v in pano.get("views", [])
        )
        return {"composite": composite, "scene": scene}


# ── NodeSet ──


class EnvHabitatNodeSet(BaseNodeSet):
    name = "env_habitat"
    description = "Habitat-Sim VLN-CE environment"

    # Habitat-Sim is binary-locked to Python 3.8, so this defaults the
    # auto-hosted server to the `vlnce` conda env.
    server_python = os.environ.get("VLNCE_PYTHON", "python")

    async def initialize(self, **kwargs):
        """Initialize the Habitat simulator."""
        mgr = _get_env_mgr()
        if not mgr.initialized:
            loop = asyncio.get_running_loop()
            await loop.run_in_executor(mgr._executor, mgr.initialize,
                "vlnce_baselines/config/r2r_baselines/cma_pm_da.yaml",
                "val_unseen", 0, 500)
        log.info("EnvHabitatNodeSet initialized")

    async def shutdown(self):
        """Shutdown the Habitat simulator."""
        mgr = _get_env_mgr()
        if mgr.initialized:
            loop = asyncio.get_running_loop()
            await loop.run_in_executor(mgr._executor, mgr.shutdown)
        log.info("EnvHabitatNodeSet shut down")

    def get_tools(self):
        return [
            HabitatObserve(),
            HabitatStep(),
            HabitatState(),
            HabitatEpisodeInfo(),
            HabitatPanorama(),
        ]

4. Way 1: Local

Load the nodeset in-process. Simpler, lower latency, but the AgentCanvas backend must have Habitat-Sim installed.

# Via API
curl -X POST "localhost:8000/api/components/nodesets/env_habitat/load"

Or it loads automatically at startup when WorkspaceComponentRegistry scans workspace/nodesets/.

WorkspaceComponentRegistry.scan_all()
  → finds EnvHabitatNodeSet
  → calls initialize() → loads Habitat simulator
  → calls get_tools() → 5 canvas nodes (Localize, Observe, Step, EpisodeInfo, StepNavigate…)
  → registers each via register_node()
  → nodes appear in sidebar under "Environment"
Pro Con
Simple — one file, one process AgentCanvas needs habitat-sim installed
Low latency — no HTTP Shares GPU with the web server
Easy debugging Can't use a different Python version

5. Way 2: Auto-Hosted Server (ADR-server-001)

Load the same file in server mode. The framework auto-generates the server — zero extra code.

# Via API — just add ?mode=server
curl -X POST "localhost:8000/api/components/nodesets/env_habitat/load?mode=server"

5.1 What happens

  1. Registry finds EnvHabitatNodeSet class via inspect.getfile()
  2. Picks a free port (e.g. 9200)
  3. Launches: python -m app.server.auto_host --file workspace/nodesets/env/env_habitat.py --class EnvHabitatNodeSet --port 9200 using server_python (defaults to $VLNCE_PYTHON or the ac-vlnce conda env)
  4. AutoServerApp introspects get_tools() → auto-generates ServerFunction for each tool from input_ports/output_ports
  5. Fetches /manifest, generates proxy nodes, registers them — identical to any server-mode nodeset

5.2 Using a different Python interpreter

If the server subprocess needs a different conda env (e.g. Habitat requires Python 3.8 with specific CUDA):

class EnvHabitatNodeSet(BaseNodeSet):
    name = "env_habitat"
    server_python = "python"
    ...

The framework uses server_python instead of sys.executable when spawning the subprocess.

Pro Con
Zero extra code — same file as Way 1 HTTP latency (~1-5ms per call)
Process isolation (GPU, memory) Slightly more complex debugging
Different conda env via server_python Subprocess management overhead
Switch between local/server at load time

5.3 Env-Specific ClassVars and the Env panel

An env nodeset (anything that holds simulator state, like Habitat) carries four extras that the simplified § 3 listing skips. The production EnvHabitatNodeSet declares all of them:

class EnvHabitatNodeSet(BaseNodeSet):
    name = "env_habitat"
    server_python = os.environ.get("VLNCE_PYTHON", "python")

    # ── ADR-server-003: parallelism contract ──
    parallelism = "replicated"            # stateful sim: one instance per eval worker

    # ── per-step budget for batch eval (overrides BaseNodeSet default 30 s) ──
    default_per_step_budget_sec = 30.0    # raise for slow scenes, lower for fast ones

    # ── ADR-server-002: BaseEnvPanel for per-episode reset over RPC ──
    env_panel = HabitatEnvPanel        # exposes set_episode / list_episodes / get_split / …

    # ── Feed the Eval page's split + episode picker ──
    async def get_eval_metadata(self) -> dict:
        return {
            "env_name": "vln_ce_r2r",
            "splits": ["train", "val_seen", "val_unseen", "test"],
            "episode_counts": {"train": 10819, "val_seen": 778, "val_unseen": 1839, "test": 3604},
            "metrics": ["spl", "success", "ndtw", "sdtw"],
            "step_budget": 500,
        }
ConceptWhat it gives youReference
parallelism "replicated" = one nodeset instance per eval worker (correct for stateful sims — each worker owns its own scene + agent). "shared" = one instance multiplexed across workers (correct for pure tools, e.g. a stateless segmenter). Habitat must be "replicated". ADR-server-003 · decision
default_per_step_budget_sec Episode timeout floor for batch eval is max_steps × per_step_budget_sec. Framework default is 30 s (LLM-heavy); 30.0 here gives Habitat per-scene GC + occasional VRAM thrashing room. Per-eval overrides win. Batch Eval tutorial § 3
env_panel RPC surface for non-canvas ops that the eval system needs: set_episode(idx), list_episodes(split), get_split(). Works identically in local mode (direct call) and server mode (auto-generated RemoteEnvPanelProxy). One canonical per-episode reset path; no canvas node needed. ADR-server-002 · Design — Nodeset Env Panels
get_eval_metadata() Drives the Eval page's split picker, episode-index slider, and per-graph metric column choices. Returned dict is opaque to the framework — your eval graph reads it via the env panel. design-docs/nodesets § 5

The minimal example in § 3 omits these because they're orthogonal to the "one file, two deployments" point — but a real env nodeset wired into batch eval needs all four.

6. Side-by-Side Comparison

Aspect Way 1: Local Way 2: Auto-Hosted Server
Files 1 (server/habitat.py) 1 (same file)
Base class BaseNodeSet BaseNodeSet (unchanged)
Load command POST .../load POST .../load?mode=server
forward() Direct Python call POST to localhost:{port}/call/{fn}
Wire types Native numpy arrays base64 PNG over HTTP (auto-serialized)
Dependencies Main process needs habitat-sim Only subprocess needs habitat-sim
Different Python No Yes (via server_python)
GPU isolation No Yes
Per-worker isolation (batch eval) Same instance shared across workers parallelism = "replicated" spawns N independent subprocesses (one scene + agent per worker)

7. Which to Choose?

Is your backend running in the same conda env as the simulator?
  └── Yes  ─ AND ─ no GPU/memory isolation needed?
  │             └── Yes → Way 1 (Local — simpler, faster)
  │             └── No  → Way 2 (Auto-Hosted)
  └── No   → Way 2 (Auto-Hosted — set server_python to the right env)

Habitat in particular: habitat-sim is binary-locked to Python 3.8 (ac-vlnce env), while the main AgentCanvas backend runs on Python 3.10 (agentcanvas env). So in practice Way 2 is the canonical and only working path for Habitat — Way 1 is documented for completeness and for nodesets that can share the backend's interpreter (e.g. SAM, VLM-Prismatic, basic-agent). The decision rule above resolves Habitat to Way 2 unconditionally.
