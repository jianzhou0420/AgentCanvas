Running an Architecture Search

Agent Architecture Search (AAS) on AgentCanvas runs through the /architect:<variant>:loop slash-skill family — seven variants sharing a common iter-tree + evaluator contract. Each variant is a methodology (ADAS-style sub-agent Reflexion, AFlow-style MCTS, knowledge-distillation, etc.) and exposes the same five-skill surface: understand → proposer → implementer → evaluator → loop. This tutorial shows how to run a search, then how to fork a variant when you need a new search algorithm.

Variant family vs the planned Python plugin API. Today's AAS surface is the variant family — Claude-conversation drivers, fast iteration on prompts / lineage / shape. A second surface — a Python BaseSearchOperator plugin API — is designed and waiting to be built (roadmap F8–F12). The two are intended to coexist; see Design — Search Operator API (RFC) for the planned plugin contract and how variants would map onto it. This tutorial covers today's working path.

1. Prerequisites

  • Backend up at :8000 with /experiment:run reachable (see Driving the Backend § Mode A).
  • A seed graph at workspace/graphs/{name}.json plus a per-graph performance profile workspace/architect/exp_profiles/{name}.yaml (the variant's evaluator auto-bootstraps conservative defaults if it's missing).
  • An experiment profile in .claude/experiment/profiles.yaml matching your workload's VRAM / GPU declaration.
  • Read the chosen variant's README.md before invoking — they have different contracts and iter-dir conventions (see the Required reading stanza at the top of each loop.md).

2. The Variant Family

VariantOne-lineMethodology heritage
adas-subagent One iter = one Claude conversation; 3 Reflexion calls become sub-agent spawns for sampling diversity ADAS (Hu et al. 2024) reframed for the coding-agent era
aflow MCTS over workflows; tree policy + backprop on operator instance AFlow (Zhang et al. 2024)
myloop Knowledge-distillation orchestrator — hard THINK → EXPERIMENT → DISTILL triple per iter; no archive, 7-file working memory per schemas.md Distillation-oriented variant

All three live under .claude/commands/architect/<variant>/. Pick the one whose contract matches your search shape:

An archive-based AAS loop with LLM proposing graph patches against top-K parents adas-subagent
MCTS / tree-search over workflows aflow

3. Anatomy of a Variant

Every variant has the same five-skill surface plus a few accessories. The standard set:

.claude/commands/architect/<variant>/
├── README.md              # core contracts + framing note + library/data
├── loop.md                # orchestrator: owns the iter conversation,
│                          #   drives proposer → implementer → evaluator
├── understand.md          # one-shot pre-search briefing for the orchestrator
├── proposer.md            # generate the next patch (mutation / proposal)
├── implementer.md         # apply patch + smoke-test it (Tier-1 eval)
├── evaluator.md           # delegate to _common/evaluator.md for Tier-2 perf eval
├── config.yaml            # variant config: perf_profile_key, evaluator settings,
│                          #   variant notes, side-effect artifacts
└── lib/                   # variant-specific Python helpers (atomic commit,
                           #   lineage rendering, archive maintenance, etc.)

What's shared across all variants lives in _common/:

.claude/commands/architect/_common/
├── files-contract.md      # universal run-dir layout, edit whitelist,
│                          #   resolve protocol, per-graph exp.yaml schema
├── evaluator.md           # Tier-2 perf eval + bootstrap_CI (verbatim from
│                          #   ADAS upstream utils.py:31-76)
└── implementer.md         # generic implementer routine (variants stub their
                           #   evaluator.md to call _common's)

The two-tier evaluation:

  • Tier 1 — Smoke (5 eps, in implementer.md) — gates entry to performance eval. If smoke crashes, the iter aborts before paying the Tier 2 cost.
  • Tier 2 — Performance (100 eps + bootstrap CI, in _common/evaluator.md) — runs once after implementer succeeds, writes staged metrics with confidence intervals.

This split makes variants cheap to fork (you swap proposer.md and maybe implementer.md, the evaluator is shared) and audits straightforward (the bootstrap CI algorithm is byte-identical across all variants).

4. Your First Search Run

Pick a variant + a graph + a max-iter budget. The variant owns the per-iter Claude conversation; you watch.

# ADAS-style 5-iter search on a NavGPT-MP3D seed
/architect:adas-subagent:loop navgpt_mp3d --max-iters 5

# myloop with an explicit goal
/architect:myloop:loop mapgpt_mp3d --goal "improve SR by 5pp" --max-iters 3

# Use a custom workspace overlay (e.g. an in-flight iter's active_workspace)
/architect:adas-subagent:loop navgpt_mp3d --max-iters 3 \
    --workspace=/path/to/vlnworkspace/outputs/design_runs/adas-subagent/navgpt_mp3d/v0/active_workspace

The loop:

  1. Bootstrap — resolves (graph, vN, iter_M); if iter_0 doesn't exist, runs baseline perf eval directly (no proposer, no implementer) so the search has a reference point.
  2. Per iter: proposer reads top-K parents from archive (variant- dependent) and emits a patch under iter_M/diff/ → implementer applies it to a fresh active_workspace/ overlay and runs 5-ep smoke → if smoke passes, evaluator runs 100-ep perf eval via /experiment:run + bootstrap CI → metrics get committed atomically to iter_M/.
  3. Atomic commit — failed iters do not partial-pollute the iter-tree; the variant's lib/helpers.py:atomic_commit ensures either everything lands or nothing does.
  4. Termination — --max-iters or variant-specific termination_check (e.g. myloop's goal.md § Termination).
The eval step itself goes through /experiment:run, which means it's subject to JobScheduler GPU admission + queueing. If a sibling session is holding GPU exclusively, your iter will wait at the evaluator step. See Running a Parallel Batch Eval for the eval-side knobs (worker count, per-step budget, episode count) — those are all set via the variant's config.yaml § evaluator + the graph's .exp.yaml.

5. Authoring a New Variant

If none of the seven fits, fork the closest existing one — that's how the family grew:

# Example: fork adas-subagent as a starting point for a new ADAS-shape variant.
cp -r .claude/commands/architect/adas-subagent .claude/commands/architect/my-variant

Then walk through the five skill files and edit what your methodology needs:

FileWhen to edit
README.md Always — restate your variant's contracts + framing note. Variants that diverge from _common/files-contract.md must document the divergence here.
loop.md When the per-iter conversation shape changes (e.g. you spawn sub-agents differently, gate on different signals, store different working memory).
proposer.md Almost always — this is where your search algorithm lives (mutation strategy, parent selection, prompt template).
implementer.md Rarely — usually stubs to _common/implementer.md. Edit only if your variant has a non-standard smoke contract.
evaluator.md Rarely — the stub binds VARIANT + VARIANT_DIR and points at _common/evaluator.md. Edit only if you need a different bootstrap algorithm (you usually don't — verbatim ADAS algorithm is the agreed common ground).
config.yaml Always — set perf_profile_key, evaluator settings, side-effect artifact paths.
lib/ If you need custom atomic-commit, lineage rendering, or archive bookkeeping beyond what the variant you forked already provides.

The hard contract you must preserve: the iter-dir layout (outputs/design_runs/<variant>/<graph>/vN/iteration/iter_M/) and the files inside it (smoke / perf metrics, snapshot, lineage edges). Everything else is your call. The contract is enforced by _common/files-contract.md — read it before deviating.

Smoke-test your new variant on a tiny budget first (--max-iters 1 --episodes 5) before turning it loose on a real graph.

6. Output Layout

outputs/design_runs/<variant>/<graph>/
└── vN/                                  # version line — bump for major method pivots
    ├── config.md                        # vN-level: methodology, baseline assumptions
    ├── trace.md                         # one row per iter — verdict + delta
    ├── lineage.md                       # parent → child edges across iters
    ├── archive.jsonl                    # variant-specific: e.g. adas-subagent uses it for top-K sampling
    ├── active_workspace/                # current in-flight overlay (Mode A's --workspace target)
    └── iteration/
        ├── iter_0/                      # baseline — no patch, perf eval only
        │   ├── perf/                    # 100-ep eval (bootstrap CI), from evaluator
        │   │   ├── metrics.json         # mean + 95% CI
        │   │   └── run_id.txt           # pointer into outputs/eval_runs/{id}/
        │   └── snapshot/                # frozen state of graphs + nodesets at this iter
        ├── iter_1/
        │   ├── diff/                    # patch produced by proposer
        │   ├── proposal.md              # proposer's hypothesis (free-text)
        │   ├── smoke/                   # 5-ep gating eval, from implementer
        │   │   └── result.json
        │   ├── perf/                    # 100-ep perf eval (only if smoke passed)
        │   ├── snapshot/                # iter's frozen state
        │   └── verdict.md               # outcome + lineage edge
        └── iter_2/ …

Everything here is conformant with .claude/commands/architect/_common/files-contract.md. Variants extend this set (e.g. myloop writes a 7-file working memory per schemas.md) but never break the core layout.

7. Planned: Python BaseSearchOperator Plugin API

The variant family above is one of two planned AAS surfaces. The other is a Python plugin API (BaseSearchOperator + SearchLoop + Graph Mutation API), still on the roadmap as F8–F12. It is not built yet. Read Design — Search Operator API (RFC) for the planned contract, the algorithm- compatibility analysis (ADAS / AFlow / EvoAgentX / Voyager-stub all fit the 3-method interface with one selection_mode amendment), and the intended coexistence with this variant family.

Why both? The variant family is the fast, agile path — iterate on prompts, lineage shape, orchestration without touching Python. The plugin API is the packaged, citeable path — once a variant stabilises, porting it to a BaseSearchOperator subclass gives you a distribution-ready artifact with a tighter version contract and language-level type safety. Same iter-tree, same eval harness, same fitness signals — only the driver changes.
