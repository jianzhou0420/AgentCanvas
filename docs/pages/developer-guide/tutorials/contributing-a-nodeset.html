Contributing a NodeSet

The fastest way to extend AgentCanvas is to write a nodeset — a group of canvas nodes with a shared lifecycle — or compose a graph out of existing nodesets, and contribute it by PR into workspace/. This page walks a nodeset end-to-end: what can be one, the file template, the rules, local testing, how to open the PR, and how your work gets credited.

For core framework work instead (fix a bug, extend an executor, improve the canvas UI), see CONTRIBUTING.md — open a Discussion first for anything big.

1. How contributing a nodeset works

You add your files under workspace/, AgentCanvas's WorkspaceComponentRegistry auto-discovers them at startup, and you open a PR. Three things make this low-friction:

1.1 Light review

Content contributions get a light review: your nodeset loads, follows the component conventions, and stays inside workspace/ with no edits to the framework. There's no architectural gate — the heavy review is reserved for core changes that touch the plugin contract every nodeset author depends on.

1.2 You keep the credit

Every nodeset and graph is attributed to its author/maintainer on the Credits board — maintained in the README and a dedicated doc-site page — so contributing into the shared repo, rather than a separate one, doesn't cost you authorship. If your contribution is part of a paper, its citation link goes on the Credits board too.

1.3 It versions with the repo

Because your nodeset lives in the repo, it moves with the framework — when the platform refactors, your code is migrated alongside everything else, and breaking changes to the public API (BaseCanvasNode, BaseNodeSet, PortDef, wire types) only land through an ADR. You don't have to track a moving target from the outside.

2. What you can contribute

Any of the three extension types that WorkspaceComponentRegistry auto-discovers from workspace/:

Type What it is Lives in
NodeSet A BaseNodeSet subclass with its canvas nodes — a loadable group of tools with shared lifecycle (e.g. Habitat env, SAM segmentation, a custom memory system) workspace/nodesets/*.py
Graph A GraphDefinition JSON saved as an editable template (kind="graph") or frozen composite (kind="node") workspace/graphs/*.json or workspace/graph_nodes/*.json
Search operator A BaseSearchOperator subclass implementing an Agent Architecture Search algorithm (e.g. evolutionary, MCTS, RL-based) workspace/search_operators/*.py — see Writing a Search Operator

This tutorial focuses on nodesets — the most-used extension point with the most-stable API. A tool nodeset wraps a capability into nodes (a reconstructor, a SLAM system, a detector, a simulator); a method nodeset encodes an agent's reasoning (a NavGPT or MapGPT loop). For search operators, read Writing a Search Operator — that surface ships but its public API is still experimental.

3. Minimum viable nodeset

A minimum viable nodeset is a single Python file under workspace/nodesets/<role>/ (env / method / model / policy / common — see ADR-platform-008). That's the entire mechanism:

workspace/nodesets/common/my_nodeset.py    # the actual nodeset code (one file)

Drop the file in, restart the backend (or POST /api/components/reload), and WorkspaceComponentRegistry auto-discovers the BaseNodeSet subclass on startup — the nodes appear in the sidebar. No package registry, no install script, no build step. Research code is read, modified, and cited: a file next to the user's own graphs in a familiar working tree lowers the friction of "I want to tweak one line for my experiment" to near zero.

4. Writing a nodeset

This section is the packaging surface. For the authoring API itself — every ClassVar, every UI knob, every lifecycle hook — see the worked-example tutorial Component Cookbook and the spec-level docs Base Canvas Node and NodeSets.

4.1 File template

Every nodeset starts from this skeleton (matches workspace/nodesets/common/example.py):

# my_nodeset.py
from __future__ import annotations

from typing import Any, ClassVar

from app.components import (
    BaseCanvasNode,
    BaseNodeSet,
    ConfigField,
    NodeUIConfig,
    PortDef,
)


class MyTool(BaseCanvasNode):
    """A single canvas node — appears in the sidebar, wires to other nodes."""

    node_type = "my_nodeset__my_tool"         # convention: nodeset__node
    display_name = "My Tool"
    description = "Does one useful thing"
    category = "example"                       # sidebar grouping
    icon = "Wand2"                             # lucide-react icon name

    input_ports: ClassVar[list] = [PortDef("text", "TEXT", "Input text")]
    output_ports: ClassVar[list] = [PortDef("result", "TEXT", "Output text")]

    ui_config: ClassVar[NodeUIConfig] = NodeUIConfig(
        color="sky",
        config_fields=[
            ConfigField("suffix", "text", label="Suffix", default="!"),
        ],
    )

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        text = inputs.get("text", "")
        suffix = self.config.get("suffix", "!")
        self._self_log("input_length", len(text))
        return {"result": text.upper() + suffix}


class MyNodeSet(BaseNodeSet):
    """The NodeSet class — groups tools with a shared lifecycle."""

    name = "my_nodeset"
    description = "My custom tools for AgentCanvas"

    def get_tools(self) -> list:
        return [MyTool()]

4.2 The 5 universal rules

  1. from __future__ import annotations at the top of every file — required for Python 3.8 compatibility in the ac-vlnce env and for forward references.
  2. One NodeSet per file: all the node classes live in the same file as the BaseNodeSet subclass, declared above it.
  3. node_type naming convention: "<nodeset_name>__<node_name>" (double underscore separator). Pick a nodeset name that's unlikely to collide with others.
  4. ClassVar[...] annotations: input_ports, output_ports, and ui_config must be declared with ClassVar to prevent Python from treating them as instance attributes.
  5. Wire types are fixed: use one of IMAGE, DEPTH, ACTION, STATE, TEXT, BOOL, METRICS, OBSERVATION, STEP_RESULT, ANY. No custom wire types in a nodeset (register via a core PR + ADR if you genuinely need a new one).

4.3 What you can and cannot import

Can import (stable public API):

  • from app.components import BaseCanvasNode, BaseNodeSet, PortDef, NodeUIConfig, ConfigField, DisplayField
  • from app.standard.wire_types import IMAGE, DEPTH, ACTION, ... (or the string literals directly)
  • Any Python standard library, any PyPI package (declare your extra dependencies in the nodeset's README or its env)
  • Habitat, habitat-sim, torch, transformers, numpy, PIL — these are framework-external and nodeset code may use them freely

Cannot import (unstable internal APIs, subject to breakage):

  • Anything under app.agent_loop.* — the executor internals may refactor
  • app.graph_def internals — use the types only through BaseCanvasNode.forward() signature
  • Private modules (anything with _ prefix)

Framework import boundary: the core framework (agentcanvas/backend/app/) is forbidden from importing domain libraries like habitat. Your nodeset, living in workspace/nodesets/, is on the correct side of that boundary — feel free to import whatever simulators or ML libraries you need.

4.4 Testing locally

Before opening the PR, verify the nodeset loads and runs:

# Your file lives at workspace/nodesets/<role>/my_nodeset.py — start the backend
cd /path/to/agentcanvas-checkout/
bash agentcanvas/run_dev.sh

# Check that your NodeSet appears
curl http://localhost:8000/api/components/nodesets | jq '.[] | .name'

If your NodeSet shows up in the list, it's loaded. If not, check the backend log — import errors are printed at startup. Then open http://localhost:5173, drag your node from the sidebar into a fresh graph, wire it to an input and an output, and press Play. If it produces the expected output, you're ready to open the PR.

5. Opening the PR

Open a PR against master adding your files under workspace/. Keep it to one concern, and make sure the local checks pass (pre-commit run --all-files, backend pytest including the import-boundary test). See CONTRIBUTING § PR format for the description template and the full check set.

5.1 Document it

Make the nodeset self-explanatory: a clear class description, per-node display_name / description, and — if it has setup steps or extra dependencies — a short README.md next to the file. State up front which AgentCanvas behaviours, datasets, or model checkpoints it expects.

5.2 License and citation

Contributions are merged under the repo's Apache-2.0 license. If your nodeset reproduces a published method, add its citation in the file's docstring and in your PR — it gets carried onto the Credits board so anyone using your nodeset can cite the source.

5.3 Compatibility

Until AgentCanvas reaches v1.0 the public API is still firming up, but because your nodeset lives in the repo it's migrated alongside framework refactors. Changes that affect the contract every nodeset depends on — BaseCanvasNode, BaseNodeSet, PortDef, NodeUIConfig, ConfigField, DisplayField, wire types — only land through an ADR. Skim the last few ADRs in the Decisions index after a big release.

6. Reference nodesets

Use the bundled nodesets in workspace/nodesets/ as your reference — copy the one that most closely matches your use case and adapt it:
