Driving the Backend from a Coding Agent

You have an AgentCanvas backend, you have a coding agent (Claude Code, Cursor, etc.), and you want the agent to run experiments: kick off batch evals, watch progress, pull metrics, iterate on the graph. There are three programmatic surfaces today; this tutorial walks each one end-to-end so you can pick.

Just want to click "Start" yourself? This tutorial is for programmatic / agent-driven use. For UI-driven evals (and the worker-pool / batched-inference knobs that apply to all three modes below), see Running a Parallel Batch Eval. For the design of these surfaces — endpoint contracts, MCP slot pool, PDEATHSIG crash cleanup — see Design — Coding-Agent Backend Surface.
ModeSurfaceLifecycleBest when
A. /experiment:run Slash-skill submits one graph to the JobScheduler over HTTP Backend is yours on :8000 and stays up; skill submits, polls, prints metrics on exit. Companion: /experiment:status, /experiment:teardown One-off graph eval, hyperparameter sweep, ad-hoc smoke test against your already-running backend with GPU admission control
B. MCP server (agentcanvas-backend) Typed MCP tools — eval_start, eval_status, eval_export, eval_stop, graph_list, eval_runs_list One backend per coding-agent conversation, isolated via a flock- based 5-slot pool on 8765-8769; PDEATHSIG-armed against crashes; backend self-exits when the conversation ends You want conversational backend driving without owning a :8000 backend yourself — debugging, ad-hoc queries, "run X, now compare with Y"
C. /architect:<variant>:loop (AAS family) Per-variant loop skill drives proposer → implementer → evaluator; evaluator internally calls /experiment:run Variant owns an iter-tree under outputs/design_runs/<variant>/<task>/... with snapshots, bootstrap CI, archive.jsonl. Eval-step lifecycle delegated to A You're inside an architecture-search loop and want every cycle to land in a versioned iter dir with reproducible metrics

All three surfaces drive the same backend code (FastAPI app under agentcanvas/backend/app/). They differ in how the agent addresses it and how much policy / bookkeeping the skill layers on top.

1. Pick a Mode

You're about to…Use
Run a graph once to confirm a bug fix worked /experiment:run (Mode A)
Pull a finished run's metrics for analysis MCP eval_export (Mode B) or read outputs/eval_runs/{run_id}/summary.json
Conversationally explore: "run X, what's SR, now try Y" MCP (Mode B)
Compare two graph variants on a pinned episode subset MCP eval_start with episode_indices=[...], called twice
Start a new architecture-search loop (will produce 5+ iters) /architect:<variant>:loop (Mode C)
Detach from a running submission and poll later /experiment:status <run_id> --watch
Cancel everything in the scheduler before restarting /experiment:teardown

2. Mode A — /experiment:run

Foreground submitter to the backend's JobScheduler (M1 design, 2026-05-07). Multiple jobs queue and run concurrently when their VRAM declarations fit; Ctrl-C posts a cancel and exits. Replaces the legacy /architect:experiment-http.

2.1 Prerequisites

  • Backend up at :8000 (yours — start it once with the usual cd agentcanvas/backend && uvicorn app.main:app --host 127.0.0.1 --port 8000 or its auto_host wrapper). The skill never spawns a backend.
  • A profile in .claude/experiment/profiles.yaml matching your workload's VRAM + GPU declarations. Unknown names fall back to defaults with a stderr warning.
  • A graph at workspace/graphs/{name}.json. Optional companion {name}.exp.yaml for per-graph eval defaults.

2.2 Run

/experiment:run mapgpt-mp3d mapgpt_mp3d episode_count=10 worker_count=2

Arguments:

  • mapgpt-mp3d — profile key in profiles.yaml (GPU admission rule).
  • mapgpt_mp3d — graph name under workspace/graphs/ (no extension).
  • key=value overrides — common keys: episode_count, worker_count, step_budget, per_step_budget_sec, dataset, split, start_episode_index.

Optional --workspace=<abs-path> overlay — let the backend pick up an architect iter's active_workspace/ without dirtying frozen workspace/:

/experiment:run mapgpt-mp3d mapgpt_mp3d \
    --workspace=/path/to/vlnworkspace/outputs/design_runs/myloop/mapgpt_mp3d/v0/iter_3/active_workspace \
    episode_count=10

2.3 What Happens

  1. Read profiles.yaml → build payload (eval block + scheduling block with marginal_vram_mb, exclusive_gpu).
  2. POST /api/eval/v2/start with via_subprocess=true → returns {"run_id": "...", "initial_state": "pending"}.
  3. Poll GET /api/eval/v2/runs/{run_id} until the run reaches a terminal state (done, error, aborted, cancelled).
  4. On non-success, dump the tail of stderr.log markers from the scheduler subprocess so the failure isn't silent (see memory feedback_experiment_run_silent_failures).
  5. Print final metrics and exit.

2.4 Detach and Poll Later

If you Ctrl-C the foreground submitter without intent to cancel, the run continues on the backend. Reattach with the printed run_id:

/experiment:status                       # queue + running snapshot
/experiment:status <run_id>              # one run, one shot
/experiment:status <run_id> --watch      # poll until terminal
/experiment:status <run_id> --cancel     # POST cancel and exit

And to drain the whole queue (without touching the backend process — which is yours):

/experiment:teardown

2.5 Output

Results land in the backend-owned per-run dir, identified by the returned run_id:

outputs/eval_runs/{run_id}/
  summary.json                     aggregate metrics (written at completion)
  episodes/ep{NNNN}/
    episode.json                   per-episode result (worker_id, status, metrics)
    log.jsonl                      this episode's firings
    assets/                        screenshots, intermediate dumps
  stdout.log / stderr.log          scheduler subprocess output

No iter-tree, no trace.md, no token-cost rollup — those belong to Mode C, which wraps Mode A.

3. Mode B — MCP Server (raw)

The MCP server (agentcanvas/mcp_server/) packages the eval API as Model Context Protocol tools — the coding agent talks to the backend through typed tools with JSON-Schema-validated inputs, not curl. The server itself encapsulates pool-slot acquisition on 127.0.0.1:8765-8769 plus PDEATHSIG-armed crash cleanup — the agent never sees a port number, a PGID, or a signal handler.

The full tool set (6 tools, all in agentcanvas/mcp_server/server.py):

ToolWhat it does
eval_start Start a batch eval. Accepts graph_name, episode_count, worker_count, step_budget, episode_indices, etc.
eval_status Live progress + currently-running episode per worker
eval_export Read outputs/eval_runs/{run_id}/summary.json after completion — backend-independent
eval_stop Cancel the live run on this conversation's backend
graph_list Enumerate graphs in workspace/graphs/ with their workspace/architect/exp_profiles/{graph}.yaml profile fields
eval_runs_list Newest-first listing of past runs from disk — works without a live backend

3.1 Prerequisites

  • Mode A prerequisites minus the running backend — the MCP server spawns its own.
  • mcp Python package in the agentcanvas conda env (already in agentcanvas/backend/requirements.txt).
  • Claude Code CLI on $PATH (claude command) for registration.

3.2 Register

Once per machine:

claude mcp add agentcanvas-backend \
  --scope local \
  -e PYTHONPATH=/path/to/vlnworkspace/agentcanvas \
  -- conda run -n agentcanvas --no-capture-output python -m mcp_server
FlagWhy
--scope local This user, this project only. Use --scope user to share across all your projects, --scope project to commit a .mcp.json for the team.
-e PYTHONPATH=… claude mcp add has no --cwd; PYTHONPATH is the cleanest way to let python -m mcp_server find the package.
--no-capture-output Critical — without it, conda buffers stdout and breaks MCP JSON-RPC framing.
conda run -n agentcanvas Pin the env so the mcp package resolves correctly.

Verify:

claude mcp list | grep agentcanvas-backend
# → agentcanvas-backend: conda run -n agentcanvas ...  ✓ Connected

Restart Claude Code after registration — MCP tool catalogs load once at session start, not hot-refreshed.

3.3 Use

In any Claude Code session, just ask:

List the available agent graphs.

Claude calls agentcanvas-backend / graph_list() and reports the graphs with their .exp.yaml profiles.

Run navgpt_mp3d for 3 episodes with 1 worker. Show me the success
rate when it finishes.

Claude orchestrates the whole flow without you scripting it:

  1. eval_start(graph_name="navgpt_mp3d", episode_count=3, worker_count=1) → {"run_id": "abcd1234", "status": "pending"}
  2. Repeated eval_status() until status == "completed"
  3. eval_export(run_id="abcd1234") → per-episode results
  4. Claude summarises aggregate_metrics.success back to you

3.4 Pool-Slot Lifecycle

Brief — full design in Design — Coding-Agent Backend Surface.

  • Start. On the first tool call, BackendManager walks ports 8765-8769, tries flock(LOCK_EX | LOCK_NB) on each ~/.cache/agentcanvas-mcp/locks/{port}.lock. First unlocked = our slot. subprocess.Popen uvicorn with os.setsid() + prctl(PR_SET_PDEATHSIG, SIGTERM). Polls /health for up to 90 s.
  • Concurrent conversations. Each Claude Code conv grabs a different pool slot — they coexist trivially (up to 5 sessions).
  • Clean exit (stdio close → SIGTERM): BackendManager.stop() walks uvicorn descendants for app.server.auto_host children, killpg(SIGTERM)s the backend group + each auto_host group, sleeps 2 s, SIGKILLs survivors.
  • Unclean exit (SIGKILL of MCP, OOM-kill, segfault): kernel fires the re-armed PDEATHSIG inside uvicorn (~106 ms after MCP dies), the cascade cleans up. No orphan processes, no held ports. Linux-only — see § 7.5.

4. Mode C — /architect:<variant>:loop (AAS Family)

An architecture-search variant owns a per-iter trajectory: each cycle proposes a graph patch, implements + smoke-tests it, runs a perf eval with bootstrap CI, and atomically commits the result to an iter dir. The variant family pivots methodology (search strategy, prompt shapes, lineage tracking), but all variants share the same eval-step contract via _common/evaluator.md — which internally calls /experiment:run. Mode C wraps Mode A in iter-tree bookkeeping, not the other way around.

Replaces the legacy /architect:experiment-mcp and /architect:loop.

4.1 Variants

VariantOne-lineREADME
adas-subagent ADAS one-iter-one-conversation; sub-agent per role .claude/commands/architect/adas-subagent/README.md
aflow AFlow-style search .claude/commands/architect/aflow/README.md
myloop Knowledge-distillation orchestrator (THINK → EXPERIMENT → DISTILL) .claude/commands/architect/myloop/README.md

4.2 Prerequisites

  • Mode A prerequisites (running backend on :8000 with /experiment:run reachable).
  • Variant's per-graph perf profile in workspace/architect/exp_profiles/{graph}.yaml (the evaluator auto-bootstraps conservative defaults if missing).
  • Read the variant's README before invoking — they have different contracts and iter-dir layouts (see Required reading stanzas at the top of each loop.md).

4.3 Run

/architect:adas-subagent:loop mapgpt_mp3d --max-iters 5
/architect:myloop:loop mapgpt_mp3d --goal "improve SR by 5pp" --max-iters 3

The loop owns the per-iter Claude conversation and drives the variant's worker skills (typically proposer → implementer → evaluator). The msg_list persists across all three. _common/evaluator.md is the only step that touches the backend — via /experiment:run.

4.4 Output

outputs/design_runs/<variant>/<graph>/vN/
  iter_0/                          baseline (no patch; perf eval only)
    metrics.json                   bootstrap mean + 95% CI
  iter_1/
    diff/                          patch produced by proposer
    active_workspace/              workspace with patch applied (Mode A's --workspace overlay target)
    smoke/                         5-ep smoke eval (gating; from implementer)
    perf/                          100-ep perf eval (bootstrap CI; from evaluator)
    metrics.json
    notes.md                       proposer's hypothesis + outcome
  iter_2/ …
  archive.jsonl                    variant-specific lineage (adas-subagent only)
  trace.md                         one-row-per-iter rollup
  lineage.md                       parent → child edges across iters

6. Combining Modes

  1. Exploratory phase. Use MCP. Try graph variants, run small episode counts, get a feel for what's interesting. No iter-tree pollution from throwaway runs.
  2. Smoke / sweep. Once a hypothesis stabilises, /experiment:run with the right profile to get GPU admission control + per-graph episode budgets without iter-tree overhead.
  3. Promotion. When you want every revision tracked across runs, /architect:<variant>:loop for the iter trajectory.
  4. Spot checks during the loop. Between iters, MCP for ad-hoc queries ("what was the SR of iter_3 again?" → Claude calls eval_export on the iter's run_id from metrics.json).

All three share the filesystem (outputs/eval_runs/{run_id}/ is visible to all surfaces). They differ in which backend they hit: /experiment:run and architect variants share your :8000 backend (serialised by ExecutionGuard within that backend); MCP uses its own pool slot. Cross-mode A↔C work fine — they're the same backend; B is parallel infrastructure.

7. Troubleshooting

7.1 /experiment:run fails with "no backend at :8000"

You haven't started a backend yet. The skill doesn't spawn one (that's the M1 contract — backends are user-owned). Start it:

cd agentcanvas/backend
conda run -n agentcanvas --no-capture-output uvicorn app.main:app --host 127.0.0.1 --port 8000

Or use tools/auto_host.py if you want the multi-env supervisor. Override the URL with AGENTCANVAS_BACKEND_URL=http://... if you're on a non-default port.

7.2 /experiment:run warns "profile not found, falling back to defaults"

The profile name didn't match anything in .claude/experiment/profiles.yaml. The skill uses the defaults block + full GPU + exclusive_gpu=true — safe but serialises every other run. Add a real profile entry before the next run; see .claude/experiment/README.md.

7.3 MCP tools don't appear in Claude Code

  1. claude mcp list should show agentcanvas-backend ✓ Connected. If × Failed, check ~/.cache/agentcanvas-mcp/backend-{ISO}.log for conda / uvicorn output.
  2. You restarted Claude Code after registration. Tool catalogs load at session start.
  3. If claude mcp list doesn't show it at all, registration didn't persist. Re-run claude mcp add from § 3.2 and check ~/.claude.json has an entry under projects.{your-repo-path}.mcpServers.

7.4 MCP eval_start returns {"error": "another run is already active"}

This conversation's backend enforces single-slot execution (one batch eval at a time per backend). Either wait for the current run to finish (poll eval_status) or call eval_stop. Other Claude Code conversations are unaffected — they're on different pool slots and don't share the single-slot guard.

7.5 MCP "all 5 pool slots are in use"

You already have 5 Claude Code conversations holding backends, or a previous backend failed to release its flock. Close one of the conversations; the kernel releases the slot's lock at process exit regardless of how it died. If no conversation is alive, find the orphaned uvicorn:

ss -tlnp 2>/dev/null | grep -E '127\.0\.0\.1:876[5-9]'
# pick the PID, get its PGID:
PGID=$(ps -o pgid= -p <PID> | tr -d ' ')
kill -TERM -- -"$PGID"

As of 2026-05-01 (kernel-level orphan cleanup), this should be rare — PDEATHSIG fires when the parent MCP dies.

7.6 SIGKILL of MCP orphans the backend on macOS / BSD

PDEATHSIG is Linux-only. AgentCanvas's primary platform is Linux (habitat-sim is Linux-first), so this is a known constraint, not a bug. On Linux the MCP → backend cascade dies in ~106 ms.

7.7 /architect:<variant>:loop fails on iter_0 baseline

The baseline call goes through _common/evaluator.md --mode baseline, which runs perf eval against the current workspace. If this fails, the variant skill aborts before any proposer runs. Three usual causes:

  • Backend not up — same as § 7.1.
  • Perf profile missing required fields — check the variant's config.yaml § evaluator.profile_key and confirm the matching key exists in workspace/architect/exp_profiles/{graph}.yaml.
  • Workspace overlay mismatch — for variants that build an active_workspace/, ensure the variant's iter init step ran (see the variant's loop.md § "Bootstrap").
