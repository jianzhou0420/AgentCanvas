  • A. Quick Reference

1. Your First Node

A canvas node is a Python class that inherits from BaseCanvasNode. It declares its identity, ports, and implements forward(). That's it — the framework handles rendering, wiring, and execution.

1.1 Minimal Example

Create a file at workspace/nodesets/my_first.py:

from __future__ import annotations

from typing import Any, ClassVar

from app.components import BaseCanvasNode, BaseNodeSet, PortDef


class GreetNode(BaseCanvasNode):
    """Generate a greeting message."""

    node_type = "tutorial__greet"
    display_name = "Greet"
    description = "Generate a greeting for a given name"
    category = "tutorial"
    icon = "Hand"

    input_ports: ClassVar[list] = [
        PortDef("name", "TEXT", "Person's name"),
    ]
    output_ports: ClassVar[list] = [
        PortDef("greeting", "TEXT", "The greeting message"),
    ]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        name = inputs.get("name", "World")
        return {"greeting": f"Hello, {name}!"}


class MyFirstNodeSet(BaseNodeSet):
    name = "tutorial"
    description = "Tutorial companion nodes"

    def get_tools(self) -> list:
        return [GreetNode()]

1.2 What Each Part Does

Attribute Purpose Convention
node_type Unique identifier in graph JSON {nodeset}__{function} (double underscore)
display_name Human label in sidebar catalog Short, descriptive
description Tooltip and search text One sentence
category Sidebar grouping "environment", "tool", "reasoning", "control", "output", or custom
icon Lucide icon name PascalCase (e.g. "ScanEye", "Globe")
input_ports What data the node receives List of PortDef
output_ports What data the node produces List of PortDef
forward() The node's logic Async, receives inputs dict, returns outputs dict

1.3 PortDef and Wire Types

Each port is declared with PortDef(name, wire_type, description, optional):

PortDef("image", "IMAGE", "RGB observation", optional=False)  # required by default
PortDef("hint", "TEXT", "Optional hint", optional=True)        # won't block execution

Available wire types:

Wire Type Python Type Use For
TEXT str Strings, JSON, prompts
IMAGE np.ndarray (H×W×3 uint8) RGB images
DEPTH np.ndarray (H×W float32) Depth maps
ACTION int Discrete actions (0=STOP, 1=FWD, 2=LEFT, 3=RIGHT)
STATE dict Agent state (position, orientation)
BOOL bool Flags, conditions
METRICS dict Evaluation metrics (SPL, SR, etc.)
OBSERVATION dict Full observation bundle
STEP_RESULT dict Step output (obs + reward + done + info)

1.4 Auto-Discovery

Drop your file in workspace/nodesets/ and the backend discovers it at startup. No registration code needed — WorkspaceComponentRegistry scans for all BaseNodeSet subclasses.

To load at runtime without restarting:

curl -X POST "http://localhost:8000/api/components/nodesets/tutorial/load"

After loading, your node appears in the sidebar catalog under its category.

1.5 Entry Nodes (No Inputs)

A node with an empty input_ports list is an entry node — it fires first without waiting for upstream data:

class TimestampNode(BaseCanvasNode):
    node_type = "tutorial__timestamp"
    display_name = "Current Time"
    description = "Output the current timestamp"
    category = "tutorial"
    icon = "Clock"

    input_ports: ClassVar[list] = []   # ← entry node
    output_ports: ClassVar[list] = [
        PortDef("time", "TEXT", "ISO timestamp"),
    ]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        from datetime import datetime, timezone
        return {"time": datetime.now(timezone.utc).isoformat()}

2. Configuring Nodes

Nodes often need user-editable settings — a temperature slider, a model selector, a text template. ConfigField renders inline controls directly on the node UI, no custom .tsx needed.

2.1 Adding Config Fields

from app.components import BaseCanvasNode, ConfigField, NodeUIConfig, PortDef

class TemperatureTextNode(BaseCanvasNode):
    node_type = "tutorial__temp_text"
    display_name = "Temperature Text"
    description = "Convert a temperature value to descriptive text"
    category = "tutorial"
    icon = "Thermometer"

    input_ports = [PortDef("value", "TEXT", "Temperature in Celsius")]
    output_ports = [PortDef("description", "TEXT", "Descriptive text")]

    ui_config = NodeUIConfig(
        color="orange",
        config_fields=[
            ConfigField("unit", "select", label="Unit",
                        options=[
                            {"value": "celsius", "label": "Celsius"},
                            {"value": "fahrenheit", "label": "Fahrenheit"},
                        ], default="celsius"),
            ConfigField("threshold", "slider", label="Hot threshold",
                        default=30, min=0, max=50, step=1),
            ConfigField("verbose", "toggle", label="Verbose output",
                        default=False),
        ],
    )

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        temp = float(inputs.get("value", "0"))
        unit = self.config.get("unit", "celsius")
        threshold = self.config.get("threshold", 30)
        verbose = self.config.get("verbose", False)

        if unit == "fahrenheit":
            temp = (temp - 32) * 5 / 9  # convert to Celsius for comparison

        if temp >= threshold:
            label = "hot"
        elif temp >= threshold - 15:
            label = "mild"
        else:
            label = "cold"

        if verbose:
            return {"description": f"{temp:.1f}°C is {label} (threshold: {threshold}°C)"}
        return {"description": label}

2.2 All Six Field Types

field_type Widget Key Parameters Example
"label" Read-only text default ConfigField("version", "label", label="Version", default="1.0")
"text" Single-line input placeholder, default ConfigField("api_key", "text", label="API Key", placeholder="sk-...")
"textarea" Multi-line input placeholder, default ConfigField("prompt", "textarea", label="Prompt", default="Describe...")
"slider" Range slider min, max, step, default ConfigField("temp", "slider", label="Temperature", default=0.7, min=0, max=2, step=0.1)
"select" Dropdown options, default ConfigField("model", "select", label="Model", options=[{"value":"gpt4","label":"GPT-4"}])
"toggle" Checkbox/switch default ConfigField("debug", "toggle", label="Debug mode", default=False)

2.3 Reading Config in forward()

Config values live in self.config — a plain dict populated from the graph JSON before execution:

async def forward(self, inputs: dict, ctx: Any) -> dict:
    # Always provide defaults — config may be empty for new nodes
    model = self.config.get("model", "gpt-4")
    temperature = float(self.config.get("temperature", 0.7))
    max_tokens = int(self.config.get("max_tokens", 256))
    ...

The user changes these values on the canvas node UI. The values persist in the graph JSON under node.config.

3. Node UI Customization

NodeUIConfig controls how a node looks on the canvas — color, size, layout — all from Python, without writing React code.

3.1 Color Palette

The color attribute accepts any Tailwind color key. If left empty, it auto-derives from category:

Common color choices: "sky", "blue", "violet", "purple", "rose", "orange", "amber", "emerald", "teal", "cyan", "slate", "zinc", "green".

Two layouts available:

Block (default) — standard rectangular node:

ui_config = NodeUIConfig(

    min_width="200px",        # optional width constraint
    max_width="320px",        # optional max width
)

ui_config = NodeUIConfig(

    layout="strip",
    width="44px",             # narrow width

)

4. Live Display Fields

While ConfigField is for user input, DisplayField is for runtime output — live data pushed to the node via WebSocket during execution.

4.1 How It Works

  1. Your node's forward() writes output ports
  2. Sink nodes (viewers) with DisplayField receive wired data
  3. The executor serializes the data and emits a viewer_data WebSocket event
  4. The frontend field renderer displays it on the node

4.2 Display Types

display_type Widget Use For
"image_viewer" <img> tag RGB images (base64)
"log_list" Scrollable log Text entries that accumulate over iterations
"metric_table" Key-value table Metrics like SPL, SR, nDTW
"text_viewer" Text block Single text value, replaced each iteration

4.3 Example: A Counter Display Node

class IterationCounterNode(BaseCanvasNode):
    """Display a live count of how many iterations have passed."""

    node_type = "tutorial__iter_counter"
    display_name = "Iteration Counter"
    description = "Show live iteration count"
    category = "output"
    icon = "Timer"

    input_ports = [
        PortDef("trigger", "TEXT", "Any input to trigger counting", optional=True),
    ]
    output_ports = [
        PortDef("count", "TEXT", "Current count as string"),
    ]

    ui_config = NodeUIConfig(
        color="rose",
        display_fields=[
            DisplayField("counter", "metric_table",
                         label="Iteration Stats",
                         data_key="metrics"),
        ],
    )

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        step = getattr(ctx, "step", 0)
        return {"count": str(step)}

4.4 Accumulating Log Fields

For fields that grow over time (like action history), set accumulate=True:

ui_config = NodeUIConfig(
    color="indigo",
    display_fields=[
        DisplayField("history", "log_list",
                     label="Action History",
                     data_key="action",
                     max_visible=20,      # show last 20 entries
                     accumulate=True),    # append, don't replace
    ],
)

5. Voluntary Logging

Every node can record internal details during execution via _self_log(). These appear in the OutputDrawer Logs tab and are persisted as JSONL in outputs/runs/.

5.1 Basic Usage

async def forward(self, inputs: dict, ctx: Any) -> dict:
    text = inputs.get("text", "")

    # Log simple scalars
    self._self_log("input_length", len(text))

    # Log structured data
    self._self_log("analysis", {
        "word_count": len(text.split()),
        "char_count": len(text),
    })

    # Log step-by-step processing
    self._self_log("step", "tokenizing input")
    tokens = text.split()
    self._self_log("step", f"found {len(tokens)} tokens")

    result = " ".join(tokens[:10])
    self._self_log("output_preview", result[:80])

    return {"result": result}

5.2 What Gets Logged Automatically

The executor captures this for every node firing without any code on your part:

  • Input ports: what data arrived (values truncated for large payloads)
  • Output ports: what was produced
  • Timing: execution duration in milliseconds
  • Iteration: which loop iteration this firing belongs to

Your _self_log() calls add voluntary interior logs on top of these automatic exterior logs.

5.3 Viewing Logs

  • Frontend: OutputDrawer → Logs tab (click a node to filter)
  • REST API: GET /api/logs/{execution_id} returns the JSONL (same endpoint serves both canvas and eval episodes)
  • Files (canvas): outputs/runs/{execution_id}/log.jsonl
  • Files (eval, per ADR-eval-004): outputs/eval_runs/{run_id}/episodes/ep{NNNN}/log.jsonl — one file per episode, no JSONL filter needed

For the full log system guide, see Using the Execution Log System.

6. Dynamic Ports

Most nodes have fixed ports declared as ClassVars. But some nodes need ports that vary per instance — for example, a merge node where the user adds input slots, or iteration gates with domain-specific ports.

6.1 The _resolve_ports() Classmethod

Override _resolve_ports(config) to return instance-specific ports:

class FlexibleMergeNode(BaseCanvasNode):
    """Merge multiple text inputs into one output.

    The user configures how many inputs via config.ports.
    """

    node_type = "tutorial__flex_merge"
    display_name = "Flexible Merge"
    description = "Merge N text inputs into a single output"
    category = "tool"
    icon = "Merge"

    # Class-level defaults (used when config has no ports override)
    input_ports = [
        PortDef("a", "TEXT", "First input"),
        PortDef("b", "TEXT", "Second input"),
    ]
    output_ports = [
        PortDef("merged", "TEXT", "Merged text"),
    ]

    ui_config = NodeUIConfig(
        color="sky",
        config_fields=[
            ConfigField("separator", "text", label="Separator", default="\n"),
            ConfigField("ports", "port_list", label="Input Ports"),
        ],
    )

    @classmethod
    def _resolve_ports(cls, config: dict) -> tuple[list, list]:
        """If config.ports exists, use those as input ports."""
        custom_ports = config.get("ports")
        if custom_ports:
            inputs = [
                PortDef(p["name"], p.get("wire_type", "TEXT"), p.get("name", ""))
                for p in custom_ports
            ]
            return (inputs, cls.output_ports)
        return (cls.input_ports, cls.output_ports)

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        sep = self.config.get("separator", "\n")
        parts = [str(v) for v in inputs.values() if v is not None]
        return {"merged": sep.join(parts)}

6.2 How Dynamic Ports Work

  1. Backend: When a graph loads, the executor calls _resolve_ports(node.config) for each node instance to determine its actual ports
  2. Frontend: resolveInstancePorts() reads config.ports from the graph JSON and renders the correct handles
  3. Graph JSON: The config.ports array is stored per node instance:
{
  "id": "merge_abc123",
  "type": "tutorial__flex_merge",
  "config": {
    "separator": " | ",
    "ports": [
      {"name": "title", "wire_type": "TEXT"},
      {"name": "body", "wire_type": "TEXT"},
      {"name": "footer", "wire_type": "TEXT"}
    ]
  }
}

6.3 When to Use Dynamic Ports

  • IterIn/IterOut: domain-specific data channels configured per graph
  • Merge/Split nodes: variable fan-in or fan-out
  • Adapter nodes: bridge between nodesets with different port layouts

For fixed-port nodes (the vast majority), skip _resolve_ports() entirely — the class-level input_ports / output_ports work out of the box.

7. Grouping into a NodeSet

A BaseNodeSet is a deployment unit — it groups related nodes that share lifecycle (initialize/shutdown) and load/unload atomically.

7.1 NodeSet Structure

"""Tutorial nodeset — companion to the Component Cookbook.

Load:  POST /api/components/nodesets/tutorial/load
"""

from __future__ import annotations

import logging
from typing import Any, ClassVar

from app.components import (
    BaseCanvasNode, BaseNodeSet, ConfigField, DisplayField,
    NodeUIConfig, PortDef,
)

log = logging.getLogger("agentcanvas.tutorial")


# ── Shared module-level state ──

_counter: int = 0


# ── Nodes ──

class GreetNode(BaseCanvasNode):
    node_type = "tutorial__greet"
    display_name = "Greet"
    description = "Generate a greeting"
    category = "tutorial"
    icon = "Hand"
    input_ports = [PortDef("name", "TEXT", "Name")]
    output_ports = [PortDef("greeting", "TEXT", "Greeting")]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        return {"greeting": f"Hello, {inputs.get('name', 'World')}!"}


class ShoutNode(BaseCanvasNode):
    node_type = "tutorial__shout"
    display_name = "Shout"
    description = "Convert text to uppercase"
    category = "tutorial"
    icon = "Volume2"
    input_ports = [PortDef("text", "TEXT", "Input text")]
    output_ports = [PortDef("shouted", "TEXT", "UPPERCASED text")]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        return {"shouted": str(inputs.get("text", "")).upper()}


# ── NodeSet Registration ──

class TutorialNodeSet(BaseNodeSet):
    name = "tutorial"
    description = "Tutorial companion nodes from the Component Cookbook"

    def get_tools(self) -> list:
        return [GreetNode(), ShoutNode()]

    async def initialize(self, **kwargs: Any) -> None:
        global _counter
        _counter = 0
        log.info("TutorialNodeSet initialized")

    async def shutdown(self) -> None:
        global _counter
        _counter = 0
        log.info("TutorialNodeSet shut down")

7.2 Key Design Decisions

One file = one nodeset. The auto-discovery system scans each .py file in workspace/nodesets/ for exactly one BaseNodeSet subclass. All nodes for that nodeset live in the same file.

Module-level state for shared data. If multiple nodes need shared mutable state (like the scratch pad in basic_agent.py), use a module-level dict or variable. Clear it in shutdown().

initialize() for heavyweight setup. Called once when the nodeset loads. Use it for:

  • Loading ML models onto GPU
  • Opening database connections
  • Allocating large buffers
  • Starting background processes
async def initialize(self, **kwargs: Any) -> None:
    self._model = load_model("path/to/weights.pth")
    log.info("Model loaded: %.1f MB", get_model_size(self._model))

shutdown() for cleanup. Called when the nodeset unloads or the app exits:

async def shutdown(self) -> None:
    del self._model
    torch.cuda.empty_cache()
    log.info("GPU memory released")

7.3 The Naming Convention

The node_type must follow the {nodeset}__{node} pattern (double underscore). This enables:

  • Auto-loading: when a graph opens, the frontend parses node types to identify required nodesets and loads them automatically
  • Catalog grouping: the sidebar groups nodes by nodeset prefix
  • Collision avoidance: two nodesets can both have a step node without conflict
env_habitat__observe      → nodeset "env_habitat", node "observe"
basic_agent__web_search   → nodeset "basic_agent", node "web_search"
tutorial__greet           → nodeset "tutorial", node "greet"

7.4 Loading and Unloading

From the UI: use the NodeSet Manager page (sidebar → NodeSets).

From the API:

# Load
curl -X POST "http://localhost:8000/api/components/nodesets/tutorial/load"

# Unload
curl -X POST "http://localhost:8000/api/components/nodesets/tutorial/unload"

# List loaded
curl "http://localhost:8000/api/components/nodesets"

When a nodeset loads, all its nodes appear in the sidebar catalog. When it unloads, they disappear. Graphs that reference unloaded nodes show placeholder nodes until the nodeset reloads.

8. Server Mode

By default, nodesets run in-process — same Python, same memory space as the backend. But some nodesets need a different Python environment (e.g. Habitat-Sim requires Python 3.8) or GPU isolation. Server mode runs the nodeset in a separate process, communicating over HTTP.

8.1 Enabling Server Mode

Add a server_python ClassVar pointing to the target interpreter:

import os

class HabitatNodeSet(BaseNodeSet):
    name = "env_habitat"
    description = "Habitat VLN-CE environment"

    # This triggers server mode — the nodeset runs under this Python
    server_python = os.environ.get(
        "VLNCE_PYTHON",
        "python",
    )

    def get_tools(self) -> list:
        return [ObserveNode(), StepNode(), ...]

When server_python is set and the nodeset is loaded with ?mode=server:

  1. AutoServerApp wraps the nodeset as a FastAPI HTTP server
  2. BaseServer launches it as a subprocess under the specified Python
  3. Proxy nodes are auto-generated from the server manifest
  4. All forward() calls become HTTP requests to /call/{fn}

8.2 The Manifest Protocol

Every server exposes three endpoints:

Endpoint Method Purpose
/manifest GET Returns ServerManifest (name, functions, ports)
/call/{fn} POST Executes a node function
/health GET Health check

The manifest is auto-generated from your BaseNodeSet — no extra code needed.

8.3 Where to Put Server-Mode Nodesets

Placement follows the role-directory standard (ADR-platform-008): envs live under workspace/nodesets/env/ regardless of deployment — server mode is just the server_python ClassVar:

workspace/nodesets/env/
  env_habitat.py       env_habitat (vlnce env, Python 3.8)
  env_hmeqa/           env_hmeqa
  env_libero/          env_libero
  …

The framework auto-hosts these via AutoServerApp (ADR-server-001) — no extra glue. workspace/servers/*.yaml is the older managed-server path; it still works (registry.py scans for it), but the design-doc recommends the ClassVar pattern above for new code. See Design — NodeSets § Auto-Hosted Mode and § ServerNodeSet API for the full lifecycle (proxy generation, manifest, ports_mode).

8.4 Loading in Server Mode

# Local mode (default — in-process)
curl -X POST "http://localhost:8000/api/components/nodesets/env_habitat/load"

# Server mode (separate process)
curl -X POST "http://localhost:8000/api/components/nodesets/env_habitat/load?mode=server"

8.5 When to Use Server Mode

Scenario Use Server Mode?
Node needs Python 3.8 (Habitat-Sim) Yes — server_python to vlnce env
Node loads a large model (SAM, CLIP) Consider — isolates GPU memory
Node is a simple text processor No — overhead not worth it
Node needs a different conda env Yes — point server_python to it

The key insight: write the nodeset once, deploy either way. The same BaseNodeSet code works both locally and as a server thanks to AutoServerApp (ADR-server-001).

9. Env Metadata for Evaluation

If your nodeset represents a simulator environment (like Habitat), override get_eval_metadata() to expose its capabilities to the batch evaluation system.

9.1 The Metadata Interface

class MySimulatorNodeSet(BaseNodeSet):
    name = "my_simulator"
    description = "My custom simulator environment"

    async def get_eval_metadata(self) -> dict:
        return {
            "env_name": "my_simulator_v1",
            "splits": ["train", "val_seen", "val_unseen", "test"],
            "episode_counts": {
                "train": 10000,
                "val_seen": 778,
                "val_unseen": 1839,
                "test": 3604,
            },
            "metrics": ["spl", "success", "ndtw", "sdtw"],
            "supports_set_episode": True,
            "step_budget": 500,
        }

9.2 What the Eval Page Uses

The eval page reads this metadata to populate:

  • Split dropdown: which dataset splits are available
  • Episode count: how many episodes per split
  • Metric cards: which metrics to display after evaluation
  • Max steps: default iteration limit

Non-environment nodesets return {} from get_eval_metadata() (the default), signaling they don't participate in evaluation.

10.4 Execution Flow

When you press Play on the canvas:

  1. POST /run sends the graph definition to the backend
  2. flatten() expands any composite nodes (see Chapter 11)
  3. The executor builds a dependency graph from edges
  4. Entry nodes (no inputs) fire first
  5. Downstream nodes fire as their inputs arrive
  6. For DAG graphs, execution completes when all nodes have fired

11. Composites and Graph Nodes

A composite is a group of nodes packaged as a single reusable node. Drag it onto any canvas and it behaves like an atomic node — but internally it expands into its constituent graph.

11.1 Two Kinds of Saved Artifacts

Graph Graph Node
Kind kind="graph" kind="node"
Storage workspace/graphs/ workspace/graph_nodes/
Purpose Editable template (opens in tabs) Frozen reusable composite
Canvas action Open in new tab Drag onto canvas
Semantics Reference (live editing) Snapshot (deep copy)

11.2 Creating a Composite

  1. Select multiple nodes on the canvas
  2. Right-click → "Group into Composite" (or use the grouping shortcut)
  3. The system wraps them with PortIn / PortOut boundary nodes
  4. Click "Save as Node" to save to workspace/graph_nodes/

11.3 PortIn and PortOut

Composites have boundary nodes that define their external interface:

  • PortIn: receives data from the parent graph → routes to internal nodes
  • PortOut: collects outputs from internal nodes → sends to the parent graph

When you wire a composite node on the canvas, you're connecting to its PortIn/PortOut handles.

11.4 Flatten Before Execute

Before the executor runs a graph, flatten.py recursively expands all composites:

Composite "NavGPT Reasoning"
├── PortIn (rgb, instruction)
├── CaptionNode
├── LLMCallNode
├── ActionParserNode
└── PortOut (action, thought)

After flatten:
CaptionNode, LLMCallNode, ActionParserNode
(PortIn/PortOut removed, edges rewired to parent)

This means composites have zero runtime overhead — they're a purely organizational abstraction.

11.5 Snapshot Semantics

When you drag a Graph Node onto the canvas, it creates a deep copy. Changes to the original graph node file do not affect instances already on a canvas. This prevents accidental breakage — each canvas owns its nodes.

12. Dataflow Loops

For agents that iterate (observe → think → act → repeat), AgentCanvas provides the Graph Executor with IterIn / IterOut iteration gates.

12.1 The Loop Pattern

Observe → [IterIn] → Reason → Act → Step → [IterOut] → back to IterIn
                                        ↓
                                  [Termination]
  • IterIn: entry point of the loop — receives initial data, then loop-back data
  • IterOut: end of the loop — copies outputs back to the paired IterIn
  • Termination: receives a done signal and stops the loop

12.2 Configuring Loop Ports

IterIn and IterOut use dynamic ports (Chapter 6). Configure them in the graph JSON:

{
  "id": "iterIn_abc123",
  "type": "iterIn",
  "config": {
    "pairedWith": "iterOut_def456",
    "ports": [
      {"name": "rgb", "wire_type": "IMAGE"},
      {"name": "instruction", "wire_type": "TEXT"},
      {"name": "action", "wire_type": "ACTION"}
    ]
  }
}

IterIn mirrors config.ports to both its inputs and outputs (data passes through). IterOut mirrors config.ports to its inputs only (it's a sink — sends data back to IterIn).

12.3 How Iteration Works

  1. Iteration 0: Entry nodes fire → data reaches IterIn → IterIn passes it through
  2. Middle: Nodes between IterIn and IterOut fire in dependency order
  3. At IterOut: Data is copied back to the paired IterIn's output ports
  4. Next iteration: IterIn re-fires with loop-back data → downstream re-fires
  5. Termination: When a done=True signal reaches the Termination node, the loop stops

12.4 step_budget Safety Limit

The step_budget field in the graph JSON caps how many times the loop can repeat:

{
  "name": "Agent Loop",
  "step_budget": 500,
  ...
}

If the agent hasn't stopped after 500 iterations, execution halts. This prevents runaway loops.

12.5 State Across Iterations

Within a loop, state persists via:

  • State containers (Chapter 13): explicit, visible shared state
  • graph_state: auto-accessible dict for all nodes (via ctx)
  • Module-level variables: nodeset-scoped shared state

13. State Containers

State containers are visible canvas elements that hold persistent data across iterations. They use the dual-wire system — data edges for dataflow, state edges for state access.

13.1 What Is a State Container?

A StateContainer is a dict of named BaseState entries. Each entry has a reducer that determines how writes merge:

Reducer Behavior Use For
Accumulator Appends to a list Action history, observation log
LastWrite Replaces previous value Current position, latest plan
Counter Increments an integer Step count, visit frequency

Per-iteration clearing used to live on a separate Ephemeral reducer; that's gone now. Declare lifetime="step" (or episode / run / custom) on any reducer and the signal bus will clear it at the matching boundary.

13.2 Wiring State

State containers connect to nodes via state edges (dashed violet lines), not data edges. A state edge grants a node read/write access to the container — it doesn't carry data through ports.

On the canvas:

  1. Add a StateContainer node from the catalog
  2. Configure its entries (name, type, reducer) in the container panel
  3. Draw state edges from the container to nodes that need access
  4. Toggle state edge visibility with the state toggle button

13.3 Accessing State in forward()

Nodes with state edge access use the ctx parameter:

async def forward(self, inputs: dict, ctx: Any) -> dict:
    # Read from a state entry
    history = ctx.get_state("action_history")  # returns the accumulated list

    # Write to a state entry (reducer determines merge behavior)
    ctx.set_state("action_history", {"step": ctx.step, "action": action})

    # Read the auto-accessible graph_state (available to all nodes)
    instruction = ctx.graph_state.get("instruction", "")
    ...

13.4 State vs Data: When to Use Which

Need Use
Pass data from node A to node B Data edge (wire)
Accumulate history across iterations State container (Accumulator)
Track current position / latest decision State container (LastWrite)
Count occurrences State container (Counter)
Temporary scratch space per iteration State container (lastWrite + lifetime="step")
Global context all nodes can read graph_state

13.5 Signal-driven lifetime

State clearing is declarative: set a state's lifetime field (step / episode / run / custom) and the runtime subscribes it to the matching framework signal. At IterOut, the executor calls broadcast_signal("step_end", …) — each container's on_signal fans to every state, and those with "step_end" in their reset_on list reset to initial_value. Checkpoints are captured right after, so the snapshot is "state ready for N+1 to begin." See State Containers → Lifetime via Signals. The old on_iter_boundary() hook and ephemeral reducer were removed on 2026-04-18.

14. Wrapping a Neural Policy

Neural policies (CMA, Seq2Seq, HAMT, DUET) are exposed as canvas nodes via a BaseNodeSet — workspace/nodesets/policy/policy_cma.py is the reference. Each policy is wrapped as a BaseCanvasNode with a forward-style tool so graphs can drop it in the same way as any other node.

Legacy path retired. An older path registered policies under vlnworkspace/conf/policy/*.yaml and ran them via python -m vlnworkspace.eval …. That package was deleted with the Hydra harness. Batch evaluation now ships on EnvWorkerPool + BatchedInferenceServer (ADR-eval-002) over POST /api/eval/v2/start; the per-episode log layout shipped as ADR-eval-004; the Eval UI is live. See Running a Parallel Batch Eval for the user-facing surface, Design — Batch Eval System for the architecture.

14.1 Policy as a Canvas Node

To use a policy on the canvas, wrap it as a BaseCanvasNode in a nodeset:

class MyPolicyForwardNode(BaseCanvasNode):
    node_type = "my_policy__forward"
    display_name = "Policy: My Model"
    description = "Run inference with my custom policy"
    category = "reasoning"
    icon = "Brain"

    input_ports = [
        PortDef("raw_obs", "TEXT", "Serialized observation dict"),
    ]
    output_ports = [
        PortDef("action", "ACTION", "Predicted action"),
    ]

    async def forward(self, inputs: dict, ctx: Any) -> dict:
        # Access the policy model (loaded during nodeset initialize)
        obs = json.loads(inputs.get("raw_obs", "{}"))
        action = self._policy.predict(obs)
        return {"action": int(action)}

The policy model is loaded in the nodeset's initialize() and stored as an instance attribute. See workspace/nodesets/policy/policy_cma.py for a complete example.

14.2 The Policy Hierarchy

Policies loaded by the VLN-CE family still follow the upstream hierarchy:

BaseVLNPolicy (ABC)
├── RecurrentPolicy (RNN state + CategoricalNet action head)
│   ├── CMAPolicy (Cross-Modal Attention — default)
│   └── Seq2SeqPolicy (GRU baseline)
└── (External models via predict_action interface)

The singleton load pattern used by policy_cma__forward (BaseCanvasNode.batched = True + class-level model load — ADR-eval-002 Phase C) keeps a single GPU-resident model across all canvas runs and all eval workers.

All policies use the VLN-CE action space: 0=STOP, 1=FORWARD, 2=LEFT, 3=RIGHT.

14.3 Batched Inference for GPU Policies

When the same GPU policy is called by K parallel eval workers, declare it batched so all K calls fold into one forward pass (VRAM stays flat as worker_count grows):

class PolicyForwardTool(BaseCanvasNode):
    node_type = "my_policy__forward"
    batched: ClassVar[bool] = True
    batch_dim: ClassVar[str] = "raw_obs"   # input port carrying the per-sample slot
    # … input_ports + output_ports as usual …

Per-call state (RNN hidden, prior action) must move from ctx onto an explicit input port (e.g. hidden_in) and feed back via IterIn/IterOut — the batched server is pure-functional, so anything on ctx is shared across K workers and corrupts. The full mechanics (declaration, state-on-wire pattern, OUTPUTS_KEY/SAMPLES_KEY forward() shape, flush_timeout_ms tail) are in Running a Parallel Batch Eval § 4.
