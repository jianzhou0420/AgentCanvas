What we ported into /architect:adas:loop

What we preserve verbatim from upstream (paradigm-independent elements):

  • R1 + R2 Reflexion prompts verbatim from the paper's mmlu_prompt.py (in the supplementary). These drive /architect:adas:revise.
  • Two-stage proposal cycle: first-pass proposal (R1) followed by reflective revision (R2) on the same parent state.
  • Hill-climb on a single graph: each iter is an edit on the parent's revised state, not a fresh re-search.

What we deliberately did not port (kept in /architect:myloop:loop instead):

  • Sub-stage targets (R1.observation_text, R2.action_parser, etc.) — that's an AAS extension.
  • Target-met judging in the report phase — ADAS uses pure metric-Δ, not "did the declared target effect happen".
  • Atomic-SG enforcement on revised graphs.

What we added on top of strict-upstream (and labelled as such in algorithm.html):

  • Rollback — ADAS upstream doesn't define one; we borrowed AAS' policy-neutral rollback because reverting a regressed iter never injects ADAS-vs-AAS bias.
  • Surrogate sanity gate (P4) — 1-ep node-firing + metric-key check before the expensive batch experiment. Cost-saving, not algorithm-changing.
  • Analyze phase (P1) — ADAS upstream doesn't produce an explicit analysis.md; our analyze writes the proposal seeds R2's prompt context expects.
