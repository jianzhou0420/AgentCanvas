# HTML Authoring (doc-site)

Rules for editing pages under `docs/*.html`. The site is HTML-first as of 2026-05-18 — there is no markdown source. Look at sibling pages in the same section for live examples before authoring.

## Page anatomy

Every page is wrapped in the shared layout from `_layout.py`. The structure as served is:

```
<header class="site-header">…</header>          ← top tabs (Developer Guide / Research)
<div class="site-layout has-right-toc">
  <aside class="sidebar-left" id="sidebar-left"></aside>  ← empty shell; nav rendered client-side from assets/nav.json
  <main class="doc-body <section-class>">
    <div class="topbar">                         ← breadcrumbs + last-updated
      <div class="crumbs">…</div>
      <div class="last-updated">YYYY-MM-DD HH:MM</div>
    </div>
    … your authored content goes here …
  </main>
  <aside class="sidebar-right">…</aside>         ← right-TOC (from H2/H3 ids)
</div>
```

Only the content inside `<main>` (excluding the topbar div) is yours to edit. The chrome is regenerated by `_lib/_wrap_handwritten.py` when nav/layout changes — see "Re-wrapping" below.

## Content conventions

- **Open with an H1** (`<h1>…</h1>`) inside a `<header class="page-header <section-class>">`. The browser title (`<title>`) is set by the wrapper from `<title>` in `<head>`; keep them in sync.
- **Short nav label (optional)**: add `<meta name="nav-title" content="Short Name">` in `<head>` before `<!-- site-layout -->` to make the sidebar + breadcrumb show a short label while the `<h1>`/`<title>` keep the long title. The wrap carries forward only `<style>` blocks and this meta — any other custom head tag is dropped on re-wrap. If you change `_nav.py`/`_wrap_handwritten.py`, restart the dev server first (it imports `_lib` modules once at startup).
- **Lede paragraph** — one paragraph after the H1, summarizing what the page is for. Use `<p class="lede">` if you want the larger-font treatment.
- **Numbered sections** — use `<h2 id="1-section-name"><span class="num">1.</span> Section name</h2>` for top-level sections and `<h3 id="11-subsection"><span class="num">1.1</span> Subsection</h3>` for subsections. The right TOC is built from `<h2>`/`<h3>` `id`s — pick stable, kebab-case slugs.
- **Last updated** lives in the topbar div. The wrap script preserves whatever `<div class="last-updated">...</div>` it finds in the existing topbar. To bump it, edit the page's HTML directly and re-run the wrap.
- **Paper references** — `<a href="https://arxiv.org/abs/NNNN.NNNNN">Paper Name</a> (YY.MM)` with the arXiv-id 2-digit year/month after the link.

## Cross-links

- Within `docs/pages/`: use **page-relative hrefs** (e.g. `<a href="../../core/blueprint.html">` from a dev-guide page). The dev server is a plain file server — relative paths just work.
- Cross-tab (e.g. dev-guide → research, or vice-versa): count `../` levels carefully — `pages/{tab}/<section>/X.html` is 3 levels deep, `pages/{other-tab}/<section>/Y.html` shares the `pages/` ancestor.
- Don't write absolute paths starting with `/` — they break under GitHub Pages subpath deployments.

## Tables, code, lists

- Tables: standard `<table>/<thead>/<tbody>`. Style is inherited.
- Code blocks: pre-highlighted static spans inside `<pre class="hl">` — generate them with `docs/_lib/_hl.py` (Pygments, GitHub palette from `assets/style.css`). **Never** wire a client-side highlighter or hand-write token spans. Full rules: `.claude/standard/code-highlighting.md`.
- Inline code: `<code>foo</code>`.
- Lists: `<ul>` / `<ol>` / nested as you'd expect. For a checklist look, use a `<ul class="checklist">` (defined in style.css).

## Re-wrapping after layout/nav changes

The **top tabs + left sidebar are rendered client-side** by `assets/nav.js` from a generated `assets/nav.json` (built by `_wrap_handwritten.py` via `_nav.build_nav_data`). Each page bakes only empty `#site-tabs` / `#sidebar-left` shells plus page-local `data-tab` / `data-page-rel` on `<body>`. Consequence: **adding / removing / renaming a page touches only that page + `nav.json` — never every sibling.** Breadcrumbs and the right-TOC are still baked per page (they're page-local, so no cross-page churn). The AAS cross-nav strip works the same way since v2: `_aas_nav.py` injects an empty tab container + a small fetch script, never baked links.

**Gitignore = visibility switch (local-only sections).** The wrap pass runs `git check-ignore` over everything under `pages/`: gitignored dirs/pages are pruned from the committed `nav.json` / `search-index.json` (the public view) but kept in the gitignored `nav.local.json` / `search-index.local.json` twins, which the dev server serves in place of the committed files. So a private section shows up locally and never leaks to the published site. To add one: `mkdir docs/pages/<name>/` (or a subdir/page anywhere) + one `.gitignore` line — nothing else to configure.

When you change things, three cases:

1. **CSS/JS-only or nav-tree changes (add/move/rename a page)**: nothing to re-bake across pages. Run `python3 docs/_lib/_wrap_handwritten.py` (or let the dev server auto-wrap) to refresh `nav.json` + the search index; the browser picks up the new asset on next load (dev server sends `Cache-Control: no-store`).
2. **Per-page chrome changes (breadcrumbs, right-TOC, meta, `<body>` data attrs, the empty shells)**: these are still baked, so run `python3 docs/_lib/_wrap_handwritten.py` — it walks `docs/index.html` + `docs/pages/**/*.html`, extracts each `<main>` block, and re-renders chrome via `_layout.render()`. Only changed pages are rewritten.
3. **`_layout.py` / `_nav.py` / `_wrap_handwritten.py` / `assets/nav.js` edits in a long-lived dev server**: restart the dev server once — it caches its own bytecode/module state at import.

## Dev server

```
bash docs/run_dev.sh                   # :8092, live reload via SSE on mtime change
bash docs/run_dev.sh 9000              # custom port
HOST=0.0.0.0 bash docs/run_dev.sh      # bind all interfaces
```

The server is pure stdlib — no deps. Edits land in the browser within ~500 ms.
